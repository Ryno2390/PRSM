## 3. Threat model (auditor priorities)

### 3.1 `_handleConsensusMismatch` with `consensus_group_id` binding

Priority verification questions:

1. **Check ordering** — is the guard order correct (self-reference → group_id non-zero → job/shard/hash binding → conflicting batch exists → group_id match + distinct provider → Merkle verify)? Cheapest checks first, expensive last. Any order-dependent assumption I'm missing?

2. **Group_id prediction attacks** — a `bytes32` is 256 bits; brute-force collision is infeasible. But if the off-chain dispatch uses a weak derivation (e.g., `keccak(job_id || shard_index)` with a predictable job_id), an attacker could pre-commit a batch with the "right" group_id before the legitimate dispatch. The Phase 7.1 E2E uses `keccak(f"consensus-group-{job_id}".encode())`. Should the derivation include a requester-controlled random nonce to harden against pre-commit? Note: even with predictable group_id, the `b.provider != other.provider` check still forces the attacker to control k distinct provider keys — pre-commit alone isn't a full attack.

3. **Open third-party auth** — removing the requester-only check opens the 70% bounty to any observer. Is there any griefing vector this enables that the requester-only MVP prevented? Specifically — can a third party with access to both committed receipts but no stake at risk trigger a challenge that harms the legitimate requester (e.g., by consuming the slashable target before the actual requester could)?

4. **Storage layout** — `Batch.consensus_group_id` is inserted between `tier_slash_rate_bps` and `metadataURI`. Greenfield deploy (no mainnet state); layout is fresh. Audit-flag as such but no migration risk.

### 3.2 `MIN_SLASH_GAS` floor

Priority questions:

1. **Floor value calibration** — 150,000 gas sits ~50K above Solidity's 63/64 forwarding requirement for a ~200K slash path (actual slash cost observed at 110K–160K in tests + gas report). Is 150K the right floor for current StakeBond, or should it be higher to handle future storage layout changes (e.g., a new event field, an additional SSTORE)?

2. **Catch-path preservation** — the `try { stakeBond.slash(...) } catch {}` is preserved. The floor only excludes the OOG case; `NotSlashable` (provider withdrew) and `NothingToSlash` (stake fully drained) still silently succeed through the catch. Is that the intended semantics for all three slashable reason codes (DOUBLE_SPEND, INVALID_SIGNATURE, CONSENSUS_MISMATCH)?

3. **No `max_gas` symmetric cap** — the floor enforces a minimum; there's no ceiling. A very high gas tx could (in principle) do more work inside the catch block than expected. Low-priority but worth noting.

### 3.3 Python-layer concurrency (`ConsensusChallengeQueue`)

Priority questions:

1. **Single-process invariant** — the docstring explicitly says "one submitter process per DB path." Is that a strong enough invariant given that `sqlite3` itself can't enforce it, and operators might accidentally run two submitters against the same DB? If multi-process is a concrete operational concern, a claim-lease column (`claimed_at`, `claimed_by`) with timeout-based re-lease would be the standard fix. See §5.3.

2. **Lock scope** — `_lock = threading.Lock()` is held across each multi-statement operation (e.g., `record_batch_commit` does 3 UPDATEs + 1 promotion-SQL + commit under one lock). No nested lock acquisitions. No operations that could deadlock.

3. **`process_submittable_queue` fetch-then-act** — `list_submittable` returns rows under the lock; lock is released before `submit_one` (which does network I/O); `mark_submitted` / `mark_failed` re-acquires the lock. A second runner fetching SUBMITTABLE rows in parallel could double-submit the same row. Known hazard, single-process invariant covers it. See §5.3.

### 3.4 Runner retry classification

Priority questions:

1. **Allowlist completeness** — only `BroadcastFailedError` is retryable. Are there other genuinely-transient error types we should allow? (The submitter intentionally raises typed Web3 errors; all other failure types represent either contract rejection, tx-may-have-landed uncertainty, or genuine bugs.)

2. **`max_attempts = 5` default** — arbitrary; tuned against "transient blips should clear within 1–2 ticks, persistent failures should escalate quickly." Is 5 the right value for expected operator cadence?

3. **Attempt-counter persistence** — `attempts` is stored on the row; a process restart preserves the count. Which means a new runner instance inherits the retry budget. Correct behavior (prevents infinite retries across restarts).

## 5. Known issues carried into the audit bundle

Ranked by auditor priority. None are merge-blockers; all are documented follow-ups.

### 5.1 `MIN_SLASH_GAS` re-benchmark policy (P4 from cumulative review)

`MIN_SLASH_GAS = 150_000` is a constant in `BatchSettlementRegistry`. If a future `StakeBond` refactor adds storage writes or event fields, the slash cost could push above ~200K and reduce the 63/64-forwarded gas below what the slash needs to complete. Current comment on the constant flags the assumption but doesn't document a re-benchmark policy.

**Auditor: please suggest a re-benchmark trigger** (e.g., "run `hardhat test --gas-reporter` on any `StakeBond.slash` change and confirm peak gas < `MIN_SLASH_GAS`").

### 5.2 `SettlementContractClient.commit_batch` abstract is behind (P2) ✅ RESOLVED

**Original:** `prsm/settlement/client.py` declared an abstract `commit_batch` with the Phase 3.1 signature (6 params). Phase 7 added `tier_slash_rate_bps`; Phase 7.1x §8.7 added `consensus_group_id`. The abstract didn't carry either, so any concrete implementation would have committed with stale defaults.

**Resolved 2026-04-22 (post-review):**

- `SettlementContractClient.commit_batch` signature extended with `tier_slash_rate_bps: int` and `consensus_group_id: bytes` (both required; mock tests pass them via defaults).
- `BatchedReceipt` dataclass gains both fields with safe defaults (0 / all-zero bytes) so existing callers continue to work unchanged. Validation in `__post_init__` rejects out-of-range rates and wrong-length group_ids.
- `PendingBatch` captures both on first append and asserts homogeneity on subsequent appends — a batch's on-chain struct carries one rate + one group_id, so mixed receipts can't share a batch.
- `ReceiptAccumulator` key extended from `(requester, provider)` to `(requester, provider, group_id, slash_rate_bps)`. Non-consensus receipts (group_id = zero) still batch together by (requester, provider) as in Phase 3.1 — the new key degenerates for legacy behavior.
- `BatchSettlementClient._commit_one` destructures the new 4-tuple key and passes both fields through to the contract.

Test coverage: 10 new tests (6 in `test_receipt_accumulator.py`, 4 in `test_batch_settlement_client.py`) covering mixed-group-id separation, mixed-slash-rate separation, legacy zero-group batching, homogeneity enforcement, pass-through to `commit_batch`, and input validation.

Regression: 274 Python unit + 2 E2E + 142 Solidity = 418 green (up from 365 at the tag).

### 5.3 `ConsensusChallengeQueue` claim-lease (P3) ✅ RESOLVED

**Original:** the SQLite queue was thread-safe within one process but had no multi-process coordination. Two runners sharing one DB path could double-submit SUBMITTABLE rows. Docstring-only invariant.

**Resolved 2026-04-22 (post-review):**

- Schema extended with three new columns: `claimed_by TEXT`, `claimed_at REAL`, `lease_expires_at REAL`. Idempotent ALTER-TABLE migration runs on every queue open so pre-existing DBs upgrade transparently.
- New `claim_submittable(claimant_id, lease_seconds=300, limit=100)` method atomically claims SUBMITTABLE rows that are either unclaimed OR whose lease has expired. Single UPDATE-with-subquery = no read-then-write race. Claim token is `"{claimant_id}:{uuid4}"` so concurrent claims from the same claimant never collide on lookup.
- Expired leases are automatically reclaimable, so a crashed runner doesn't strand rows — a new runner picks them up after the lease runs out (default 5 minutes; tunable per call).
- `mark_submitted` and `mark_failed` both clear the lease fields so a terminal row isn't stuck pointing at its last claimant, and a retryable row becomes immediately reclaimable by any runner.
- `process_submittable_queue` gains opt-in `claimant_id` + `lease_seconds` kwargs. When `claimant_id` is None, legacy `list_submittable` path is used (no claim overhead, single-runner semantics). When set, the multi-runner-safe `claim_submittable` path is used.

Test coverage (11 new tests): basic claim + lease metadata, two-runner disjoint claims, "all claimed" returns empty for new claimants, expired-lease reclaim, mark_submitted / mark_failed clear lease, runner integration with claimant_id routes through the claim path, legacy no-claim path still works, DEFAULT_LEASE_SECONDS pinned at 300, schema-migration test on a pre-lease DB, lease state survives DB reopen.

Regression: 427 total green (142 Solidity + 285 Python unit and E2E).

**Auditor: no contract-security action required.** The change is Python-layer operational hardening. If the auditor sees any concern with the single UPDATE-with-subquery atomicity under SQLite WAL mode concurrency, that's the one call-out to double-check.
