# PRSM-PROV-1 T6.5 — Disputed-band arbitration queue design

**Date:** 2026-05-07
**Status:** Design — ready to implement (NOT calibration-gated)
**Track:** PRSM-PROV-1 (content provenance correctness)
**Plan §:** 5.3 T6.5
**Depends on:** T6.1 (`dedup_thresholds.yaml`) + T6.2 (`ThresholdResolver`) — both shipped 2026-05-06

## Why this is unblocked

The plan-doc memo "Item 6 calibration gated 30+ days testnet traffic"
applies to **T6.4** (calibration corpus → threshold ROC sweep). T6.5
is a separate concern: the *workflow* by which a sub-threshold but
non-zero similarity hit is routed for review rather than silently
auto-attributing. The thresholds T6.5 reads come from the YAML
defaults that T6.1 / T6.2 already shipped; T6.4 will later refine
those numbers, but T6.5 is correct at any threshold values.

In other words: T6.5 ships the *plumbing*; T6.4 later tunes the *dial*.

## Current behaviour (`content_uploader.py:887-916`)

```
embedding = await self._get_embedding(content)
match     = self._semantic_index.find_nearest(embedding)
if match.similarity >= DERIVATIVE_THRESHOLD:
    near_dup_cid, near_dup_sim = match.cid, match.similarity
    parents = [match.cid] + parents   # ← AUTO-ATTRIBUTE
```

Two branches:
| Similarity                          | Action |
|-------------------------------------|--------|
| `>= DUPLICATE_THRESHOLD` (≈0.98)    | Auto-attribute, log warning |
| `>= DERIVATIVE_THRESHOLD` (≈0.85)   | Auto-attribute, log info    |
| `< DERIVATIVE_THRESHOLD`            | No-op (treated as unique)   |

The gap shape: the auto-attribute decision is binary at the
DERIVATIVE_THRESHOLD boundary, with no graceful handling of the
"plausibly derivative but not confidently so" middle band that
calibration data will reveal at e.g. 0.70-0.85.

## Proposed behaviour (T6.5)

Introduce a third band, the **disputed band**, between `arbitration_floor`
and `derivative_threshold`. Hits in this band:

1. Allow the upload to complete (don't block the user).
2. Do **NOT** auto-prepend `match.cid` as a parent — auto-attribution
   in this band is what creates the dispute risk T6.5 is fixing.
3. Persist a `DisputedAttributionRecord` in a queue table.
4. Open a Phase 6 governance proposal of category
   `ProposalCategory.ARBITRATION_DISPUTE` (new enum variant) to let
   the candidate-parent's creator and the new uploader argue, with
   council adjudication.

| Similarity                          | Action |
|-------------------------------------|--------|
| `>= duplicate_threshold`            | Auto-attribute, log warning |
| `>= derivative_threshold`           | Auto-attribute, log info    |
| `[arbitration_floor, derivative_threshold)` | **Upload OK, NO auto-parent, queue for arbitration** |
| `< arbitration_floor`               | No-op (treated as unique)   |

`arbitration_floor` is a new field in `dedup_thresholds.yaml`. The
T6.4 calibration sweep (deferred) is what eventually fits the
floor's per-kind value to the ROC; until then a conservative default
of `derivative_threshold − 0.10` is correct (errs toward over-flagging
borderline cases for human review, which is the right failure mode).

### `prsm/data/dedup/arbitration.py` — NEW module

```python
@dataclass(frozen=True)
class DisputedAttributionRecord:
    new_cid: str                  # the upload that triggered the flag
    new_creator: str              # 0x… address of the uploader
    candidate_parent_cid: str     # the existing CID it might be derivative of
    candidate_parent_creator: str # creator of the candidate parent
    similarity: float             # the score that landed in the disputed band
    fingerprint_kind: str         # text / image / audio / video / code / structural
    flagged_at: int               # unix seconds
    proposal_id: Optional[UUID]   # set after Phase-6 proposal is created

class ArbitrationQueue(Protocol):
    async def enqueue(self, record: DisputedAttributionRecord) -> UUID: ...
    async def get(self, record_id: UUID) -> Optional[DisputedAttributionRecord]: ...
    async def list_pending(self) -> list[DisputedAttributionRecord]: ...
    async def resolve(
        self, record_id: UUID, decision: ArbitrationDecision, by_council: list[str]
    ) -> None: ...

class ArbitrationDecision(str, Enum):
    UPHELD_PARENT     = "upheld_parent"      # council confirms derivative; auto-attribute now
    REJECTED_PARENT   = "rejected_parent"    # council rejects derivative claim
    INSUFFICIENT      = "insufficient"       # not enough info; record kept open
```

Two implementations:
- `InMemoryArbitrationQueue` for unit tests (mirrors the
  InMemoryModelRegistry pattern from Phase 3.x.2).
- `FilesystemArbitrationQueue` for node-local persistence (mirrors
  the Phase 3.x.4 PrivacyBudgetStore pattern). DHT-backed
  cross-node arbitration is deferred — single-node arbitration is
  sufficient for the testnet bringup horizon.

### `prsm/node/content_uploader.py` — patch `upload`

Replace the binary branch at lines 894-916 with the three-band
form. Sketch:

```python
match = self._semantic_index.find_nearest(embedding)
if match is not None:
    cid, sim, _ = match
    thresholds = self._threshold_resolver.resolve(kind, content_type_hint)
    if sim >= thresholds.derivative_threshold:
        # auto-attribute (current behaviour)
        near_dup_cid, near_dup_sim = cid, sim
        if cid not in parents:
            parents = [cid] + parents
    elif sim >= thresholds.arbitration_floor:
        # disputed band — flag for arbitration, do NOT auto-attribute
        record = DisputedAttributionRecord(
            new_cid="<pending — assigned post-IPFS-add>",
            new_creator=self.creator_address,
            candidate_parent_cid=cid,
            candidate_parent_creator=match.creator,
            similarity=sim,
            fingerprint_kind=kind.value,
            flagged_at=int(time.time()),
            proposal_id=None,
        )
        # enqueue AFTER ipfs_add so we have new_cid; see "ordering" below
        self._pending_arbitration = record
    # else: < arbitration_floor → no-op (unique)
```

**Ordering subtlety:** the `new_cid` is only known *after*
`_ipfs_add()`. So the enqueue happens immediately post-add, with
the candidate parent + similarity captured pre-add. If `_ipfs_add`
fails, the candidate is discarded — no zombie arbitration records.

### Phase 6 governance hook

```python
# Inside upload, after ipfs_add succeeds and pending_arbitration is set:
if self._pending_arbitration is not None and self._governance is not None:
    record = dataclasses.replace(self._pending_arbitration, new_cid=cid)
    proposal_id = await self._governance.create_proposal(
        proposer_id=self.creator_address,
        proposal=GovernanceProposal(
            category=ProposalCategory.ARBITRATION_DISPUTE,
            title=f"Disputed attribution: {cid[:12]}… vs {record.candidate_parent_cid[:12]}…",
            body=_render_arbitration_body(record),
            metadata={"arbitration_record_id": str(record_id)},
        ),
    )
    await self._arbitration_queue.set_proposal_id(record_id, proposal_id)
```

Add `ARBITRATION_DISPUTE = "arbitration_dispute"` to the existing
`ProposalCategory` enum in `prsm/economy/governance/voting.py`.

## What we deliberately defer

- **T6.4 calibration corpus.** Threshold tuning is correctness-of-numbers,
  not correctness-of-flow. T6.5 ships at the conservative default;
  T6.4 later refines.
- **DHT-backed cross-node arbitration.** The current scope is
  node-local. A creator on node A flagging node B's upload as
  derivative is a v2 concern (requires consistent flag-set replication
  + Sybil-resistant council membership), parked under R10.
- **On-chain arbitration contract.** Per plan §5.2.3, this is the
  long-term target — the ProposalCategory enum reserved for
  ARBITRATION_DISPUTE is the off-chain stepping-stone. Don't
  prematurely contract it.
- **Anti-griefing rate-limiting.** A malicious uploader could drown
  the queue with low-similarity-but-just-above-floor uploads.
  Mitigated short-term by the existing per-creator FTNS bond
  requirement (Phase 7); a per-creator daily flag cap is a v2 hardening
  hook documented but not implemented here.

## Test plan

### Unit (~15 tests, ~250 LOC)
- `ThresholdResolver` exposes `arbitration_floor`; YAML round-trip
- `InMemoryArbitrationQueue` enqueue / get / list_pending / resolve
- `DisputedAttributionRecord` dataclass equality + serialisation
- `_render_arbitration_body` formats the proposal body deterministically

### Integration (~8 tests, ~400 LOC)
- Upload at sim ≥ derivative_threshold → auto-attribute (regression)
- Upload at sim in disputed band → no auto-parent, queue length +=1
- Upload at sim < arbitration_floor → no auto-parent, queue length unchanged
- Two uploads in same disputed band → 2 distinct records
- ARBITRATION_DISPUTE proposal created when governance is wired
- Resolve UPHELD_PARENT → subsequent CID query shows the parent attribution
- Resolve REJECTED_PARENT → subsequent CID query shows no parent attribution
- IPFS-add failure post-flag → record NOT persisted (no zombie)

### E2E (~1 test, ~150 LOC)
- 3-node Sepolia: node A uploads, node B re-uploads at 0.80 similarity,
  assert node B's upload completes, no on-chain auto-parent registration,
  arbitration record visible to council members on both nodes (or fail
  gracefully on the cross-node propagation gap with a TODO marker for
  the R10 deferral).
