# Phase 3.x.7 — Cross-Host ChainExecutor — Design Plan

**Status:** draft
**Date:** 2026-04-28
**Author:** PRSM core
**Predecessors:** Phase 3.x.6 (Parallax scheduler — `phase3.x.6-merge-ready-20260427` tag at `d34ad281`), Phase 2 Ring 8 (single-host `TensorParallelExecutor`), Phase 6 Tasks 1/3/5/6/7 (transport hardening, gRPC streaming chunker, liveness, NAT traversal), Phase 3.x.2 (`FilesystemModelRegistry`), Phase 3.x.5 (`ManifestDHT`), Phase 7 (stake/slash), Phase 7.1 (consensus mismatch).

---

## 1. Why this phase exists

Phase 3.x.6 shipped the **brain** of decentralized inference: the scheduler decides — given a pool of GPUs and a model's layer count — which contiguous layer slices each GPU runs (Phase-1 allocation) and which chain serves each request (Phase-2 routing). It produces a `GPUChain` and hands it to a `ChainExecutor`.

Phase 3.x.6 left the `ChainExecutor` Protocol **abstract**. Tests inject a fake. There is no production implementation that takes a `GPUChain` and actually runs inference across the chain's stages on real network nodes. The single-host `TensorParallelExecutor` (Phase 2 Ring 8) handles in-process tensor parallelism over local shards — but cross-host activation handoff between stages on different machines is not wired.

Phase 3.x.7 closes that gap. After this phase, the canonical PRSM workflow runs end-to-end across multiple nodes:

1. Publish model → Phase 3.x.2 + 3.x.5 (done).
2. Distribute encrypted shards → Phase 7-storage (done).
3. Decide chain → Phase 3.x.6 (done).
4. **Run inference across the chain's stages** → Phase 3.x.7 (this plan).
5. Sign aggregated receipt → existing `prsm.compute.inference.receipt.sign_receipt` path.

This is the last "brain has no hands" gap on the inference critical path.

## 2. Scope

### 2.1 In scope

- **RPC protocol** for cross-host activation handoff between consecutive chain stages. Wire format, message types, error codes, signed handoff tokens.
- **`LayerStageServer`** — per-node server that receives an activation handoff, loads the requested layer slice from its local `FilesystemModelRegistry`, runs the slice via local `TensorParallelExecutor`, and forwards (or returns the final result).
- **`RpcChainExecutor`** — production `ChainExecutor` Protocol implementation. Orchestrator model: the executor calls each stage in sequence, threading the activation tensor through the chain.
- **Activation tensor encoding** — tensor → bytes (numpy or torch, dtype-preserving) + chunked streaming for activations >10MB. Reuse Phase 6 Task 6b chunker.
- **Per-stage TEE attestation collection** — each stage produces an attestation; the final receipt aggregates them under a documented policy (worst-case TEE type wins).
- **Stage authentication** — signed handoff tokens prevent random nodes from dispatching layer execution at the publisher's / requester's expense.
- **Settlement-side integration** — each stage's compute is metered; final receipt's `cost_ftns` reflects the chain total; per-stage failures route to Phase 7 slashing via the existing Phase 7.1 mismatch path.
- **E2E integration test** — 3 simulated nodes, **real** small `ShardedModel`, **real** chain execution end-to-end (not the abstract chain executor fake from 3.x.6 Task 7).
- **Wiring** — `ParallaxScheduledExecutor` accepts a `RpcChainExecutor` instance via the `chain_executor=` kwarg; default factory for production callers.

### 2.2 Out of scope (deferred or already shipped)

- **Streaming-token output** (chat-style incremental decode). v1 returns the complete output at the end of the chain. Streaming requires bidirectional staging and is a Phase 3.x.7.x follow-up.
- **Relay model** (head-stage forwards directly to next stage without round-tripping through the executor). Orchestrator model is simpler to debug and audit; relay can be a perf optimization later.
- **Cross-region pipeline merging.** Phase 3.x.6 allocator partitions by region and never produces cross-region pipelines by construction. Phase 3.x.7 inherits this constraint; cross-region inference is multi-region scheduling, not chain execution.
- **KV-cache reuse across requests.** Each request runs the chain stateless. KV reuse is R7 research-track work.
- **Re-routing mid-chain on stage failure.** v1: stage failure surfaces as `InferenceResult.failure` with a structured reason. Mid-chain failover is Phase 3.x.7.x.
- **Shard distribution.** Phase 7-storage already handles this. v1 assumes each stage already has its layer-slice shards locally. Stages without shards return a `SHARD_MISSING` error code that surfaces as failure.

### 2.3 Non-goals

- Replacing Phase 2 Ring 8's `TensorParallelExecutor`. The cross-host executor wraps it per-stage, not replaces it.
- Replacing Phase 6 transport. The RPC layer rides on top of Phase 6's gRPC channel + signed bootstrap discovery.
- Adding a new TEE runtime. Per-stage TEE bracketing reuses Phase 2 Ring 8's `TEERuntime` (`SoftwareTEERuntime` for dev, `SgxTEERuntime` etc. for prod).

## 3. Architecture

### 3.1 Mental model

```
                      ┌─────────────────────┐
                      │ ParallaxScheduled-  │
                      │ Executor            │
                      │ (Phase 3.x.6)       │
                      └──────────┬──────────┘
                                 │
                       chain = scheduler.route(req)
                                 │
                                 ▼
                      ┌─────────────────────┐
                      │ RpcChainExecutor    │   ← Phase 3.x.7 (this plan)
                      │ (orchestrator)      │
                      └──┬───────────┬──────┘
                         │           │
        signed handoff   │           │  signed handoff
        + activation     │           │  + activation
                         ▼           ▼
        ┌────────────────────┐  ┌────────────────────┐
        │ alice (us-east)    │  │ bob (us-east)      │
        │ LayerStageServer   │  │ LayerStageServer   │
        │ layers 0..L1       │  │ layers L1..L2      │
        │ TensorParallel-    │  │ TensorParallel-    │
        │ Executor (local)   │  │ Executor (local)   │
        └────────────────────┘  └────────────────────┘
                                          │
                                          ▼
                                  final activations +
                                  per-stage attestations
                                          │
                                          ▼
                                  build & sign receipt
```

The executor calls each stage in chain order, passing the upstream activation as input to the next stage's `RunLayerSliceRequest`. Each stage signs its handoff response under its own `NodeIdentity`; the executor verifies stage signatures against the Phase 3.x.3 anchor before accepting the next-stage input as authentic.

### 3.2 Wire format — three message types

```python
@dataclass(frozen=True)
class RunLayerSliceRequest:
    """Handoff from the executor (or upstream stage) to a stage."""
    request_id: str             # echoed from InferenceRequest
    model_id: str
    layer_range: Tuple[int, int]  # half-open [start, end)
    privacy_tier: PrivacyLevel
    content_tier: ContentTier
    activation_blob: bytes      # tensor encoding (see §3.3)
    activation_shape: Tuple[int, ...]
    activation_dtype: str       # "float32" / "float16" / etc.
    upstream_token: bytes       # signed handoff token (see §3.5)
    deadline_unix: float        # request budget; stage must reject if exceeded

@dataclass(frozen=True)
class RunLayerSliceResponse:
    """Successful stage output."""
    request_id: str
    activation_blob: bytes
    activation_shape: Tuple[int, ...]
    activation_dtype: str
    duration_seconds: float
    tee_attestation: bytes      # this stage's attestation
    tee_type: TEEType
    epsilon_spent: float        # 0.0 unless this is the final stage and DP applied
    stage_signature: bytes      # signed by this stage's NodeIdentity
    stage_node_id: str

@dataclass(frozen=True)
class StageError:
    """Structured error response — server NEVER raises through the wire."""
    request_id: str
    code: StageErrorCode        # enum: SHARD_MISSING, TIER_GATE, TIMEOUT,
                                # DEADLINE_EXCEEDED, INVALID_TOKEN,
                                # MODEL_NOT_FOUND, INTERNAL
    message: str
```

Wire codec: same JSON-canonical-bytes pattern as Phase 3.x.5 manifest DHT (`json.dumps(payload, sort_keys=True)`); large `activation_blob` rides via the Phase 6 Task 6b gRPC streaming chunker (>10MB activations stream chunk-by-chunk).

### 3.3 Activation encoding

```python
def encode_activation(tensor: np.ndarray) -> Tuple[bytes, Tuple[int, ...], str]:
    """Returns (raw_bytes, shape, dtype_str). C-contiguous layout
    enforced; non-contiguous tensors copied into a fresh buffer."""

def decode_activation(blob: bytes, shape, dtype_str) -> np.ndarray:
    """Reconstructs the tensor with the given shape + dtype."""
```

`numpy.ndarray.tobytes()` / `np.frombuffer(...).reshape(...)` round-trip. `torch.Tensor` callers convert via `.detach().cpu().numpy()` at the boundary. Float16 supported. No per-element compression in v1 (R7 research track).

### 3.4 LayerStageServer — server-side handler

```python
class LayerStageServer:
    def __init__(
        self,
        identity: NodeIdentity,
        registry: ModelRegistry,
        tensor_executor: TensorParallelExecutor,
        tee_runtime: TEERuntime,
        anchor: AnchorLookup,            # to verify upstream tokens
        clock: Callable[[], float] = time.time,
    ): ...

    def handle(self, request_bytes: bytes) -> bytes:
        """Parse → validate → run → encode response. NEVER raises."""
```

Validation order (matches existing PRSM "validate before work" pattern):

1. Parse the request (malformed → `StageError(MALFORMED_REQUEST)`).
2. Verify `upstream_token` against the Phase 3.x.3 anchor (invalid → `INVALID_TOKEN`).
3. Check `deadline_unix > clock()` (expired → `DEADLINE_EXCEEDED`).
4. `registry.get(model_id)` (missing → `MODEL_NOT_FOUND`); registry's existing signature + shard sha256 check runs here.
5. Confirm the requested `layer_range` is fully covered by shards in this node's local manifest (otherwise → `SHARD_MISSING`).
6. If `privacy_tier != NONE`, confirm `tee_runtime.tee_type` is hardware-backed (otherwise → `TIER_GATE`).
7. Decode the activation; run the layer slice via `tensor_executor.run_layer_range(model, layer_range, activation, privacy_tier)`; encode the output.
8. Sign the response under `self.identity`.

Symmetric "never raises" invariant to Phase 3.x.5's `ManifestDHTServer` and Phase 3.x.6's `ProfileDHTServer`.

### 3.5 Stage authentication — signed handoff tokens

A signed handoff token prevents arbitrary network nodes from dispatching layer execution at the requester's expense. Format:

```python
@dataclass(frozen=True)
class HandoffToken:
    request_id: str
    settler_node_id: str        # the executor's node, the paying party
    chain_stage_index: int      # 0 for chain head; N-1 for tail
    chain_total_stages: int
    deadline_unix: float
    signature: bytes            # signed by the settler under signing_payload()

    def signing_payload(self) -> bytes:
        return json.dumps({
            "request_id": self.request_id,
            "settler_node_id": self.settler_node_id,
            "chain_stage_index": self.chain_stage_index,
            "chain_total_stages": self.chain_total_stages,
            "deadline_unix": self.deadline_unix,
        }, sort_keys=True).encode("utf-8")
```

The `RpcChainExecutor` mints one token per stage, signed under the settler `NodeIdentity` (same identity that signs the final `InferenceReceipt`). Each stage verifies its token via the Phase 3.x.3 anchor — same pattern as `SignedProfileEntry.verify_with_anchor`.

A stage CANNOT replay a token to forge work; the token is bound to a specific `chain_stage_index` and `request_id`. A stage CAN refuse to serve (no slashing for refusal — that just routes to a different chain; persistent refusal is reputation-tracked separately).

### 3.6 RpcChainExecutor — client-side orchestrator

```python
class RpcChainExecutor:
    def __init__(
        self,
        settler_identity: NodeIdentity,
        send_message: Callable[[str, bytes], bytes],  # Phase 6 transport
        anchor: AnchorLookup,
        clock: Callable[[], float] = time.time,
    ): ...

    def execute_chain(
        self,
        *,
        request: InferenceRequest,
        chain: GPUChain,
    ) -> ChainExecutionResult:
        """Orchestrates: encode initial activation (prompt → embedding) →
        for each stage in chain: build RunLayerSliceRequest with signed
        handoff token → send → verify response signature against anchor →
        thread output as next stage's input → assemble final result."""
```

Failure handling:
- Stage returns `StageError` → executor wraps in `ChainExecutionError(stage_node_id, code, message)`. `ParallaxScheduledExecutor.execute()` already maps this to `InferenceResult.failure(...)`.
- Stage signature fails anchor verification → executor treats as a stage compromise; surfaces structured failure AND fires Phase 7.1 challenge (the response was unauthentic, which is a stronger signal than output divergence).
- Network timeout → `StageError(TIMEOUT)` at the executor level; same treatment.

The executor preserves the `phase1_recompute_count` discipline from 3.x.6: stage-level failures don't trigger Phase-1 rebuild. The 3.x.6 `ConsensusMismatchHook` continues to wrap the executor at sample rate.

### 3.7 TEE attestation aggregation

Per-stage attestations are collected; the final `InferenceReceipt.tee_type` is the **worst-case** across stages (e.g., one `SOFTWARE` stage in a chain of `SGX` stages → receipt records `SOFTWARE`). `tee_attestation` field becomes a JSON-encoded list of per-stage attestations, ordered by chain index. Verifiers iterate the list and verify each one against the corresponding stage's vendor attestation service.

This is a documented policy choice. Alternative (chain-of-trust attestation, where one stage attests the next) is more sophisticated but requires platform features not all TEE types support uniformly. v1 takes the simpler route.

### Task 3 — Activation encoding

### Task 7 — E2E integration test
**Scope:** `tests/integration/test_chain_rpc_e2e.py`. Three simulated PRSM nodes (alice, bob, charlie) each running a real `LayerStageServer` over a `FakeNetwork` bus (same pattern as Phase 3.x.5 `test_manifest_dht_e2e.py` and 3.x.6 `test_parallax_e2e.py`). A real small `ShardedModel` (e.g., 4-layer toy LLM, ~1MB) split across the chain; alice runs layers 0..2, bob runs 2..4. Validate:
1. Happy path: client builds chain → executes → receipt verifies; output is bit-identical to single-host execution of the same model.
2. Deadline propagation: deadline 100ms; chain takes 500ms (sleep injected) → client gets `DEADLINE_EXCEEDED` from the stage that exceeded.
3. Forged token: malicious client sends an unsigned `HandoffToken` → server returns `INVALID_TOKEN`.
4. Shard missing: bob is asked to run a layer he doesn't have shards for → `SHARD_MISSING`; executor returns structured failure.
5. Tier gate: `PrivacyLevel.HIGH` request, no stage has hardware TEE → `TIER_GATE` from the first stage.
6. Cross-stage signature failure: bob's response signature is corrupted → executor detects, returns failure, fires Phase 7.1 challenge.
7. Mid-chain disconnect: alice succeeds, bob disconnects → executor gets transport error → structured failure (`InferenceResult.failure`); no hung chain.

**Acceptance:** All 7 scenarios pass against a real (toy) model.

## 5. Risk register

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Activation tensor size at large models | Medium | Medium | Phase 6 chunker already shipped; v1 Task 7 uses small model. Tier-1 large-model perf is post-MVP. |
| TEE attestation chain trust assumption | Medium | High | v1 documents worst-case-wins policy explicitly. Auditor focus item in Task 8. R5 Tier-C scoping doc has follow-up. |
| Stage authentication forgery | Low | Critical | Signed handoff tokens via Phase 3.x.3 anchor; same proven pattern as `SignedProfileEntry`. Task 7 scenario 3 covers it. |
| Mid-chain stage drop → hung executor | Medium | High | Per-stage deadline budget propagated via `deadline_unix`; transport-level timeouts; Task 7 scenario 7 explicitly covers it. |
| Orchestrator model perf cost (2N round-trips for N stages) | Medium | Low | Documented as v1 tradeoff; relay-model perf optimization tracked as Phase 3.x.7.x. Investor-demo perf is acceptable at 3-5 stages. |
| Activation encoding non-determinism (numpy + non-contiguous) | Low | High | Codec enforces C-contiguous layout; round-trip equality tested per dtype in Task 3. |
| Settlement timing on stage drop | Medium | Medium | v1: only the executor (settler) pays; stages that completed successfully still get reputation credit. Slashing of dropped stages is a Phase 7 reputation-tracker hook, not an immediate slash. |
| Cross-language wire compatibility | Low | Low | Same `json.dumps(sort_keys=True)` canonicalization as Phase 3.x.5/6. Cross-language clients are a Phase 4+ SDK concern, not 3.x.7. |
