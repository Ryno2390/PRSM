**Goal:** Close the four findings from the codex re-review of Phase 1.1 so the on-chain provenance system is finally safe to deploy to Base mainnet.

**Status of Phase 1.1:** 8 of 9 codex findings closed cleanly. Re-review caught:
- **P1 #1 (still partial)** — `compute_content_hash` helper, CLI, and the `_try_onchain_distribute` reader path are in place, but the upload/serve/API surface (`content_uploader.py`, `content_provider.py`, `content_economy_routes.py`) never populates `provenance_hash` in `content_metadata`. Real traffic still falls back to local.
- **NEW P2** — `broadcast_pending` payments are reported as `COMPLETED` to API callers because `process_content_access` sets the status unconditionally after `_distribute_royalties` returns.
- **NEW P2** — A malformed `provenance_hash` raises `ValueError` from `bytes.fromhex()` outside the protected try block, so the entire payment is marked `FAILED` instead of falling back to local.
- **NEW P3** — `ProvenanceRegistryClient.register_content` still validates `royalty_rate_bps <= 10000`; the contract caps at 9800.

## Architecture Decisions

### 1. How does the uploader learn the creator's 0x address?

The on-chain creator is the 0x address tied to `FTNS_WALLET_PRIVATE_KEY`, NOT the Ed25519-derived NodeIdentity ID. These are distinct identities. To compute `provenance_hash = keccak256(creator_address || sha3_256(file_bytes))` at upload time, the uploader needs the 0x address.

**Decision:** `ContentUploader.__init__` gains an optional `creator_address: Optional[str]` parameter. The node wires it from `OnChainFTNSLedger._connected_address` if available, else from `PRSM_CREATOR_ADDRESS` env var, else `None`.

When `None`, the uploader does NOT compute or persist a `provenance_hash`, so the on-chain branch in `content_economy` skips and the local fallback handles the payment. Backward compatible — existing nodes that don't run the on-chain stack lose nothing.

### 2. Where is `provenance_hash` stored?

**At upload time:** computed once in `ContentUploader.upload()` (and `_upload_with_sharding`) from `creator_address` + raw file bytes. Stored in:
  1. `UploadedContent.provenance_hash` — new dataclass field, persisted to DB.
  2. `provenance_data["provenance_hash"]` — gossip payload, so other nodes know it.
  3. `content_provider.register_local_content(metadata={"provenance_hash": ...})` — so the serving path can read it back.

**At serve time:** `content_provider._handle_content_request` reads `provenance_hash` from `_local_content[cid]["metadata"]` and passes it to `content_economy.process_content_access(content_metadata={..., "provenance_hash": ...})`.

**At API time:** the FastAPI `ContentAccessRequest` model gains an optional `provenance_hash: Optional[str]` field (hex with or without `0x` prefix). The handler forwards it to `process_content_access`.

### 3. `PaymentStatus.PENDING_ONCHAIN`

New enum value alongside `PENDING`, `ESCROWED`, `COMPLETED`, `FAILED`, `REFUNDED`. Set by `process_content_access` when `_distribute_royalties` returns a list that contains an entry with `type == "broadcast_pending"`. The API serializes the enum value as `"pending_onchain"` to clients, so callers know to wait for reconciliation rather than treat it as final.

### 4. Malformed `provenance_hash` handling

`_try_onchain_distribute` parses the hex string to bytes. Today the parse happens before the protected try block, so a bad string raises and propagates. Move the parse inside a localized try/except and return `None` on failure (safe fallback to local). Log a warning so operators see the bad input.

### 5. Tighten client-side rate validation

`ProvenanceRegistryClient.register_content` validates `0 <= royalty_rate_bps <= 10000`. The on-chain registry now caps at 9800. Library callers can submit 9801..10000 and burn gas on a guaranteed revert. Change the bound to `<= 9800` and update the existing rejection test.

### Out of Scope

- Backfilling `provenance_hash` for content uploaded before Phase 1.2. Old content keeps falling back to local. Documented limitation.
- A separate "register provenance" endpoint on the API. Operators use the CLI for now.
- Multi-creator content (Alice and Bob both register the same file as separate creators). The data model supports it; surfacing it in the API is Phase 2+.
