## 5. Operations after deploy

### 5.1 Anchor cutover checklist (operator migration footgun)

Once the contract is live and an operator decides to flip a node's `FilesystemModelRegistry` or `FilesystemPrivacyBudgetStore` to `anchor=`-mode, the trust upgrade is **strict by design** â€” verification refuses to fall back to the local sidecar. If a publisher's key isn't on-chain yet, the registry / store will refuse to serve their artifacts.

Before flipping `anchor=` on a populated registry / store:

1. **Inventory all publishers** that have models / journals on the node. For the model registry, that's the unique `publisher_node_id` set across `<root>/<model_id>/manifest.json`. For the privacy-budget store, the journal's `node.pubkey` sidecar identifies the single signer.
2. For each publisher, confirm they've called `register_self()` on the deployed contract (`anchor.lookup(node_id)` returns their pubkey).
3. Only after every publisher is registered: flip `anchor=` on the registry / store and restart the node.

A `bulk_register_publishers([identity_a, identity_b, ...])` helper is NOT shipped in v1; operators do this manually via `PublisherKeyAnchorClient.register_self(identity)` per identity. (Helper deferred to a v2 enhancement once a real bulk-migration use case appears.)

### 5.2 Cache invalidation after `adminOverride`

The `PublisherKeyAnchorClient` caches `lookup` results with a default 1-hour TTL. After a Foundation `adminOverride` rotation (compromised-key revocation), verifiers' caches will continue serving the old key for up to TTL.

Two ways to handle:

- **Targeted**: instruct verifier operators to call `client.invalidate(node_id)` for the rotated node_id. Cache forgets immediately; next lookup re-fetches from chain.
- **Coarse**: restart the verifier process. Cache is in-memory only; restart drops everything.

For **emergency revocations** (active key compromise), prefer the targeted approach + an out-of-band notification to operators. For routine rotations (none planned in v1), the TTL handles it organically.

### 5.3 Pre-existing artifacts after rotation

`adminOverride` invalidates every signature produced under the OLD key. Operators with a model registry containing manifests signed under the rotated key will see those models stop verifying. The publisher MUST re-sign and re-publish under the new key. There is no on-chain pointer from the old â†’ new key; this is a deliberate v1 simplification (see design plan Â§1.2).

## 6. What this runbook does NOT cover

- **Base mainnet deploy.** Bundles into the existing Phase 1.3 audit clock; no engineering action until that gate clears.
- **Multisig setup.** Operator's responsibility (e.g., deploying a Gnosis Safe and adding co-signers). Out of this phase's scope.
- **Funding the deployer wallet.** Use the Sepolia faucet (https://sepoliafaucet.com/ or similar).
- **Migrating existing publishers.** No publishers are registered yet â€” the contract is fresh on first deploy. The Python anchor client (Task 3) handles `register_self()` per-publisher.

## 7. Rollback

If the deploy goes wrong (wrong admin, wrong RPC, wrong network), the rollback is "deploy a new contract." There is no `selfdestruct` and `admin` is `immutable`. The aborted contract sits on Sepolia harmlessly; just don't commit its address to `contract_addresses.json`.

The cost of a do-over on Sepolia is negligible (~0.001 ETH). On Base mainnet, the cost is real but still <$5 at typical gas; that scenario is gated by the Phase 1.3 audit anyway, so a clean redeploy is always feasible.
