## 1. Scope

### 1.1 In scope

Phase 7.1 adds **Tier B (redundant-execution verification)** to the PRSM compute marketplace. k providers execute the same shard in parallel; the requester compares output hashes; the majority wins; minority is slashable on-chain via a new `CONSENSUS_MISMATCH` reason code.

**Solidity (modified):**
- `contracts/contracts/BatchSettlementRegistry.sol` — `ReasonCode` enum gains `CONSENSUS_MISMATCH = 5`; new `_handleConsensusMismatch` internal function (cross-batch pattern mirroring `_handleDoubleSpend`); challenge dispatch chain extended; slash-hook reason allowlist extended.

**Solidity (unchanged):**
- `contracts/contracts/StakeBond.sol` — Phase 7's `slash()` path absorbs CONSENSUS_MISMATCH without any modification. Self-slash protection (`challenger == provider` → 100% to Foundation) still fires.

**Python (new):**
- `prsm/compute/shard_consensus.py` — pure-function resolver.
- `prsm/compute/multi_dispatcher.py` — asyncio.gather wrapper + `ConsensusShardReceipt` dataclass + `InsufficientResponsesError` / `ConsensusFailedError`.

**Python (modified):**
- `prsm/economy/web3/stake_manager.py` — `ReasonCode` IntEnum, `SLASHABLE_REASONS` set, `REASON_*_KECCAK` constants, `_reason_keccak` helper.
- `prsm/marketplace/policy.py` — `DispatchPolicy.consensus_mode` + `consensus_k` fields.
- `prsm/marketplace/orchestrator.py` — consensus-routing branch, `_select_top_k` helper, `_queue_consensus_challenges` accumulator + `consensus_minority_queue` attribute.

### 1.2 Out of scope (unchanged since Phase 7)

- Phase 7 Tier C stake + slashing primitives — `phase7-merge-ready-20260421`.
- Phase 3.1 batched-settlement registry — `phase3.1-merge-ready-20260421`.
- Phase 1.1 FTNS token / provenance / royalty stack.

### 1.3 Excluded by design (deferred to Phase 7.1x)

- **BYZANTINE consensus mode.** `ShardConsensus.__init__` raises `NotImplementedError` at construction. MAJORITY + UNANIMOUS ship.
- **Adaptive k.** Phase 7.1 MVP has a fixed k per job (default 3) supplied by the requester.
- **Consensus-submitter service.** The orchestrator accumulates minority receipts in an in-process `consensus_minority_queue`; a separate production service (Phase 7.1x) drains it post-batch-commit, submits on-chain challenges, and handles retry/persistence. The queue IS a clean seam — see §5.3 below.
- **`consensus_group_id` third-party challenger.** MVP auth is requester-only (mirrors NO_ESCROW). Phase 7.1x introduces a batch-level binding so any holder of k signed receipts can challenge, which would close the sybil-requester griefing vector (§5.4 below).
- **Per-provider price handshake in consensus path.** MVP uses `policy.max_price_per_shard_ftns` as the total budget, divided evenly across k. Per-provider price discovery in k-of-n is Phase 7.1x.
- **Floating-point tolerance.** Output comparison stays hash-exact (Phase 2's float64 bitwise-reproduction contract). Fuzzy comparison exists at the job-result layer in `prsm/node/result_consensus.py` — intentionally NOT used at the shard-receipt layer. GPU non-determinism flagged as §8.4 of the design doc.

## 4. Threat model (auditor priorities)

### 4.1 Cross-batch handler correctness

`_handleConsensusMismatch` uses a cross-batch proof pattern. The challenged leaf sits in the **minority** batch (b.provider = minority's committer); the auxData carries a conflicting batch id + majority leaf + proof for that OTHER batch.

Verification order (cheapest → most expensive):
1. `msg.sender == minorityBatch.requester`
2. `conflictingBatchId != batchId` (distinct batches; blocks self-referential challenges)
3. `leaf.jobIdHash == majorityLeaf.jobIdHash` (same job)
4. `leaf.shardIndex == majorityLeaf.shardIndex` (same shard)
5. `leaf.outputHash != majorityLeaf.outputHash` (genuine disagreement)
6. `batches[conflictingBatchId]` exists (reverts `ConflictingBatchNotCommitted` otherwise)
7. `MerkleProof.verify(majorityProof, conflictingBatch.root, hash(majorityLeaf))`

**Specific auditor questions:**
- Is the verification order correct (auth before any storage reads; binding checks before proof verification)?
- Can a malicious challenger pair leaves from UNRELATED k-of-n jobs to forge "disagreement" and farm the 70% bounty? (Expected answer: no — checks 3 + 4 bind leaves to the same (jobIdHash, shardIndex).)
- Can a provider self-slash by committing two conflicting batches themselves and having a colluding "requester" challenge? This is the §8.7 sybil-requester griefing vector — please confirm the math (negative EV, not farming).

### 4.2 Self-slash protection

Phase 7's `StakeBond.slash()` routes 100% to Foundation when `challenger == provider`. Under Phase 7.1's cross-batch CONSENSUS_MISMATCH flow, `b.provider` is the minority's committer and `msg.sender` (challenger) is the minority batch's requester. If an attacker has BOTH addresses (distinct EOAs but same principal), the `==` check doesn't trigger, and the 70/30 split fires — as designed for requester bounties, but opens the §8.7 griefing vector.

Please verify:
- The `challenger == provider || challenger == address(0)` check at `StakeBond.sol:357` is sufficient as the self-slash guard under the current MVP auth model.
- The §8.7 griefing scenario has negative expected value and the reputation damage to P is a non-trivial real cost (i.e., it genuinely is griefing, not a rent-extraction attack).

### 4.3 Authorization

`_handleConsensusMismatch` reverts with `CallerNotRequester` if `msg.sender != b.requester` (the minority batch's requester). The MVP relies on the requester being the single entity that dispatched the k-of-n job and holds all k signed receipts.

Please verify:
- No bypass via flash-loan / delegate-call / proxy setup that could let a non-requester pose as the requester.
- The MVP restriction has a planned relaxation path (Phase 7.1x `consensus_group_id`) that is compatible with forward-contract-upgrade expectations.

### 4.4 `ShardConsensus` arithmetic

- MAJORITY threshold: `(k // 2) + 1`. Correct for k=3 (→ 2), k=5 (→ 3), k=4 (→ 3), k=7 (→ 4). Please confirm no off-by-one.
- UNANIMOUS: requires `len(receipts) == k AND winning_count == k` — rejects both disagreement AND partial response.
- Tie-detection enumerates all hashes at `winning_count` and rejects if `len(top_groups) > 1`. 2-2 at k=4 correctly yields no-agreement.

### 4.5 Storage layout

No storage additions in Phase 7.1 — the `ReasonCode` enum extension is `uint8` on the wire only. The `Batch` struct from Phase 3.1 is unchanged; the `tier_slash_rate_bps` field added in Phase 7 still occupies the same slot. Greenfield deploy (no mainnet upgrade concerns at this stage).

## 5. Known issues (from the Phase 7.1 Task 8 review gate)

The independent code-reviewer agent returned **SAFE TO MERGE** on `phase7.1-merge-ready-20260421` with three non-blocker follow-ups, all carried forward to this audit engagement.

### 5.1 Consensus-minority queue persistence (design doc §8.6)

**Issue:** `MarketplaceOrchestrator.consensus_minority_queue` is in-memory, never drained, never persisted. A process crash between dispatch and challenge submission drops queued minority receipts on the floor.

**Impact:** Lost challenges mean minority providers go unslashed for the dropped events. Does NOT allow misbehavior to occur; only fails to punish it after the fact. Reputation record (from direct `record_slash` calls) is a separate stream.

**Current mitigation:** Design doc + commit message both explicitly mark the queue as a seam for a downstream submitter service. Orchestrator-layer operational gap, not a contract-security issue.

**Planned resolution:** Phase 7.1x "consensus-submitter service." Drain API, SQLite/Redis persistence, handoff hook from settlement accumulator. Phase 3.1 already has DB infrastructure; the submitter service reuses it.

**Auditor: no contract-security action required.** Please note in findings that minority receipts currently accumulate without automatic on-chain action.

### 5.2 Sybil-requester griefing vector (design doc §8.7)

**Issue:** Under MVP auth (requester-only challenger), an attacker controlling two EOAs (provider P + requester R, distinct addresses, same principal) can commit two conflicting batches as P with R as requester, then have R challenge. Self-slash protection doesn't fire (R ≠ P), so the 70/30 split routes 70% back to R.

**Impact:** Griefing, not farming. Net attacker cost per burn ≈ 30% of slashed amount + gas + reputation destruction of P. At `critical` tier this is ~15K FTNS net loss plus permanent exclusion of P from premium/critical dispatches. Negative expected value for the attacker.

**Current mitigation:** Economics. The 30% Foundation skim ensures the attack is strictly unprofitable.

**Planned resolution:** Phase 7.1x `consensus_group_id` binding. Batches carry a group id that requires k SIGNED RECEIPTS (one per provider) to commit under one consensus umbrella — forging would require the attacker to control k distinct provider keys, multiplying the cost k×.

**Auditor question:** Please confirm the math is negative-EV across realistic parameter ranges (tiers from `standard` to `critical`, typical gas costs, market price of FTNS at deploy time per the attack cost model in PRSM-ECON-WP-1).

### 5.3 asyncio timeout/cancel propagation (design doc §8.8)

**Issue:** `MultiShardDispatcher` classifies `ShardPreemptedError`, `MissingAttestationError`, and `ShardDispatchError` as "did not respond." Other exceptions propagate (so outer-loop cancellation aborts the gather rather than being silently tolerated). The reviewer flagged whether `asyncio.TimeoutError` / `asyncio.CancelledError` paths are correctly wrapped inside the single dispatcher.

**Current state:** `RemoteShardDispatcher` wraps transport-level timeouts into `ShardDispatchError`. An unwrapped `TimeoutError` at the gather level would mean outer-scope cancellation, where propagation is the correct behavior.

**Disposition:** low-priority audit item. Worth a grep pass during audit confirming no timeout path escapes the single dispatcher without wrapping.
