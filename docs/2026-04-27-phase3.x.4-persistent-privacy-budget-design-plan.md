## 1. Context & Goals

`PrivacyBudgetTracker` enforces cumulative differential-privacy ε limits per node. The Phase 3.x.1 inference flow already uses it correctly (pre-flight `can_spend` gate + post-execute `record_spend` commit, per the round-1 review remediation in commit `46102bd0`). What's missing is durability + tamper-evidence:

- **Restart amnesia**: the tracker is `Dict`-backed in-memory. A node restart resets `total_spent` to 0; an operator can dodge MAXIMUM-tier ε ceilings by simply restarting after a heavy spend.
- **Reset abuse**: `tracker.reset()` is documented as a reset-for-new-epoch helper but provides no audit trail.
- **Forgery surface**: the tracker has no signature chain; a malicious operator handing an audit log to a regulator could fabricate or backdate entries.

This phase closes all three by wrapping the existing tracker in a signed, append-only journal — same idiom Phase 3.x.2 used for model manifests.

### 1.1 Why now

The phase 3.x.1 + 3.x.2 surface ships as the canonical "verifiable inference" story. The privacy-budget primitive is the only remaining state-bearing component on the inference hot path that is not durable + not signed. Closing the gap in the same week the registry shipped:
- Keeps the "every state-bearing primitive is verifiable" narrative coherent for any partner doing a deep review.
- Reuses idioms (sign helpers, sidecar pubkey, filesystem trust boundary, cross-artifact domain separation) that are still warm from 3.x.2.
- Keeps 3.x.3 (DHT + on-chain anchor) fully isolated as a strategic-scoping track instead of accumulating "small extras" alongside it.

### 1.2 Non-goals for Phase 3.x.4

- **Not on-chain anchoring of budget journals.** Same deferral as 3.x.2: cross-node verifiable budgets need an authoritative pubkey resolver, which is the 3.x.3 track.
- **Not multi-publisher journals.** Each node's tracker maintains its own journal under its own key. Cross-node aggregation is a discovery/audit-layer concern.
- **Not encrypted entries.** The metadata (epsilon, operation, model_id, timestamp) is auditor-facing by design; encrypted entries would defeat the audit trail.
- **Not budget federation.** A budget-pool spanning multiple physical nodes is out of scope.
- **Not a hard-flip of `PrivacyBudgetTracker`.** The class continues to exist and stay in-memory-only by default. Persistence is opt-in via a new `store=` kwarg on a thin subclass.
- **Not retro-signing of entries created before the persistent tracker was wired in.** Migration: an operator switching from in-memory to persistent starts a new journal at sequence 0; the historical in-memory state is treated as "not yet signed."

### 1.3 Backwards compatibility

- `prsm.security.PrivacyBudgetTracker` continues to exist with the existing API (`can_spend / record_spend / total_spent / remaining / reset / get_audit_report`).
- `PersistentPrivacyBudgetTracker(max_epsilon, store=None, identity=None)` is a subclass; with `store=None` it behaves exactly like the parent (no disk writes, no signatures). Existing tests pass unchanged.
- The single construction site at `prsm/node/node.py:825` swaps to `PersistentPrivacyBudgetTracker(max_epsilon=100.0)` with no other changes; the store is wired in by node-init code that has access to a configured root path.
- Existing call sites (`record_spend` / `can_spend` / `get_audit_report`) all continue to work without signature awareness — the persistent layer adds signing transparently.

### 1.4 Trust model

Mirrors Phase 3.x.2:

| Concern | Defense |
|---|---|
| Operator wipes the tracker by restarting the process | On construction, the persistent tracker replays the on-disk journal to reconstitute `total_spent`. Restart-survival verified by the parity tests. |
| Operator deletes the journal on disk to "reset" cumulative ε | A v1 limitation: filesystem-level deletion is invisible. **Documented in the production-deployment notes**: regulators auditing a node should pin the journal root to backed-up storage and check `latest.json.sequence` monotonically increases across snapshots. Cross-node anchor (3.x.3) is the proper fix. |
| Operator backdates a single entry to make the budget look healthier | Each entry chains to `prev_entry_hash`; tampering with entry N invalidates entry N+1's signature, and so on through the latest. `verify_journal()` walks the chain end-to-end. |
| Operator forges an entry under a different key | Each entry signed by the node identity stashed via `publisher.pubkey` sidecar. Verifier rejects entries whose signature doesn't verify under that key. |
| `tracker.reset()` abuse | Reset becomes a special signed entry of type `"reset"` rather than an in-place wipe; the journal preserves the full pre-reset history. Auditors can see exactly when and at what cumulative ε. |
| Same-key cross-artifact replay (e.g., reuse a manifest signature as a budget entry) | Domain separator: `b"prsm-privacy-budget-entry:v1"` — separate from `b"prsm-model-manifest:v1"` and `b"prsm-inference-receipt:v1"`. |

### 3.1 PrivacyBudgetEntry format

```python
class PrivacyBudgetEntryType(str, Enum):
    SPEND = "spend"
    RESET = "reset"

@dataclass(frozen=True)
class PrivacyBudgetEntry:
    sequence_number: int            # 0-indexed, monotonic, gap-free
    entry_type: PrivacyBudgetEntryType
    node_id: str                    # signer's NodeIdentity.node_id
    epsilon: float                  # 0.0 for RESET
    operation: str                  # e.g., "inference", "forge_query"; "" for RESET
    model_id: str                   # "" for RESET
    timestamp: float                # unix timestamp
    prev_entry_hash: bytes          # 32-byte sha256 of prev entry's signing payload (or zeros for genesis)
    schema_version: int = 1
    signature: bytes = b""          # excluded from signing payload
```

**Canonical signing payload** (excludes `signature` itself):
```
b"prsm-privacy-budget-entry:v1|"
+ str(schema_version) + "|"
+ str(sequence_number) + "|"
+ entry_type.value + "|"
+ node_id + "|"
+ f"{epsilon:.10f}" + "|"
+ operation + "|"
+ model_id + "|"
+ f"{timestamp:.6f}" + "|"
+ prev_entry_hash.hex()
```

Same separator convention as `ModelManifest.signing_payload()`. The `:.10f` epsilon and `:.6f` timestamp formats lock numerical precision so two clients reconstructing the payload agree byte-for-byte.

### 3.2 Filesystem layout

```
<root>/
├── entries/
│   ├── 000000.json    — genesis entry (prev_entry_hash = b"\x00" * 32)
│   ├── 000001.json
│   └── ...
├── latest.json        — {"sequence_number": N, "entry_hash": "<hex>"} pointer
└── node.pubkey        — b64 of the signing identity's public key (sidecar)
```

- Entries are content-addressed by sequence number; gap-free.
- `latest.json` is a small atomically-rewritten pointer that the tracker reads on construction to know where to start replay-from. Lets restart skip a full directory scan when the journal is large.
- **Append-only invariant**: `store.append()` refuses to write entry N if N != current_max_sequence + 1. Tested.
- **Chain invariant**: `entry[N].prev_entry_hash == sha256(entry[N-1].signing_payload())`. Tested for both happy path and tamper.

### 3.3 Tracker integration contract

```python
class PersistentPrivacyBudgetTracker(PrivacyBudgetTracker):
    def __init__(
        self,
        max_epsilon: float = 100.0,
        *,
        store: Optional[PrivacyBudgetStore] = None,
        identity: Optional[NodeIdentity] = None,
    ):
        super().__init__(max_epsilon=max_epsilon)
        self._store = store
        self._identity = identity
        if store is not None:
            # Replay journal → reconstitute in-memory state
            for entry in store.replay():
                if entry.entry_type == PrivacyBudgetEntryType.RESET:
                    super().reset()
                else:
                    super().record_spend(entry.epsilon, entry.operation, entry.model_id)

    def record_spend(self, epsilon, operation, model_id=""):
        # Pre-flight: check budget BEFORE writing the journal entry, so a
        # rejected spend doesn't pollute the chain.
        if not self.can_spend(epsilon):
            return False
        if self._store is not None and self._identity is not None:
            self._append_entry(PrivacyBudgetEntryType.SPEND, epsilon, operation, model_id)
        return super().record_spend(epsilon, operation, model_id)

    def reset(self):
        if self._store is not None and self._identity is not None:
            self._append_entry(PrivacyBudgetEntryType.RESET, 0.0, "", "")
        super().reset()
```

When `store=None` or `identity=None`, behavior matches the in-memory parent exactly — no disk I/O, no signatures.

## 6. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Journal corruption on filesystem (partial write, FS bug) | Low | High | Atomic writes via `.tmp + os.replace`; `verify_chain()` walks the chain on every store construction → corruption surfaces as `JournalCorruptionError`, not silent data loss. |
| Operator wipes journal between restarts | Medium | Medium | Documented limitation; cross-node anchor (3.x.3) is the proper fix. Operators audited externally (e.g., regulators) should pin journal root to backed-up storage and check `latest.json.sequence` monotonicity across snapshots. |
| Performance: per-spend disk write + Ed25519 sign on the inference hot path | Medium | Low | Ed25519 sign is ~50 µs; atomic file write + fsync is ~1-5 ms on SSD. Inference latency is already ms-scale; budget I/O adds <5% overhead. If benchmarks show otherwise, batch-commit groups of N entries (deferred). |
| `record_spend` returns False but entry written → ledger drifts | Low | High | Pre-flight `can_spend` check happens BEFORE journal append; a refused spend never writes. Tested. |
| Cross-artifact replay: someone tries to use a manifest signature as a budget-entry signature | Low | Medium | Domain separator (`b"prsm-privacy-budget-entry:v1"`); cross-artifact replay test pinned in Task 2. |
| Genesis entry edge case: prev_entry_hash for entry 0 | Low | Low | Documented as `b"\x00" * 32`; tested. |
