## 1. Why this addendum exists

Phase 3.x.11 closes the tail-only contract that's been load-bearing since Phase 3.x.8: each chain stage now runs its layers *once per generated token*, with KV-cache state surviving locally between iterations, and activations crossing the wire at every per-token boundary.

This is a substantial architectural shift. Three threat surfaces change shape under sharded decode and require explicit re-characterization:

1. **Per-token wire dispatch** introduces a NEW timing observation surface that the existing Phase 3.x.10 memo doesn't model.
2. **KV-cache state on stages** is privacy-sensitive (encodes the prompt + generated tokens) and lives across multiple network-observable events.
3. **Cross-stage activation handoff frequency** multiplies the activation-inversion surface from R3 by a factor of `max_tokens` (per-request).

Phase 3.x.10's tail-only autoregressive decode is unchanged by this addendum — its threat model still applies when operators run with `enable_sharded_decode=False`. Sharded decode is OPT-IN; this addendum applies only to the opt-in path.

## 2. Threat model deltas vs. R3 baseline

### 2.1 What the attacker still controls (R3 §2.2 unchanged)

- Their SPRK's runtime + memory.
- Their node's network interface.
- Timing + ordering of their own responses.

### 2.2 What the attacker NEWLY observes under sharded decode

- **Per-token wire arrivals.** A network observer on any segment of the chain sees one `RunLayerSliceRequest` per stage per generated token. For a `T`-stage chain producing `N` tokens, that's `N × T` observable wire events per request — vs. exactly `1` in the Phase 3.x.7 unary path and `N` in the Phase 3.x.8 streaming-tail path (tail-only emits N TokenFrames).
- **Per-token activations crossing each stage boundary.** Each stage forwards its boundary hidden state as the next stage's input; the hidden-state tensor is observable on the wire (encrypted under TLS at the transport layer per Phase 6, but visible to anyone with TLS termination — including the receiving stage itself).
- **KV-cache state on each stage that hosts the request.** The cache lives in the stage's process memory across the full decode loop; an attacker with read access to the stage's memory (R3 S1 attacker on that specific stage) sees not just one activation but ALL cached K/V tensors covering every token position so far.
- **Cache-eviction signals.** `EvictCacheRequest` envelopes carry the request_id in cleartext at the wire boundary (no signature on the eviction path — see honest scope in §3.6 of design plan).

### 2.3 What the attacker still does NOT control (R3 §2.3 unchanged + additions)

- Other SPRKs' memory (cache state on a stage Mallory doesn't control stays opaque to her).
- Topology decisions before dispatch.
- TEE-sealed attestation roots.
- **NEW**: Eviction race resolution. `KVCacheManager.evict` is idempotent — duplicate broadcasts (e.g., from retried close paths) and TTL-sweeper races resolve cleanly without exposing the attacker any extra state.

### 2.4 Attacker scale revisited

R3's S1 / S2 / S3 scales remain the right framing, but the per-request observation surface MULTIPLIES under sharded decode:

| Scale | Phase 3.x.7 unary | Phase 3.x.8 streaming-tail | Phase 3.x.11 sharded |
|---|---|---|---|
| S1: one shard, one layer, one request | 1 activation observation | 1 activation + N text deltas | **1 + N** activation observations + KV-cache memory residue |
| S2: one shard across many requests | corpus of 1×R activations | corpus of 1×R + N×R deltas | **corpus of (1+N)×R** activations + R caches |
| S3: multi-shard coordinated | (per-shard S2)×K colluders | (per-shard S2)×K colluders | **(per-shard S2)×K colluders, each with N× the per-request surface** |

The S2 and S3 deltas are the load-bearing concern. A long-lived attacker on a sharded-decode-serving node accumulates `(1 + max_tokens) × num_requests` activation observations — for `max_tokens=512` and 1000 requests served per day, that's >513,000 activations vs. ~1,000 under the unary path.

## 3. New threat surfaces (this addendum)

### 3.1 Per-token wire timing surface

**Surface.** Each per-token chain pass produces `T` wire events (T = chain depth). A network observer on any segment of the chain measures inter-event timing. Per-token decode time on each stage is data-dependent (longer prompts → bigger attention K/V → slower per-stage forward; specific token decisions correlate with specific compute paths in attention). **The observer learns timing PER HOP, PER TOKEN.**

**Comparison to Phase 3.x.10 tail-only.** Phase 3.x.10's timing surface is *one* per-token timing per request (tail emits one TokenFrame per generated token; non-tail stages run only once during prefill). Phase 3.x.11 multiplies that by T (chain depth): `N × T` timing observations per request.

**Why the existing constant-time padding decorators don't help.** Phase 3.x.10.y's `BatchedTrailingStreamingRunner` (M2) and `FixedRateStreamingRunner` (M1) operate at the streaming-tail layer — they padd inter-token latency at the *tail* output. Sharded decode dispatches `T` unary RPCs per token, each completing at its own per-stage rate; M1/M2 never see those events because they live above `LayerStageServer.handle` (sharded uses `_dispatch_sharded`, not `handle_token_stream`).

A constant-time decorator for sharded decode would need to:
- Pad per-stage forward time at each stage uniformly (mask per-token decode-time variance).
- Pad inter-stage wire latency between stages (mask network jitter that itself reveals inter-stage compute).
- Guarantee the chain's total per-token wall-clock is data-independent.

This is structurally harder than Phase 3.x.10.y's mask because it spans multiple processes and a network. **Phase 3.x.11.q is the placeholder for that work; it's not in v1 scope.**

**v1 mitigation.** Tier C content is structurally rejected at the sharded dispatch boundary (see §3.4). Tier A + B content's timing surface is documented as honest scope; operators choosing sharded decode for Tier A/B accept the trade.

### 3.2 KV-cache state privacy on stages

**Surface.** `KVCacheHandle.payload` holds per-layer K + V tensors of shape `[batch=1, n_heads, seq_len, head_dim]`. After a 4-token decode on a stage hosting layers 0-2, that stage's cache holds 3 layers × (K + V) × `seq_len = prompt_len + 4` positions. **Each cached K/V tensor is a derived projection of the prompt + generated tokens up to that position.**

An R3 S1 attacker who runs as a stage in the chain sees:
- During each PREFILL: the full prompt's hidden state at their layer-range entry (R3 baseline — unchanged).
- During each INCREMENTAL: incrementally one new position's hidden state at their layer-range entry (R3 baseline — unchanged).
- **NEW**: The accumulated K/V cache covering ALL positions (prompt + generated tokens) for the duration of the request — even across forward passes that only feed in single-position activations.

The KV-cache content is the same kind of activation tensor R3 already considered (a projection through the model), but the *temporal coverage* is different: R3's threat assumed an attacker observes one snapshot per inference request; under sharded decode, the K/V cache aggregates information across the whole decode, sitting in stage memory the entire time.

**Mitigation in v1.**
- **Eviction on terminal.** `EvictCacheRequest` broadcast on every executor exit path (terminal, GeneratorExit, exception) calls `KVCacheManager.evict(request_id)` on every stage. The cache handle is dropped; Python GC reclaims the K/V tensors at the next cycle.
- **TTL sweeper.** `KVCacheManager.evict_idle()` drops handles past `ttl_seconds` (default 300s = 5 min). Bounds zombie-request residue when the executor crashed without an explicit eviction.
- **LRU cap.** When the pool reaches `max_cached_requests` (default 64), the LRU handle is evicted to make room. Bounds aggregate cache memory under load.

**What v1 does NOT do.**
- **No memory wipe.** Eviction drops the handle reference; Python GC eventually reclaims, but the K/V bytes may remain in process heap until reused. An attacker with malicious-process-on-same-host access (R3 S1 with extra capability) could potentially recover residual K/V state via `/proc/[pid]/mem` reads. Operator mitigation: run stages under user namespaces / TEE attestation to prevent cross-process memory reads.
- **No encryption at rest.** K/V tensors live in plain memory. R3 baseline assumed this; sharded decode doesn't change the at-rest state, just extends its duration.
- **No commitment in the receipt.** The signed `InferenceReceipt` commits to the activation bytes that crossed the wire (per Phase 3.x.7 Task 5 envelope) but NOT to the in-stage cache state. A malicious stage that swapped its cache mid-INCREMENTAL could produce a "valid" receipt while serving manipulated K/V — though this is an integrity attack, not a privacy one, and the manipulated K/V would manifest as wrong activation outputs (caught by R7 challenge-and-verify if the operator wires it).

### 3.3 Cross-stage activation handoff magnification

**Surface.** R3 §3.1 enumerated cross-stage handoff as activation-inversion attack surface H1: "Each stage receives the boundary hidden state from the prior stage and could attempt to invert it back to the prompt token sequence." Under Phase 3.x.7 unary, that handoff happens ONCE per request. Under Phase 3.x.11 sharded, **each stage sees the boundary hidden state once per generated token** — `max_tokens` activations per request instead of one.

**Quantitative impact.**
- Phase 3.x.7 unary: 1 boundary hidden state observation per request. `max_tokens` is irrelevant on the unary path because the tail does the full autoregressive decode locally.
- Phase 3.x.10 streaming-tail: 1 boundary hidden state observation per request. Same as 3.x.7 — non-tail stages run once during prefill.
- **Phase 3.x.11 sharded: `1 + max_tokens` boundary hidden state observations per request.** PREFILL gives the attacker one full-prompt activation; each INCREMENTAL gives one single-position activation. Across many requests, an attacker accumulates an N-times-larger corpus.

**R3 mitigation strength under sharded.** R3's mitigations (topology randomization, DP noise injection, TEE attestation) all carry forward, but their effective strength against attacker scale changes:

| R3 Mitigation | Phase 3.x.7 strength | Phase 3.x.11 strength |
|---|---|---|
| Topology randomization | Strong (one observation per request) | **Weaker** (attacker on a stage observes `1+max_tokens` consecutive activations per request — within-request correlation preserved) |
| DP noise injection (Ring 9) | Strong (noise budget per inference) | **Same per-noise-event, but `1+max_tokens` more events per request** — privacy budget consumes faster |
| TEE attestation (Ring 7-10) | Strong (TEE seal at each handoff) | **Same** (TEE doesn't care about handoff frequency; per-handoff attestation cost rises linearly with `max_tokens`) |

**v1 mitigation.** None beyond R3 baseline. Operators serving sharded decode accept the magnification cost as honest scope. R3-style red-team evaluation against sharded decode is research (R3-derived; not v1 engineering scope).

### 3.4 Tier C is structurally incompatible with v1 sharded decode

**Surface.** Tier C (encrypted/private content) has the strongest privacy guarantee in PRSM's content-tier hierarchy. Phase 3.x.10.y wired Tier C streaming through M1/M2 constant-time padding decorators that mask per-token timing. Sharded decode introduces a new timing surface (§3.1) that those decorators don't cover.

**Decision.** Tier C dispatches on the sharded path are STRUCTURALLY REJECTED at the dispatch boundary in `LayerStageServer._dispatch_sharded`:

```python
if request.content_tier == ContentTier.C:
    return self._error(
        request.request_id,
        StageErrorCode.INTERNAL_ERROR,
        "sharded decode does not yet support Tier C "
        "(per-token wire dispatch creates a new timing "
        "surface; see Phase 3.x.11.q honest scope)",
    )
```

This is a default-deny gate matching the Phase 3.x.10.y pattern (Tier C streaming without a constant-time decorator → INTERNAL_ERROR). Operators serving Tier C content keep using:
- Phase 3.x.7 unary path, OR
- Phase 3.x.8 streaming-tail with `enable_sharded_decode=False`.

**Future work (Phase 3.x.11.q).** A fresh memo will scope:
- Per-stage constant-time padding (mask per-stage forward time within the sharded chain).
- Per-hop wire-latency padding (mask inter-stage network jitter).
- Whole-chain per-token cadence enforcement (top-level rate limiter that pads the chain's effective per-token wall-clock to a constant).

Until that lands, Tier C operators MUST NOT enable `enable_sharded_decode`. The dispatch gate is the load-bearing enforcement.

### 3.5 Per-iteration accept-rate timing surface (Phase 3.x.11.y speculation)

**Surface.** Phase 3.x.11.y adds speculative decoding: a co-located draft model proposes K candidate tokens per round; the sharded chain verifies them in a single batched K+1-position forward; the executor accepts the longest matching prefix and broadcasts a `RollbackCacheRequest` for the rejected suffix. **The acceptance count per round is observable on the wire — both directly (the tail's `accepted_count` field on `RunLayerSliceResponse`) and indirectly (the `n_positions_to_drop` field on each round's `RollbackCacheRequest` broadcast = K - accepted_count).**

A network observer or a stage operating any chain stage learns the distribution of acceptance counts across a request. Acceptance rate correlates **weakly** with prompt content: prompts that the draft and verifier agree on (typical natural-language continuations, common idioms, deterministic completions) yield high accept-rates; ambiguous or out-of-distribution prompts yield low accept-rates.

**Comparison to §3.1 baseline.** §3.1 already documented the per-token wire timing surface (T network round-trips per token). Speculation amplifies the surface in a different direction: instead of O(token-count) time observations, it adds O(token-count) integer-valued accepted_count observations. Each is a discrete value in `[0, K]`, easier to reason about and lower-entropy than continuous timing measurements, but directly correlated with which-token-is-being-decoded in a way that timing isn't.

**Quantitative impact.** For an 8-token request with K=4:
- Pre-speculation (Phase 3.x.11): 8 wire-timing observations per stage × T stages = 8T per request.
- Post-speculation (Phase 3.x.11.y): ~2-3 VERIFY rounds (perfect-accept emits 5 tokens per round) × T stages = ~2-3T wire-timing observations PLUS 2-3 accepted_count integers in `[0, 4]`.

The accept-rate channel reveals less than the timing channel under typical SNR but is more interpretable: a single accepted_count=0 round on a non-Tier-C request leaks "the draft and verifier disagreed on the next token here" — that's a known-plaintext-style leak about prompt difficulty, not prompt content per se.

**v1 mitigation.** Tier C content remains structurally rejected at the sharded dispatch boundary (§3.4 carries forward; the gate fires before any VERIFY decoding). Tier A + B content's accept-rate surface is documented as honest scope; operators choosing speculation for Tier A/B accept the trade in exchange for the 4-6× perf win.

**Why the existing constant-time padding decorators don't help.** Phase 3.x.10.y's M1/M2 decorators mask **inter-token timing**; they don't mask **discrete accept-rate values returned in the wire response**. A constant-time decorator for speculation would need to:
- Pad each VERIFY round to constant duration regardless of accepted_count (mask the per-round forward time variance).
- Either (a) return constant K every round and emit verifier-correction tokens as if all drafts were accepted (expensive — wastes the perf win), or (b) batch multiple VERIFY rounds and only emit accepted_count distributions over a coarser time window (delayed observation).

Both options materially weaken the speculation perf gain. **Phase 3.x.11.q.y is the placeholder for that work; it's not in v1 scope.** v1 ships honest-scope: speculation is a Tier A/B perf optimization, Tier C operators continue using single-token sharded decode (which itself remains gated until Phase 3.x.11.q).

**Greedy-only invariant.** v1 speculation rejects `temperature > 0` at the executor's sharded-speculative path entry (`_execute_chain_streaming_sharded_speculative` raises `ChainExecutionError(code=PROMPT_ENCODE_ERROR)` on positive temperature). This is for two reasons:
1. **Sampling correctness.** Speculative sampling under nonzero temperature requires the [Leviathan et al. 2023] rejection-sampling correction; without it, the output distribution drifts away from the verifier's distribution, breaking the "speculation is a perf optimization, not a sampling change" invariant. v1 ships honest-scope; sampling-correct speculation under temperature > 0 is Phase 3.x.11.y.x.
2. **Threat-model containment.** Greedy speculation produces output BIT-IDENTICAL to non-speculative greedy decode for the same prompt. The new threat surfaces (accept-rate channel above) are quantifiable additions to the existing single-token sharded baseline; under temperature > 0, the output diverges and the threat-surface comparison becomes harder to reason about. v1's greedy gate keeps the threat-model story crisp.

**Side note on rollback broadcast.** `RollbackCacheRequest` is sent best-effort (no signature, no executor-side ack-failure remediation). A network adversary who drops rollback broadcasts can cause stage caches to grow unbounded (until the TTL sweeper or LRU cap fires). This is a DoS surface, not a privacy or integrity surface — Phase 3.x.11.y honest-scope; an authenticated rollback envelope (mirroring HandoffToken) is deferred to Phase 3.x.11.y.x or later if operator telemetry shows the DoS is exploited.

### 3.6 Sampling-correct speculation under temperature > 0 (Phase 3.x.11.y.x)

**Status update (Phase 3.x.11.y.x).** Phase 3.x.11.y's `temperature > 0` raise is **lifted**. The executor now routes between two paths based on `request.temperature`:

- `temperature == 0.0` → v1 (greedy) path. Bit-identical regression preserved; `apply_lm_head_and_sample_batch` returns K+1 argmaxes; the §3.5 threat surface above applies unchanged.
- `temperature > 0.0` → v2 (Leviathan-2023 stochastic) path. The draft model returns `(proposed_ids, proposed_probs)` via `propose_with_probs`; the wire carries `proposed_token_probs` co-set with `proposed_token_ids`; the tail's `apply_lm_head_and_sample_batch_with_rejection` performs §2.2 rejection sampling and returns `accepted_count + 1` verified entries (last is rejection-sampled correction OR bonus). **Marginal-output guarantee.** Under Option C.1 (q treated as point mass on d_i with mass q(d_i)), the §2.2 marginal-equals-target-p invariant holds **exactly** in the degenerate regime (q(d_i) = 1.0 — pure greedy draft) and **approximately** under stochastic drafts (q(d_i) < 1.0). The drift between the C.1 marginal and the verifier's true target distribution increases as draft entropy increases. Empirically validated for the K=1 q=1.0 case (5000 trials, atol 0.025 vs target) and the K=4 q=1.0 case (5000 trials, atol 0.025 vs target — exercises multi-position partial-accept paths). The non-degenerate-q drift is pinned numerically in `test_option_c1_drift_under_stochastic_q_documented` for the audit trail.

**Threat-surface deltas vs. §3.5.**

1. **Accept-rate channel narrows under stochastic dispatch.** Under v1 greedy, `accepted_count` was a near-deterministic function of (prompt, draft_quality) — every observation leaked correlated information about the prompt's ease-of-prediction. Under v2 stochastic, `accepted_count` becomes a random variable with `E[accept] = E_t~q[min(1, p(t)/q(t))] = Σ min(p(t), q(t))` (the Leviathan-2023 expected-acceptance formula). The accept-rate distribution is now a function of (1) prompt content (as before) AND (2) the verifier's temperature. An observer who knows T can in principle invert the channel — but the inversion is noisier than the deterministic v1 case. Operators choosing T > 0 are accepting MORE noise on the accept-rate channel for the same wire-level cost.
2. **`proposed_token_probs` on the wire is a new content-correlated surface.** Each VERIFY request now ships K floats — the draft's per-token confidence at d_1..d_K. These leak draft-distribution information to the chain stages; for prompts where the draft is highly confident (low-entropy q), this is a near-bit-for-bit leak of WHICH tokens the draft proposed. The mitigation is identical to §3.4: Tier C is structurally rejected at the sharded dispatch boundary; Tier A/B operators accept the leak. **No new structural mitigation in v1; constant-time speculation (Phase 3.x.11.q.y) is where this gets masked.**
3. **Adaptive K is a new content-correlated surface.** The executor's rolling-window adaptive K (halve on accept-rate < 25%, double on > 75%, cap at MAX_VERIFY_BATCH_TOKENS - 1) means the K-value used in subsequent rounds depends on previous rounds' accept-rates — which depend on prompt content. An observer who can correlate `proposed_token_ids` length across consecutive VERIFY requests learns coarse-grained "this request kept getting low accept-rate" signal. **This is intentional honest-scope** in the v1 threat model: the perf gain from adaptive K (10-30% throughput improvement on hard prompts) outweighs the marginal additional surface beyond what `accepted_count` already exposes. Operators wanting flat-K behavior can configure `speculation_depth` and we hold K constant — but lose the perf win.

**v2 mitigation summary.**
- Tier C structural deny carries forward (§3.4 + §3.5 unchanged).
- Greedy-only-at-T=0 invariant relaxed; v1 (greedy) remains an opt-in for operators who don't want the v2 surfaces above.
- Constant-time decorators STILL don't cover the accept-rate / proposed_token_probs / adaptive-K surfaces — Phase 3.x.11.q.y is the bundled placeholder.

**Backwards-compat at the seam.** Phase 3.x.11.y deployments without v2 capability (no `propose_with_probs` on draft, no `apply_lm_head_and_sample_batch_with_rejection` on tail) keep working unchanged at T = 0. New executor + old runner: v1 traffic flows; v2 traffic surfaces `MALFORMED_REQUEST` from the server (clear "upgrade required" signal — not a silent fallback to v1, which would violate the temperature contract).

**v2 mitigation gap (carry-forward).** The accept-rate channel under temp > 0 is fundamentally weaker than the v1 channel as a privacy leak (per §3.6.1) — but `proposed_token_probs` (§3.6.2) is strictly STRONGER than anything in v1 because it ships K floats per VERIFY round. v1 operators concerned about §3.4-style leaks were already gated to Tier A/B; v2 carries the same deny on Tier C; no new operator advisory beyond §3.5's. Phase 3.x.11.q.y is where the encrypted/padded variant of `proposed_token_probs` lands.

**`proposed_token_probs` end-to-end authentication.** `RunLayerSliceRequest.signing_payload` was extended to commit `proposed_token_probs` bytes — the executor's intent is committed via the request-level signing payload. However, a relay between executor and tail can substitute the bytes in transit BEFORE the request is parsed (the `HandoffToken` covers `(request_id, settler_node_id, chain_stage_index, chain_total_stages, deadline_unix)` only — NOT the full request bytes). The same exposure already existed for `proposed_token_ids` in Phase 3.x.11.y; this slice doesn't introduce a NEW substitution surface, but does enlarge the existing one (K floats added). **Honest scope:** end-to-end request authentication via a `request_signature` field on `RunLayerSliceRequest` is deferred (Phase 3.x.11.y.x' or later). Until then, this surface is mitigated by the existing trust model: chain stages are anchor-verified (publisher-key-anchor lookup), so the relay-layer adversary is the network observer rather than a stage operator with valid identity. The Tier C structural deny still applies; Tier A/B operators accept the relay-substitution surface under the same model Phase 3.x.11.y operated under.

### 3.7 Tier C sharded decode under chain-level constant-time padding (Phase 3.x.11.q)

**Status update (Phase 3.x.11.q).** The "Tier C structurally denied at the sharded dispatch boundary" line in §3.4 + §3.5 + §3.6 is now lifted **for the executor → caller wire** when one of two operator-wired chain-level decorators is in scope. Tier C content can flow through sharded autoregressive decode at Tier A/B perf characteristics, with the per-token timing surface masked at the chain-executor boundary. **Two decorators (mirrors §7.8 single-host pattern):**

- **`BatchedTrailingShardedExecutor` (M2)** — drains the inner executor's full stream, emits ONE `StreamToken` carrying the joined text + ONE `ChainExecutionResult`. From a wire observer on the executor → caller path, exactly two events appear regardless of inner per-token cadence. Sacrifices streaming UX for maximum executor-wire leak elimination.
- **`FixedRateShardedExecutor` (M1)** — yields per-token `StreamToken`s at fixed wall-clock cadence. The chain runs at native speed; the decorator's `yield` is what gates emission. Inter-StreamToken intervals are clamped to ≥ cadence regardless of per-token chain compute variance. Operator MUST set cadence ≥ chain native rate (recommended: 2× measured native).

**Routing.** `ParallaxScheduledExecutor.execute_streaming` routes by `request.content_tier`:
- Tier A/B → default `chain_executor` (unchanged from Phase 3.x.11.y.x)
- Tier C + `tier_c_chain_executor` wired → decorator
- Tier C + decorator unwired → structured `InferenceResult.failure(...)` with operator-fixable message naming `make_tier_c_sharded_executor`. **No silent fallback to the leaky path.**

**The per-stage wire is NOT masked.** This is the load-bearing scope-honesty point of the Phase 3.x.11.q slice. The chain-executor decorator wraps the executor → caller path; the executor → per-stage path continues to dispatch at chain native rate. A network observer with visibility into a single stage's transport learns the raw per-token cadence — the §3.1 timing surface is intact AT THE STAGE LEVEL.

For full-network masking, operators compose with per-stage cadence wrappers (Phase 3.x.11.q.x deferred — analogous to §7.8 M1 wrapping `StreamingLayerRunner` on each stage). The chain-level slice covers the highest-value threat (caller-observable timing) without the per-stage coordination problem.

**Per-stage `_dispatch_sharded` TIER_GATE deny stays as defense-in-depth.** A misconfigured executor that tried to send Tier C dispatches directly to stages still gets rejected at the stage. The chain-level decorator is the trust-policy boundary, not the per-stage server.

**Speculation under Tier C** — see §3.8 (Phase 3.x.11.q.y). The `tier_c_speculation_enabled` constructor flag on `ParallaxScheduledExecutor` is the operator's opt-in switch; until set, Tier C + temperature > 0 still surfaces a structured failure at the routing layer.

### 3.8 Constant-time speculation under Tier C (Phase 3.x.11.q.y)

**Status update (Phase 3.x.11.q.y).** The "Speculation + Tier C remains denied" line in §3.7 is now lifted **on the executor → stage wire** when the operator wires the full Phase 3.x.11.q.y opt-in stack. The slice closes two of the three §3.6 wire surfaces and structurally narrows the third:

- **`encrypted_proposed_token_probs`** (§3.6.2 closed). The K float draft probs ride encrypted under AES-GCM with AAD = `(request_id || stage_index)`. Operators distribute the PSK out-of-band; HKDF-SHA256 derives the AES-256 key. A network observer at the wire learns ciphertext bytes only — the plaintext probs are no longer wire-visible. Mutual-exclusion invariant on `RunLayerSliceRequest`: `proposed_token_probs` and `encrypted_proposed_token_probs` cannot both be set; co-set with `proposed_token_ids`. Wrong key / wrong AAD → server surfaces `MALFORMED_REQUEST` (no silent fallback).
- **`verified_token_ids` constant-K commitment** (§3.6.1 narrowed at the response surface). Tail-runner `constant_k_commitment=True` pads `verified_token_ids` to K+1 entries regardless of `accepted_count`. The wire-side response no longer leaks the actual acceptance count via the list length. Position 0 retains the §2.2 marginal-equals-target invariant (load-bearing); positions ac+1..K are greedy-argmax continuations from the v1 helper. **Honest-scope narrowing:** the multi-position §2.2 marginal claim in §3.6.1 (positions 1..K) DOES narrow under constant-K — when `accepted_count < K`, positions ac+1..K are not §2.2-correct samples but argmax fillers. The per-token output emitted to the user remains §2.2-correct because the executor honors the actual `accepted_count` field for emission; the constant-K commitment only addresses the *wire-observability* of acceptance, not the user-facing output distribution.
- **`flat_k_mode`** (§3.6.3 closed). When set on `RpcChainExecutor`, the adaptive-K state machine is gated off. Every speculation round commits exactly `speculation_depth` draft tokens on the wire regardless of rolling acceptance history. The cross-round correlation channel is closed at the cost of the perf win from runtime K adaptation.
- **`tier_c_speculation_enabled` routing-layer gate.** `ParallaxScheduledExecutor` denies Tier C + temperature > 0 by default with a structured failure pointing operators at the full opt-in contract (encrypted probs cipher + flat-K + tail constant-K). The flag is the deploy-time policy boundary; the runtime invariants are enforced at the per-stage server (cipher decrypt) and the per-stage runner (K+1 padding).

**The ResidualLeak Channel — RollbackCacheRequest n_positions_to_drop.** The Phase 3.x.11.q.y v1 scope explicitly does NOT close the *acceptance-count via rollback drop value* leak. When `accepted_count < K`, the executor emits a `RollbackCacheRequest` with `n_positions_to_drop = K - accepted_count` to align the per-stage KV-cache. A passive wire observer can read the drop value and recover the accepted_count. **This is the load-bearing scope-honesty point of the v1 slice.** Closing it requires either: (a) always issuing a constant-K rollback then re-running the accepted prefix (compute + transport tax), or (b) re-architecting the cache-alignment protocol to push the rollback decision server-side (deeper change). Both deferred to Phase 3.x.11.q.y' (or accepted as honest-scope residual leak under the threat model where rollback frequency under flat-K is itself partial information; the operator can additionally mask via Phase 3.x.11.q chain-level decorators on the executor → caller wire). The E2E pin (`tests/integration/test_phase3_x_11_q_y_constant_time_speculation_e2e.py::test_e3_residual_rollback_leak_is_documented`) ASSERTS the leak's wire presence so accidental quiet closure is caught as a regression that requires this section to be updated.

**Per-stage TIER_GATE deny stays as defense-in-depth.** Phase 3.x.11.q's `_dispatch_sharded` Tier C gate continues to reject Tier C dispatches that arrive at the per-stage server without the routing-layer-attested opt-in. The Phase 3.x.11.q.y opt-in changes the executor's *outgoing* wire (encrypted probs, flat-K, constant-K response shape) but does NOT relax the per-stage gate; that gate remains the trust boundary.

**Composition with §3.7 chain-level decorators.** The Phase 3.x.11.q decorators (M1 cadence-driven OR M2 single trailing frame) wrap the executor → caller path; this slice's encrypted/padded constant-K stack covers the executor → stage path. Operators MAY wire both: Tier C + temperature > 0 + `tier_c_speculation_enabled=True` + `tier_c_chain_executor=make_tier_c_sharded_executor(...)`. The composition's load-bearing residual leak is the rollback-drop channel above, plus the §3.7 honest-scope items (per-stage wire still leaks per-token timing; total stream duration leaks total token count under M1; total response size leaks joined-text length under M2).

**Honest scope (Phase 3.x.11.q.y v1 limits).**

1. **Rollback drop-value leak** (above). Phase 3.x.11.q.y'.
2. **Per-stage timing leaks at native chain rate** — inherited from §3.7, not addressed by this slice. Phase 3.x.11.q.x for full-network masking.
3. **PSK distribution is operator-managed.** No on-wire DH negotiation in v1; ECDH-piggybacked-on-`HandoffToken` deferred to Phase 3.x.11.q.y' to avoid changing `HandoffToken`'s wire format. PSK rotation = redeploy.
4. **Multi-position §2.2 marginal narrowing** (above). The user-facing output stays §2.2-correct because the executor emits according to `accepted_count`, but auditors reading the response field semantics should note that positions ac+1..K of `verified_token_ids` are NOT §2.2 samples under constant-K.
5. **Adaptive K under flat-K is OFF** (above). Operators trade perf for the closure of §3.6.3.

**Honest scope (chain-level decorator limits).**

1. **Per-stage wire still leaks** (above). Phase 3.x.11.q.x for full-network masking.
2. **Total stream duration leaks total token count under M1.** Cadence × frame count = duration. Operators cap `max_tokens` per Tier C request; the leak ceiling becomes `max_tokens × cadence`.
3. **Total response size leaks total joined-text length under M2.** A single big terminal frame leaks the cumulative output byte count. Operator-configurable padding to fixed max-length deferred to Phase 3.x.11.q.x.
4. **Speculation under Tier C** — Phase 3.x.11.q.y bundle.
5. **Adaptive cadence based on observed network load** — Phase 3.x.11.q.z.
