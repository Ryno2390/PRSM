## 3. Threat model (auditor priorities)

### 3.1 `_handleConsensusMismatch` with `consensus_group_id` binding

Priority verification questions:

1. **Check ordering** ‚Äî is the guard order correct (self-reference ‚Üí group_id non-zero ‚Üí job/shard/hash binding ‚Üí conflicting batch exists ‚Üí group_id match + distinct provider ‚Üí Merkle verify)? Cheapest checks first, expensive last. Any order-dependent assumption I'm missing?

2. **Group_id prediction attacks** ‚Äî a `bytes32` is 256 bits; brute-force collision is infeasible. But if the off-chain dispatch uses a weak derivation (e.g., `keccak(job_id || shard_index)` with a predictable job_id), an attacker could pre-commit a batch with the "right" group_id before the legitimate dispatch. The Phase 7.1 E2E uses `keccak(f"consensus-group-{job_id}".encode())`. Should the derivation include a requester-controlled random nonce to harden against pre-commit? Note: even with predictable group_id, the `b.provider != other.provider` check still forces the attacker to control k distinct provider keys ‚Äî pre-commit alone isn't a full attack.

3. **Open third-party auth** ‚Äî removing the requester-only check opens the 70% bounty to any observer. Is there any griefing vector this enables that the requester-only MVP prevented? Specifically ‚Äî can a third party with access to both committed receipts but no stake at risk trigger a challenge that harms the legitimate requester (e.g., by consuming the slashable target before the actual requester could)?

4. **Storage layout** ‚Äî `Batch.consensus_group_id` is inserted between `tier_slash_rate_bps` and `metadataURI`. Greenfield deploy (no mainnet state); layout is fresh. Audit-flag as such but no migration risk.

### 3.2 `MIN_SLASH_GAS` floor

Priority questions:

1. **Floor value calibration** ‚Äî 150,000 gas sits ~50K above Solidity's 63/64 forwarding requirement for a ~200K slash path (actual slash cost observed at 110K‚Äì160K in tests + gas report). Is 150K the right floor for current StakeBond, or should it be higher to handle future storage layout changes (e.g., a new event field, an additional SSTORE)?

2. **Catch-path preservation** ‚Äî the `try { stakeBond.slash(...) } catch {}` is preserved. The floor only excludes the OOG case; `NotSlashable` (provider withdrew) and `NothingToSlash` (stake fully drained) still silently succeed through the catch. Is that the intended semantics for all three slashable reason codes (DOUBLE_SPEND, INVALID_SIGNATURE, CONSENSUS_MISMATCH)?

3. **No `max_gas` symmetric cap** ‚Äî the floor enforces a minimum; there's no ceiling. A very high gas tx could (in principle) do more work inside the catch block than expected. Low-priority but worth noting.

### 3.3 Python-layer concurrency (`ConsensusChallengeQueue`)

Priority questions:

1. **Single-process invariant** ‚Äî the docstring explicitly says "one submitter process per DB path." Is that a strong enough invariant given that `sqlite3` itself can't enforce it, and operators might accidentally run two submitters against the same DB? If multi-process is a concrete operational concern, a claim-lease column (`claimed_at`, `claimed_by`) with timeout-based re-lease would be the standard fix. See ¬ß5.3.

2. **Lock scope** ‚Äî `_lock = threading.Lock()` is held across each multi-statement operation (e.g., `record_batch_commit` does 3 UPDATEs + 1 promotion-SQL + commit under one lock). No nested lock acquisitions. No operations that could deadlock.

3. **`process_submittable_queue` fetch-then-act** ‚Äî `list_submittable` returns rows under the lock; lock is released before `submit_one` (which does network I/O); `mark_submitted` / `mark_failed` re-acquires the lock. A second runner fetching SUBMITTABLE rows in parallel could double-submit the same row. Known hazard, single-process invariant covers it. See ¬ß5.3.

### 3.4 Runner retry classification

Priority questions:

1. **Allowlist completeness** ‚Äî only `BroadcastFailedError` is retryable. Are there other genuinely-transient error types we should allow? (The submitter intentionally raises typed Web3 errors; all other failure types represent either contract rejection, tx-may-have-landed uncertainty, or genuine bugs.)

2. **`max_attempts = 5` default** ‚Äî arbitrary; tuned against "transient blips should clear within 1‚Äì2 ticks, persistent failures should escalate quickly." Is 5 the right value for expected operator cadence?

3. **Attempt-counter persistence** ‚Äî `attempts` is stored on the row; a process restart preserves the count. Which means a new runner instance inherits the retry budget. Correct behavior (prevents infinite retries across restarts).

## 5. Known issues carried into the audit bundle

Ranked by auditor priority. None are merge-blockers; all are documented follow-ups.

### 5.1 `MIN_SLASH_GAS` re-benchmark policy (P4 from cumulative review)

`MIN_SLASH_GAS = 150_000` is a constant in `BatchSettlementRegistry`. If a future `StakeBond` refactor adds storage writes or event fields, the slash cost could push above ~200K and reduce the 63/64-forwarded gas below what the slash needs to complete. Current comment on the constant flags the assumption but doesn't document a re-benchmark policy.

**Auditor: please suggest a re-benchmark trigger** (e.g., "run `hardhat test --gas-reporter` on any `StakeBond.slash` change and confirm peak gas < `MIN_SLASH_GAS`").

### 5.2 `SettlementContractClient.commit_batch` abstract is behind (P2)

`prsm/settlement/client.py` declares an abstract `commit_batch` with the **Phase 3.1** signature (6 params). Phase 7 added `tier_slash_rate_bps`; Phase 7.1x ¬ß8.7 added `consensus_group_id`. The abstract doesn't carry either, so any concrete implementation would commit with stale defaults.

**Current state:** the E2Es bypass this abstract entirely ‚Äî they call `challengeReceipt` / `commitBatch` via direct web3 contracts. Nothing production-critical depends on the abstract today.

**Disposition:** acknowledge as known divergence. When the settlement accumulator becomes the on-chain commit path (Phase 7.1x.next+), the abstract + any concrete impl will be brought up to date. Not a blocker for the tag.

### 5.3 `ConsensusChallengeQueue` single-process invariant (P3)

The SQLite queue is thread-safe within one process but has no multi-process coordination. Two runners sharing one DB path could double-submit SUBMITTABLE rows.

**Docstring explicitly documents this as the operational invariant.** The planned fix is a claim-lease column (`claimed_at`, `claimed_by`) with timeout-based re-lease ‚Äî standard pattern for shared job queues. Not a blocker for single-operator deployments; flag for multi-operator deployments.

### 5.4 Phase 7 ¬ß8.8 cross-process nonce-race (unchanged)

Not a regression from Phase 1.1. Operator-runbook-mitigated via `OPERATOR_GUIDE.md` ¬ßOn-chain Keypairs ("one keypair per OS process"). Phase 7's extra Web3 client widens the exposure factor by 1 but the mitigation pattern stands.
