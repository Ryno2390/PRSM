## 1. Why this addendum exists

Phase 3.x.11 closes the tail-only contract that's been load-bearing since Phase 3.x.8: each chain stage now runs its layers *once per generated token*, with KV-cache state surviving locally between iterations, and activations crossing the wire at every per-token boundary.

This is a substantial architectural shift. Three threat surfaces change shape under sharded decode and require explicit re-characterization:

1. **Per-token wire dispatch** introduces a NEW timing observation surface that the existing Phase 3.x.10 memo doesn't model.
2. **KV-cache state on stages** is privacy-sensitive (encodes the prompt + generated tokens) and lives across multiple network-observable events.
3. **Cross-stage activation handoff frequency** multiplies the activation-inversion surface from R3 by a factor of `max_tokens` (per-request).

Phase 3.x.10's tail-only autoregressive decode is unchanged by this addendum — its threat model still applies when operators run with `enable_sharded_decode=False`. Sharded decode is OPT-IN; this addendum applies only to the opt-in path.

## 2. Threat model deltas vs. R3 baseline

### 2.1 What the attacker still controls (R3 §2.2 unchanged)

- Their SPRK's runtime + memory.
- Their node's network interface.
- Timing + ordering of their own responses.

### 2.2 What the attacker NEWLY observes under sharded decode

- **Per-token wire arrivals.** A network observer on any segment of the chain sees one `RunLayerSliceRequest` per stage per generated token. For a `T`-stage chain producing `N` tokens, that's `N × T` observable wire events per request — vs. exactly `1` in the Phase 3.x.7 unary path and `N` in the Phase 3.x.8 streaming-tail path (tail-only emits N TokenFrames).
- **Per-token activations crossing each stage boundary.** Each stage forwards its boundary hidden state as the next stage's input; the hidden-state tensor is observable on the wire (encrypted under TLS at the transport layer per Phase 6, but visible to anyone with TLS termination — including the receiving stage itself).
- **KV-cache state on each stage that hosts the request.** The cache lives in the stage's process memory across the full decode loop; an attacker with read access to the stage's memory (R3 S1 attacker on that specific stage) sees not just one activation but ALL cached K/V tensors covering every token position so far.
- **Cache-eviction signals.** `EvictCacheRequest` envelopes carry the request_id in cleartext at the wire boundary (no signature on the eviction path — see honest scope in §3.6 of design plan).

### 2.3 What the attacker still does NOT control (R3 §2.3 unchanged + additions)

- Other SPRKs' memory (cache state on a stage Mallory doesn't control stays opaque to her).
- Topology decisions before dispatch.
- TEE-sealed attestation roots.
- **NEW**: Eviction race resolution. `KVCacheManager.evict` is idempotent — duplicate broadcasts (e.g., from retried close paths) and TTL-sweeper races resolve cleanly without exposing the attacker any extra state.

### 2.4 Attacker scale revisited

R3's S1 / S2 / S3 scales remain the right framing, but the per-request observation surface MULTIPLIES under sharded decode:

| Scale | Phase 3.x.7 unary | Phase 3.x.8 streaming-tail | Phase 3.x.11 sharded |
|---|---|---|---|
| S1: one shard, one layer, one request | 1 activation observation | 1 activation + N text deltas | **1 + N** activation observations + KV-cache memory residue |
| S2: one shard across many requests | corpus of 1×R activations | corpus of 1×R + N×R deltas | **corpus of (1+N)×R** activations + R caches |
| S3: multi-shard coordinated | (per-shard S2)×K colluders | (per-shard S2)×K colluders | **(per-shard S2)×K colluders, each with N× the per-request surface** |

The S2 and S3 deltas are the load-bearing concern. A long-lived attacker on a sharded-decode-serving node accumulates `(1 + max_tokens) × num_requests` activation observations — for `max_tokens=512` and 1000 requests served per day, that's >513,000 activations vs. ~1,000 under the unary path.

## 3. New threat surfaces (this addendum)

### 3.1 Per-token wire timing surface

**Surface.** Each per-token chain pass produces `T` wire events (T = chain depth). A network observer on any segment of the chain measures inter-event timing. Per-token decode time on each stage is data-dependent (longer prompts → bigger attention K/V → slower per-stage forward; specific token decisions correlate with specific compute paths in attention). **The observer learns timing PER HOP, PER TOKEN.**

**Comparison to Phase 3.x.10 tail-only.** Phase 3.x.10's timing surface is *one* per-token timing per request (tail emits one TokenFrame per generated token; non-tail stages run only once during prefill). Phase 3.x.11 multiplies that by T (chain depth): `N × T` timing observations per request.

**Why the existing constant-time padding decorators don't help.** Phase 3.x.10.y's `BatchedTrailingStreamingRunner` (M2) and `FixedRateStreamingRunner` (M1) operate at the streaming-tail layer — they padd inter-token latency at the *tail* output. Sharded decode dispatches `T` unary RPCs per token, each completing at its own per-stage rate; M1/M2 never see those events because they live above `LayerStageServer.handle` (sharded uses `_dispatch_sharded`, not `handle_token_stream`).

A constant-time decorator for sharded decode would need to:
- Pad per-stage forward time at each stage uniformly (mask per-token decode-time variance).
- Pad inter-stage wire latency between stages (mask network jitter that itself reveals inter-stage compute).
- Guarantee the chain's total per-token wall-clock is data-independent.

This is structurally harder than Phase 3.x.10.y's mask because it spans multiple processes and a network. **Phase 3.x.11.q is the placeholder for that work; it's not in v1 scope.**

**v1 mitigation.** Tier C content is structurally rejected at the sharded dispatch boundary (see §3.4). Tier A + B content's timing surface is documented as honest scope; operators choosing sharded decode for Tier A/B accept the trade.

### 3.2 KV-cache state privacy on stages

**Surface.** `KVCacheHandle.payload` holds per-layer K + V tensors of shape `[batch=1, n_heads, seq_len, head_dim]`. After a 4-token decode on a stage hosting layers 0-2, that stage's cache holds 3 layers × (K + V) × `seq_len = prompt_len + 4` positions. **Each cached K/V tensor is a derived projection of the prompt + generated tokens up to that position.**

An R3 S1 attacker who runs as a stage in the chain sees:
- During each PREFILL: the full prompt's hidden state at their layer-range entry (R3 baseline — unchanged).
- During each INCREMENTAL: incrementally one new position's hidden state at their layer-range entry (R3 baseline — unchanged).
- **NEW**: The accumulated K/V cache covering ALL positions (prompt + generated tokens) for the duration of the request — even across forward passes that only feed in single-position activations.

The KV-cache content is the same kind of activation tensor R3 already considered (a projection through the model), but the *temporal coverage* is different: R3's threat assumed an attacker observes one snapshot per inference request; under sharded decode, the K/V cache aggregates information across the whole decode, sitting in stage memory the entire time.

**Mitigation in v1.**
- **Eviction on terminal.** `EvictCacheRequest` broadcast on every executor exit path (terminal, GeneratorExit, exception) calls `KVCacheManager.evict(request_id)` on every stage. The cache handle is dropped; Python GC reclaims the K/V tensors at the next cycle.
- **TTL sweeper.** `KVCacheManager.evict_idle()` drops handles past `ttl_seconds` (default 300s = 5 min). Bounds zombie-request residue when the executor crashed without an explicit eviction.
- **LRU cap.** When the pool reaches `max_cached_requests` (default 64), the LRU handle is evicted to make room. Bounds aggregate cache memory under load.

**What v1 does NOT do.**
- **No memory wipe.** Eviction drops the handle reference; Python GC eventually reclaims, but the K/V bytes may remain in process heap until reused. An attacker with malicious-process-on-same-host access (R3 S1 with extra capability) could potentially recover residual K/V state via `/proc/[pid]/mem` reads. Operator mitigation: run stages under user namespaces / TEE attestation to prevent cross-process memory reads.
- **No encryption at rest.** K/V tensors live in plain memory. R3 baseline assumed this; sharded decode doesn't change the at-rest state, just extends its duration.
- **No commitment in the receipt.** The signed `InferenceReceipt` commits to the activation bytes that crossed the wire (per Phase 3.x.7 Task 5 envelope) but NOT to the in-stage cache state. A malicious stage that swapped its cache mid-INCREMENTAL could produce a "valid" receipt while serving manipulated K/V — though this is an integrity attack, not a privacy one, and the manipulated K/V would manifest as wrong activation outputs (caught by R7 challenge-and-verify if the operator wires it).

### 3.3 Cross-stage activation handoff magnification

**Surface.** R3 §3.1 enumerated cross-stage handoff as activation-inversion attack surface H1: "Each stage receives the boundary hidden state from the prior stage and could attempt to invert it back to the prompt token sequence." Under Phase 3.x.7 unary, that handoff happens ONCE per request. Under Phase 3.x.11 sharded, **each stage sees the boundary hidden state once per generated token** — `max_tokens` activations per request instead of one.

**Quantitative impact.**
- Phase 3.x.7 unary: 1 boundary hidden state observation per request. `max_tokens` is irrelevant on the unary path because the tail does the full autoregressive decode locally.
- Phase 3.x.10 streaming-tail: 1 boundary hidden state observation per request. Same as 3.x.7 — non-tail stages run once during prefill.
- **Phase 3.x.11 sharded: `1 + max_tokens` boundary hidden state observations per request.** PREFILL gives the attacker one full-prompt activation; each INCREMENTAL gives one single-position activation. Across many requests, an attacker accumulates an N-times-larger corpus.

**R3 mitigation strength under sharded.** R3's mitigations (topology randomization, DP noise injection, TEE attestation) all carry forward, but their effective strength against attacker scale changes:

| R3 Mitigation | Phase 3.x.7 strength | Phase 3.x.11 strength |
|---|---|---|
| Topology randomization | Strong (one observation per request) | **Weaker** (attacker on a stage observes `1+max_tokens` consecutive activations per request — within-request correlation preserved) |
| DP noise injection (Ring 9) | Strong (noise budget per inference) | **Same per-noise-event, but `1+max_tokens` more events per request** — privacy budget consumes faster |
| TEE attestation (Ring 7-10) | Strong (TEE seal at each handoff) | **Same** (TEE doesn't care about handoff frequency; per-handoff attestation cost rises linearly with `max_tokens`) |

**v1 mitigation.** None beyond R3 baseline. Operators serving sharded decode accept the magnification cost as honest scope. R3-style red-team evaluation against sharded decode is research (R3-derived; not v1 engineering scope).

### 3.4 Tier C is structurally incompatible with v1 sharded decode

**Surface.** Tier C (encrypted/private content) has the strongest privacy guarantee in PRSM's content-tier hierarchy. Phase 3.x.10.y wired Tier C streaming through M1/M2 constant-time padding decorators that mask per-token timing. Sharded decode introduces a new timing surface (§3.1) that those decorators don't cover.

**Decision.** Tier C dispatches on the sharded path are STRUCTURALLY REJECTED at the dispatch boundary in `LayerStageServer._dispatch_sharded`:

```python
if request.content_tier == ContentTier.C:
    return self._error(
        request.request_id,
        StageErrorCode.INTERNAL_ERROR,
        "sharded decode does not yet support Tier C "
        "(per-token wire dispatch creates a new timing "
        "surface; see Phase 3.x.11.q honest scope)",
    )
```

This is a default-deny gate matching the Phase 3.x.10.y pattern (Tier C streaming without a constant-time decorator → INTERNAL_ERROR). Operators serving Tier C content keep using:
- Phase 3.x.7 unary path, OR
- Phase 3.x.8 streaming-tail with `enable_sharded_decode=False`.

**Future work (Phase 3.x.11.q).** A fresh memo will scope:
- Per-stage constant-time padding (mask per-stage forward time within the sharded chain).
- Per-hop wire-latency padding (mask inter-stage network jitter).
- Whole-chain per-token cadence enforcement (top-level rate limiter that pads the chain's effective per-token wall-clock to a constant).

Until that lands, Tier C operators MUST NOT enable `enable_sharded_decode`. The dispatch gate is the load-bearing enforcement.

## 4. Mitigation summary (carries forward to operators)

| Threat | v1 mitigation | Honest scope deferral |
|---|---|---|
| Per-token wire timing | Tier C structural deny + operator advisory | Phase 3.x.11.q (constant-time sharded decorators) |
| KV-cache memory residue | Eviction broadcast + TTL sweeper + LRU cap | Process-level memory wipe (operator policy: TEE / namespace) |
| KV-cache cryptographic non-commit | Receipt commits to wire activations only | Phase 3.x.11.x receipt-format extension (per-token attestation chain) |
| Cross-stage activation magnification | R3 baseline (topology randomization, DP noise, TEE) | R3 red-team against sharded scale (research; not engineering scope) |
| Tier C compat | Default-deny at dispatch | Phase 3.x.11.q |

## 5. Auditor reading path

For an external auditor evaluating Phase 3.x.11's privacy posture, the recommended reading order is:

1. **This addendum** (§1 + §3.1 + §3.4) — the new timing + content-tier story.
2. **R3 threat model** (`docs/2026-04-22-r3-threat-model.md` §2-§5) — the baseline activation-inversion model that this addendum extends.
3. **Phase 3.x.10 timing-sidechannel memo** — the tail-only timing story this addendum doesn't replace.
4. **Phase 3.x.10.y design plan + audit-prep §7.8** — the constant-time padding pattern, for context on what an equivalent sharded decorator would look like.
5. **Sharded dispatch gate code** — `prsm/compute/chain_rpc/server.py:_dispatch_sharded` Tier C check.
6. **Cache lifecycle code** — `prsm/compute/chain_rpc/kv_cache.py:KVCacheManager` evict/evict_idle paths.
7. **Eviction broadcast code** — `prsm/compute/chain_rpc/client.py:_evict_cache_on_stages` + `prsm/compute/chain_rpc/server.py:_handle_evict_cache`.
