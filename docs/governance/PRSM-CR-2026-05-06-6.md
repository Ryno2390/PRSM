## 2. Background

This resolution formalizes the recommendation in `docs/governance/2026-05-06-royalty-distributor-push-vs-pull-memo.md`, which was drafted earlier today (2026-05-06) and presented to the council for discussion.

**The fork in the road:** Two `RoyaltyDistributor` variants are in production use:

- **Mainnet** at `0x3E8201B2cdC09bB1095Fc63c6DF1673fA9A4D6c2` (deployed 2026-05-04) — push-payment: funds routed directly to recipient EOAs in the distribute tx.
- **Sepolia** at `0xB790045ff826C76fe02DBc54a6ef0021951Fd892` (deployed 2026-05-05 with the T6.2 D-04 OZ pull-payment refactor) — pull-payment: `claimable[address]` mapping accumulates; recipients call `claim()`.

Both enforce the same canonical 10/2/88 split. The economic mechanic is identical. The fork is purely about settlement.

**Discovery context:** the divergence was found during the 2026-05-06 mainnet bring-up Path A trace when the trace script's `claimable()` reader reverted on mainnet. The on-chain payment had already landed correctly via push; the read-side verification failed because mainnet has no `claimable()` function.

**Trade-off summary** (full details in the underlying memo):

| Dimension | Push (mainnet) | Pull (Sepolia) |
|---|---|---|
| Recipient UX | One tx, immediate finality, no claim step | Two-step (distribute then claim) |
| Aggregate gas | Lower (~150k gas per distribute) | ~2.4x higher (claim adds ~71k gas per recipient) |
| Address-typo recovery | None (funds unrecoverable) | Possible via admin redirect path (would require contract change) |
| Mid-flight recipient compromise | Push routes to compromised address | Pull provides intervention window |
| Zombie balances | None (no escrow) | Possible (unclaimed forever if recipient loses key) |
| Contract surface | Smaller (less code, less audit area) | Larger (claimable mapping + claim() function) |

## 3. Resolved

The council hereby resolves:

**RESOLVED 1.** The push-payment settlement pattern, as currently deployed on Base mainnet at `RoyaltyDistributor` `0x3E8201B2cdC09bB1095Fc63c6DF1673fA9A4D6c2`, is **adopted as the production policy** for FTNS royalty settlement on Base mainnet. No redeploy or upgrade to a pull-payment variant is authorized at this time.

**RESOLVED 2.** The Sepolia pull-payment variant at `0xB790045ff826C76fe02DBc54a6ef0021951Fd892` is **retained as a live-comparison reference**. It is not retired or redeployed for parity. This preserves the engineering investment in the T6.2 D-04 refactor and allows ongoing measurement of the operational divergence between the two patterns.

**RESOLVED 3.** Trace tooling (`scripts/exercise_canonical_workflow_base_sepolia.py`) shall continue to support **both variants** via the variant-detection logic introduced 2026-05-06 (commit `0a0b8dba`). This dual-support is not contingent on this resolution and shall remain in place even if a future resolution unifies the two networks.

**RESOLVED 4.** A **90-day re-evaluation trigger** is established. The council shall reconsider this resolution within 90 days of adoption (by 2026-08-04) if any of the following occur:

  - One or more documented incidents of address-typo-loss on mainnet `RoyaltyDistributor` (creator or serving-node royalty going to a wrong recipient because of upstream error).
  - One or more documented incidents of mid-flight recipient compromise where a pull-pattern redirect path would have prevented loss.
  - A material change in Base mainnet gas economics (e.g., 10x gas-price spike that shifts the gas-cost calculus).
  - A specific governance event requiring contract-level pause/revoke of pending royalties (e.g., regulatory injunction, fork-recovery scenario).

**RESOLVED 5.** If by 2026-08-04 none of the §4 triggers have fired, the policy **matures into the steady-state production posture** without requiring an additional resolution. A reaffirmation note shall be appended to this document on or before that date in the §6 status record.

**RESOLVED 6.** This resolution does NOT preclude future adoption of the pull-payment pattern. A future council resolution can mandate redeploy + migration if §4 triggers fire or if the council otherwise determines the trade-offs have shifted.

## 4. Conditions

This resolution is contingent on:

- The push-vs-pull memo (`docs/governance/2026-05-06-royalty-distributor-push-vs-pull-memo.md`) remaining in tree as the underlying analysis.
- The trace tooling's variant-detection logic remaining functional. Any regression that breaks variant detection requires an immediate council notice (not a new resolution, but a status update on this document).

## 5. Explicit non-scope

This resolution does NOT:

- Change the canonical 10/2/88 split or any underlying bps constants (`NETWORK_FEE_BPS = 200`, `MAX_ROYALTY_RATE_BPS = 9800`).
- Modify `networkTreasury` (immutable on the deployed contract).
- Redeploy `RoyaltyDistributor` on either network.
- Add an admin redirect / rescue path to either contract (would require new code + redeploy).
- Govern the FTNS supply, role administration, or treasury operations (those are governed by PRSM-CR-2026-05-06-3, PRSM-POL-1, and the FTNS contract itself).
- Govern the audit-clearance posture for `RoyaltyDistributor` itself (PRSM-POL-2 governs that — agent-teams self-audit applies if a redeploy is ever proposed under POL-2 §4).
