## 1. Problem statement

Phase 3.x.10 shipped the `AutoregressiveStreamingRunner` — real autoregressive decode against `transformers.AutoModelForCausalLM` — and proved it works end-to-end against real distilgpt2 + a server-wired joined-text-invariant regression test. The runner is a structural drop-in for `SyntheticStreamingRunner` and the audit-prep §7.6 + threat-model memo are in place.

But round-1 review surfaced two honest-scope gaps that we documented as 3.x.10.x deferrals rather than blocking the tag:

**M4 — Server doesn't plumb `request=` to the runner.** `LayerStageServer.handle_token_stream` (`prsm/compute/chain_rpc/server.py:834-840`) calls the runner with no `request=` kwarg. The runner accepts `request: Any = None` and falls back to `SamplingDefaults` (max_tokens=512, temperature=1.0, top_k=50, top_p=0.95) when None. Production callers' `InferenceRequest.max_tokens` + `.temperature` therefore never reach the runner — every server-mediated dispatch uses construction-time defaults.

**M5 — Runner exported but no production caller wires it.** `grep -rn "AutoregressiveStreamingRunner" prsm/` returns only the import + `__all__` entry in `prsm/compute/inference/__init__.py`. `make_layer_stage_server` (`prsm/compute/chain_rpc/factories.py:185`) does not accept `streaming_runner=` — so in production, every `LayerStageServer` runs without a streaming runner. The `prsm_inference` MCP tool's chat-style streaming UX is dormant scaffolding until this lands.

Both are exactly the Phase-3.x.8 → Phase-3.x.8.1 pattern: 3.x.8 introduced the streaming wire path + executor generator API + MCP adapter, but no HTTP endpoint exposed it. 3.x.8.1 was the small slice that closed that gap. 3.x.10.x is the analogous closure for the runner — small slice, makes the runner user-visible.

## 2. Scope

**In scope.**
- Extend `RunLayerSliceRequest` wire format with optional `max_tokens: Optional[int]` + `temperature: Optional[float]` fields, conditionally encoded for byte-equivalence with pre-3.x.10.x signed bytes.
- `LayerStageServer.handle_token_stream` builds a sampling-shim and passes `request=` to the streaming runner.
- `ParallaxScheduledExecutor.execute_streaming` populates the new wire fields from the originating `InferenceRequest`.
- New factory `make_autoregressive_streaming_runner(...)` in `prsm/compute/inference/`.
- Extend `make_layer_stage_server(...)` to accept `streaming_runner=Optional[StreamingLayerRunner]`.
- Operator-side wiring entry point so a node-startup script can construct the runner from operator config + the existing model registry path.
- E2E test: real distilgpt2 driven through executor → wire → server → runner with `request.max_tokens=4` exercised end-to-end. Verifies the cap reaches the model and the receipt's signed payload commits correctly.

**Out of scope (deferred).**
- **Tier C constant-time padding** — Phase 3.x.10.y. This is a separate line item, not bundled here, because the design space (M1 fixed-rate vs M2 batched) needs its own scoping.
- **Sharded autoregressive** — Phase 3.x.11. Each chain stage runs once per token with KV-cache handoff. Big-lift wire-format change.
- **Stop sequences** — model-side `stop=[...]` parameter beyond EOS + `max_tokens`. Out of v1's surface.
- **Per-request seeding for deterministic sampled output** — `request.seed` propagation. Useful for replay/CI but not load-bearing for the production caller path; defer to 3.x.10.z if requested.
- **Streaming + chunked-input composition** — the v1 rejection at `RunLayerSliceRequest.activation_manifest is not None && streaming=True` carries forward unchanged; Phase 3.x.7.x revisits.

## 3. Design

### 3.1 Wire-format extension

Following the Phase 3.x.8 Task 1 pattern for the `streaming` flag (and Phase 3.x.7.1's pattern for `activation_manifest`):

- `RunLayerSliceRequest` gains two optional fields:
  - `max_tokens: Optional[int] = None`
  - `temperature: Optional[float] = None`
- Canonical signing payload uses **omit-when-None** conditional encoding so a request with both fields `None` produces byte-identical signed bytes to a pre-3.x.10.x request. No protocol-version bump required — the wire format is already at v2 (Phase 3.x.7.1) and these are additive optional fields. Older peers receiving requests with these fields set ignore them; older callers' requests still parse and verify under new servers.
- Validation:
  - `max_tokens`: positive int (rejected with `MALFORMED_REQUEST` if `<= 0` or non-int).
  - `temperature`: float in `[0.0, 2.0]` (rejected if outside; matches HF `generate()` typical-range guard). `0.0` triggers greedy.
  - Optional fields are independent — caller may set neither, either, or both.

The runner's existing `_effective_max_tokens` / `_effective_temperature` resolvers (autoregressive_runner.py:311-331) handle the `None` fallback to `SamplingDefaults`, so server-side glue is minimal.

### 3.2 Server `request=` plumbing

`LayerStageServer.handle_token_stream` constructs a small **sampling shim** dataclass after parsing the wire request:

```python
@dataclass(frozen=True)
class _StreamingSamplingShim:
    max_tokens: Optional[int]
    temperature: Optional[float]
```

and passes `request=_StreamingSamplingShim(max_tokens=req.max_tokens, temperature=req.temperature)` to `streaming_runner.run_layer_slice_streaming(...)`.

The shim is intentionally minimal — the runner's `getattr(request, "max_tokens", None)` resolver doesn't care about the shape, just that the attribute is present. Using a structural shim instead of forwarding the raw `RunLayerSliceRequest` keeps the runner decoupled from the chain_rpc protocol's full surface (the runner only needs sampling params, not the wire envelope).

The `Protocol` extension in `streaming_runner.py` for `request: Any = None` is unchanged; this is the production-side glue that fills in the previously-None default.

### 3.3 Executor populates wire fields

`ParallaxScheduledExecutor.execute_streaming` builds `RunLayerSliceRequest`s when dispatching to chain stages. The streaming path's request-builder needs to populate the new sampling fields from the originating `InferenceRequest`:

```python
RunLayerSliceRequest(
    ...,
    streaming=True,
    max_tokens=inference_request.max_tokens,
    temperature=inference_request.temperature,
)
```

This is plumbing only — no semantic change. The originating `InferenceRequest` already carries these fields per Phase 3.x.1; they were just being dropped at the wire boundary.

### 3.4 Production runner factory

New factory in `prsm/compute/inference/factories.py` (or a new file if `factories.py` doesn't exist there yet — TBD at execution time):

```python
def make_autoregressive_streaming_runner(
    *,
    model: Any,
    tokenizer: Any,
    identity: NodeIdentity,
    prompt_provider: Callable[
        [Tuple[int, int], np.ndarray, PrivacyLevel], str
    ],
    sampling_defaults: Optional[SamplingDefaults] = None,
    tee_type: TEEType = TEEType.SOFTWARE,
) -> AutoregressiveStreamingRunner:
    ...
```

The `tee_attestation` is derived from `identity.attestation_for_streaming_runner()` (or whatever the existing TEE-runtime → attestation-bytes path is — checked at execution).

### 3.5 `make_layer_stage_server` extension

Extend `make_layer_stage_server(...)` (`prsm/compute/chain_rpc/factories.py:185`) to accept `streaming_runner: Optional[StreamingLayerRunner] = None` and forward it to `LayerStageServer(streaming_runner=...)`. Default `None` preserves back-compat — operators not opting into streaming continue to ship as before.

### 3.6 Node-startup wiring

The smallest correct slice: a single integration point in the operator's node startup that:
1. Loads the model + tokenizer via the existing model-registry path.
2. Calls `make_autoregressive_streaming_runner(...)` with operator's identity.
3. Calls `make_layer_stage_server(..., streaming_runner=runner)`.

The exact integration point is `prsm/node/api.py` or similar — confirmed at execution. Operator config gains a single new field: `enable_streaming: bool = False` (default off so existing deployments don't change behavior; operators opt in).

When `enable_streaming=True` but no model is loaded, the factory raises a clear error rather than silently producing a non-functional server.

## 4. Tasks

### Task 1 — Wire-format extension

**Scope:** add `max_tokens` + `temperature` optional fields to `RunLayerSliceRequest`. Canonical signing payload uses omit-when-None encoding for byte-equivalence with pre-3.x.10.x signed bytes. Add server-side validation (`max_tokens > 0`, `temperature ∈ [0.0, 2.0]`).

**Tests:** ≥6 unit tests:
- byte-equivalence: a request with both fields None produces the same signing payload as a pre-3.x.10.x request (golden bytes assertion).
- non-None fields appear in signing payload (signature differs from None case).
- `max_tokens=0` rejected as malformed.
- `temperature=-1.0` rejected as malformed.
- `temperature=3.0` rejected as malformed.
- `max_tokens=8, temperature=0.7` round-trips through encode/decode unchanged.

**Acceptance:** existing chain_rpc tests stay green; new fields validate; byte-equivalence pinned by a golden-bytes test.

### Task 2 — Server `request=` plumbing

**Scope:** `LayerStageServer.handle_token_stream` constructs `_StreamingSamplingShim` from parsed request fields and passes `request=shim` to `streaming_runner.run_layer_slice_streaming(...)`.

**Tests:** ≥4 tests:
- with `max_tokens=4` on the wire, the runner's `model.generate` receives `max_new_tokens=4` (assert via the existing `_FakeModel.last_call`).
- with `temperature=0.0` on the wire, runner passes `do_sample=False`.
- both fields None → runner uses `SamplingDefaults`.
- runner without `request=` kwarg support still works (back-compat — confirms the shim is duck-typed pass-through, not a hard dependency).

**Acceptance:** server-wired integration tests still pass; new tests pin the plumbing.

### Task 3 — Executor populates wire fields

**Scope:** `ParallaxScheduledExecutor.execute_streaming` populates the new wire fields from `InferenceRequest`. Pure plumbing.

**Tests:** ≥3 tests:
- dispatch with `InferenceRequest(max_tokens=4, temperature=0.5)` produces a `RunLayerSliceRequest` carrying those values on the wire.
- dispatch with `InferenceRequest` defaults (no override) sends `max_tokens=None, temperature=None`.
- the sampling fields are present on the streaming request only (unary path's `RunLayerSliceRequest` continues to omit them — they're streaming-specific in this slice).

**Acceptance:** end-to-end the executor populates → wire carries → server reads → runner uses.

### Task 4 — Production runner factory + `make_layer_stage_server` extension

**Scope:**
- New `make_autoregressive_streaming_runner(...)` factory.
- Extend `make_layer_stage_server(streaming_runner=...)`.
- Wire into the operator node-startup path with an `enable_streaming` toggle.
- Module exports: factory re-exported from `prsm/compute/inference/__init__.py` (lazy via existing `__getattr__` path — circular-import constraints from Task 7 of 3.x.10 still apply).

**Tests:** ≥4 tests:
- factory builds a runner; smoke-test runs a 1-token greedy decode against a fake HF model.
- factory rejects missing model with a clear error (matches Phase 3.x.10 Task 1's RuntimeError pattern).
- `make_layer_stage_server(streaming_runner=None)` (back-compat) constructs a server without streaming.
- `make_layer_stage_server(streaming_runner=runner)` constructs a server that successfully consumes a streaming request.

**Acceptance:** SDK callers can do `from prsm.compute.inference import make_autoregressive_streaming_runner` and wire a streaming-capable node in ~5 lines.

### Task 5 — E2E with real distilgpt2 across full stack

**Scope:** new integration test in `tests/integration/test_autoregressive_runner_full_stack_e2e.py` (or a focused extension of the existing e2e file). Marked `@pytest.mark.slow`.

Drives the runner through: `InferenceRequest(max_tokens=4)` → `ParallaxScheduledExecutor.execute_streaming` → `RunLayerSliceRequest` with `max_tokens=4` on the wire → `LayerStageServer.handle_token_stream` → `AutoregressiveStreamingRunner` → real `distilgpt2.generate(max_new_tokens=4)` → token frames + StreamFinalFrame committing to the joined output.

**Tests:** ≥3 scenarios:
- `request.max_tokens=4` → exactly 4 tokens reach the wire (cap propagated end-to-end).
- `request.temperature=0.0` → greedy decode; bit-identical output across two independent dispatches.
- request with no overrides → runner falls back to `SamplingDefaults` (smoke).

**Acceptance:** real-model + real-server + real-wire + real-runner E2E green; receipt verifies under settler identity with `streamed_output=True`.

## 6. Honest scope (carries forward)

- Tier C constant-time padding still gated until Phase 3.x.10.y.
- Sharded autoregressive still gated until Phase 3.x.11.
- Stop sequences not implemented; only EOS + `max_tokens`.
- HF generate buffering: tokens reach the wire only after `.generate()` returns. Async-during-generate is Phase 3.x.10.y perf upgrade.
- Streaming + chunked-input composition still rejected.
