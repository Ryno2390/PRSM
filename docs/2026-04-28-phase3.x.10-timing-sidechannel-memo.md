## 1. Summary

Phase 3.x.10 replaces `SyntheticStreamingRunner` (which produced N synthetic chunks from a one-shot decoded output) with `AutoregressiveStreamingRunner` (which drives `transformers.AutoModelForCausalLM.generate(streamer=...)` and emits real per-token chunks as the model produces them). This is a strict capability upgrade for end users — chat-style streaming UX is finally real — but it introduces a **per-token inter-token latency side-channel** that the synthetic predecessor did not have.

This memo documents the new leak, scopes its severity per content tier, and tracks the Tier C mitigation as Phase 3.x.10.x.

## 2. The leak

When the runner emits a `StreamingChunk` for each model-produced token, the wall-clock time between consecutive chunks at the wire boundary is a function of:

- **Output length** — total chunk count is the generated token count, observable directly by counting frames.
- **Model's KV-cache state** — first-token latency (TTFT) is dominated by prefill; subsequent inter-token latency is dominated by per-step decode. Both vary with prompt + output length.
- **Vocabulary distribution / token complexity** — for byte-level tokenizers, multi-byte codepoints (emoji, CJK) span multiple BPE tokens, producing visibly different per-codepoint latencies than single-byte ASCII. The `_HFStreamerAdapter` U+FFFD buffer-and-flush *masks the codepoint structure* when tokens span boundaries (the buffer holds until whole-character emerges), which actually slightly *reduces* the leak relative to a naive per-token wire emit — but the underlying per-decode-step timing is still observable.
- **MoE routing** (when applicable) — routing decisions per token leak architectural state.

A passive on-path observer (or an upstream relay node, or any peer with wire-time measurements on the SSE stream) can extract this signal without breaking encryption, without compromising the TEE, and without holding any private key. The signal is co-emitted with the legitimate token frames by construction of the streaming protocol.

This was not present in `SyntheticStreamingRunner` because that path ran the full forward pass first and then split the result into N chunks at synchronous-loop wall-clock, with effectively-zero inter-chunk latency (limited only by Python's loop overhead). The synthetic chunks carried no per-token decode-time signal.

## 3. What the leak does NOT reveal

This is critical for scoping mitigation cost:

- **Prompt content.** The timing signal is downstream of the prompt; it does not reveal prompt tokens, input length, or input semantics directly. (It can, with sustained observation across many requests, leak *aggregate* properties — e.g., this user typically prompts about coding, not poetry — via output-distribution statistics. That is a Phase 3.x.10.x concern only at the population-statistics scale.)
- **Key material.** The runner does not branch on key bits, signature bytes, or any cryptographic state. The leak is from model decode, not from PRSM crypto.
- **TEE-protected state.** TEE attestation continues to bind the runner to its measured code; the timing leak is at the *output emission boundary* (post-TEE), not from inside the enclave.
- **Cross-tenant data.** The runner is single-request scoped per dispatch; no cross-tenant timing channel is created.

## 4. Scoping by content tier

PRSM's content-tier model (Phase 3.x.1 Task 3 + audit-prep §7.6) defines three tiers:

| Tier | Description | Leak severity here |
|---|---|---|
| **Tier A** | Public inference (e.g., open-weights models, public APIs). | **Acceptable.** The output is itself public; per-token timing reveals nothing the output doesn't already reveal once it lands. No mitigation required. |
| **Tier B** | Private inputs / public-class outputs (e.g., user prompts a public model). | **Acceptable in v1.** Output is intended to land at the user; an on-path observer who sees timing already sees the output bytes. Output-content secrecy is a transport-layer concern (TLS), not a runner concern. Long-horizon population-statistics drift is acknowledged as a residual; Phase 3.x.10.x evaluates whether jitter padding is warranted post-launch. |
| **Tier C** | Confidential / encrypted inference (e.g., private fine-tunes, key-material reasoning, regulated workloads). | **Mitigation required.** Constant-time padding is non-negotiable for Tier C semantics — the whole point of Tier C is that an on-path observer learns nothing about the output. Per-token timing breaks that. **Phase 3.x.10.x blocks Tier C from using the autoregressive runner until constant-time padding lands.** |

The v1 runner ships **Tier-A and Tier-B-eligible only** until Phase 3.x.10.x lands.

## 5. Mitigation track — Phase 3.x.10.x

**Scope:** constant-time padding mode for Tier C. Two design candidates, in order of cost-effectiveness:

### 5.1 Candidate M1: Fixed-rate token emission

The runner buffers token chunks internally and emits them at a fixed wall-clock cadence (e.g., one chunk every 50ms, padded with no-op frames if the model hasn't produced a token yet). The cadence is set by operator config + capped at the slowest realistic decode rate so the buffer doesn't grow unboundedly.

**Pros:** simple to implement; preserves streaming UX (user still sees tokens stream); zero leak about per-step decode time.
**Cons:** introduces 1 cadence × N tokens of added latency; the cadence itself is observable but is intended to be operator-uniform.

### 5.2 Candidate M2: Batched emission at fixed boundaries

Runner emits all tokens in a single trailing chunk after `max_tokens` or EOS reached. This is essentially "no streaming for Tier C" — the leak goes to zero because there are no per-token timestamps.

**Pros:** maximally leak-free; trivial to implement (already the SyntheticStreamingRunner shape).
**Cons:** loses streaming UX entirely for Tier C — Tier C operators get the unary path.

### 5.3 Recommendation

Phase 3.x.10.x design plan should default to **M1** with **M2** as a configuration toggle for the most paranoid Tier C operators. Both are straightforward extensions of the runner's existing buffer-and-flush logic.

**Estimated effort:** ~3-5 engineering days (1-2 days for M1, 0.5 day for M2 toggle, 1-2 days for tests + threat-model validation).

## 6. Acceptance / current behavior

For the Phase 3.x.10 v1 runner (this memo's subject):

- The runner SHIPS for Tier A + Tier B.
- The runner is GATED OFF for Tier C until 3.x.10.x. The gate enforcement lives in the dispatch layer (the layer that selects which streaming runner to use per request); the runner itself does not enforce content-tier gating because it has no view of the tier — that's the executor's job per Phase 3.x.1 Task 3.
- The runner's class docstring documents the timing leak; the streaming endpoint's API docs note Tier C is unsupported on streaming until 3.x.10.x.
