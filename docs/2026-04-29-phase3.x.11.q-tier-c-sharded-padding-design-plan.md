# Phase 3.x.11.q — Tier C Constant-Time Sharded Decode

**Status:** Design draft (not yet started).
**Closes:** the named structural deferral carrying across Phase 3.x.11 + 3.x.11.x + 3.x.11.y + 3.x.11.y.x — Tier C content has been structurally denied at the sharded dispatch boundary in all four phases, with `Phase 3.x.11.q` cited as the closure milestone in each threat-model addendum + audit-prep section.

## 1. Problem statement

Phase 3.x.10.y shipped constant-time padding decorators (`BatchedTrailingStreamingRunner` = M2; `FixedRateStreamingRunner` = M1) for the single-host streaming path. Tier C content can flow through Tier C-gated single-host streaming when one of those decorators is wired.

Phase 3.x.11 + 3.x.11.x + 3.x.11.y + 3.x.11.y.x introduced the sharded autoregressive decode path, which has a fundamentally different timing surface: each stage runs once per generated token, so per-token wire latency is observable to the network as `T × N` round-trips (T stages × N tokens). Phase 3.x.11.y added speculation, which compresses N but adds the per-iteration accept-rate channel.

**Tier C has been structurally denied at the sharded dispatch boundary in all four phases.** This is enforced via `_dispatch_sharded` returning `TIER_GATE` when `request.content_tier == ContentTier.C`. The deny is correct but blocks Tier C from using sharded decode at all.

**Phase 3.x.11.q lifts the deny** by extending the M1/M2 decorator pattern to the sharded chain. Two structural differences from 3.x.10.y require care:

1. **Decorator location.** 3.x.10.y wraps `StreamingLayerRunner` (single-host, one server-side runner). Sharded decode goes through `RpcChainExecutor._execute_chain_streaming_sharded`, not a per-stage runner. The decorator must wrap the chain-level execution, not a per-stage runner.

2. **Dispatch is the timing surface.** In 3.x.10.y, the runner produces tokens at variable speed; the decorator times the EMISSION. In sharded, the timing surface is the DISPATCH (each per-token chain pass). Throttling emission alone doesn't help — the per-stage server still observes dispatch arrival times. M1 (fixed-rate) requires both executor-side dispatch throttling AND server-side awareness that it shouldn't process faster than the cadence.

---

## 2. Scope

In scope:
- M2-equivalent for sharded: `BatchedTrailingShardedExecutor` decorator over `RpcChainExecutor` that runs the chain at full speed and emits a SINGLE terminal `StreamFinalFrame` with the joined output. **Trivially constant-time on the executor → caller wire** (single frame regardless of token count). The PER-STAGE wire still leaks per-token timing internally, but the network observer between executor and caller sees only one frame.
- M1-equivalent for sharded: `FixedRateShardedExecutor` decorator with cadence-driven per-token dispatch. Executor times the chain pass to each cadence tick; if chain finishes faster, sleep until tick; if slower, log telemetry (operator-visible latency miss) but proceed. The PER-STAGE wire timing collapses to cadence-aligned ticks; server-side cooperation needed to prevent leaking that the chain finished early.
- Tier C dispatch-layer gate update: lift the `TIER_GATE` deny when `tier_c_sharded_decorator` is wired AND the request matches the operator's expected decorator profile.
- Honest scope on the per-stage internal observability point — the executor-to-caller wire is masked, but the inter-stage wire still leaks. Operators wanting full-network masking need to compose with the existing Tier C dispatch gates on each stage.

Out of scope:
- **Speculation + constant-time = Phase 3.x.11.q.y bundle.** Even with a M1 decorator on the executor, accept-rate and proposed_token_probs leak through.
- **Asynchronous streaming-during-decode (true real-time)**. Same trade as 3.x.10.z — operators wanting real-time emission have to accept the timing surface.
- **Per-stage timing masking.** Operators concerned about an adversarial chain stage learning the dispatch cadence can compose Phase 3.x.10.y's decorators on each stage's `StreamingLayerRunner`. The chain-level decorators in this slice mask only the executor → caller wire.
- **Adaptive cadence based on observed accept-rate / load.** Defer to 3.x.11.q.x.
- **Mid-stream cadence change.** The cadence is a per-request constant.

## 3. Design

### 3.1 BatchedTrailingShardedExecutor (M2)

```python
class BatchedTrailingShardedExecutor:
    def __init__(self, inner: RpcChainExecutor):
        self._inner = inner

    def execute_chain_streaming(
        self, *, request: InferenceRequest, chain: GPUChain,
    ) -> Iterator[Union[StreamToken, ChainExecutionResult]]:
        tokens: List[StreamToken] = []
        result: Optional[ChainExecutionResult] = None
        for event in self._inner.execute_chain_streaming(
            request=request, chain=chain,
        ):
            if isinstance(event, StreamToken):
                tokens.append(event)
            else:
                result = event
        if not tokens:
            if result is not None:
                yield result
            return
        # Emit ONE terminal token with the joined text.
        joined = "".join(t.text_delta for t in tokens)
        last = tokens[-1]
        yield StreamToken(
            sequence_index=0,
            text_delta=joined,
            token_id=last.token_id,
            finish_reason=last.finish_reason,
        )
        if result is not None:
            yield result
```

The executor → caller wire sees exactly one StreamToken plus the terminal ChainExecutionResult — constant-time, no per-token observable. The trade is identical to 3.x.10.y M2: no streaming UX. Tier C operators accept this.

The decorator implements the same `ChainExecutor` Protocol surface as `RpcChainExecutor`, so the existing HTTP SSE endpoint + MCP plumbing pass it through unchanged.

### 3.2 FixedRateShardedExecutor (M1)

```python
class FixedRateShardedExecutor:
    def __init__(
        self,
        inner: RpcChainExecutor,
        cadence_seconds: float = 0.5,
    ):
        self._inner = inner
        self._cadence = cadence_seconds

    def execute_chain_streaming(
        self, *, request, chain,
    ) -> Iterator[Union[StreamToken, ChainExecutionResult]]:
        last_emit = self._clock()
        for event in self._inner.execute_chain_streaming(
            request=request, chain=chain,
        ):
            if isinstance(event, StreamToken):
                # Wait until the next cadence tick before emitting.
                target = last_emit + self._cadence
                now = self._clock()
                if now < target:
                    self._sleep(target - now)
                yield event
                last_emit = self._clock()
            else:
                yield event
```

Per-token emission at fixed cadence. **The `inner.execute_chain_streaming` generator drives the chain at full speed; the decorator's `yield` is what gates emission.** This is the load-bearing simplification: the chain RUNS at native speed, but the network observer sees emission at cadence-aligned intervals.

**Important caveat — per-stage wire still leaks.** A network observer who can see stage-1 dispatches watches them happen at chain-native rate, not cadence rate. The masking is on the executor → caller wire only.

For full-network masking, operators compose this decorator with Phase 3.x.10.y's `FixedRateStreamingRunner` on each stage's `StreamingLayerRunner` — the per-stage runner emits at cadence-aligned ticks too. But sharded decode doesn't use `StreamingLayerRunner` (it uses `ShardedAutoregressiveRunner` via `run_layer_slice_unary`), so a server-side cadence-aligned dispatch wrapper is the analog. **Defer to 3.x.11.q.x** if operators need it; chain-level masking covers the highest-value threat (caller-observable timing).

**Cadence vs chain native rate.** The chain's native per-token latency is `T × RTT + per-stage compute`. Cadence MUST be ≥ this latency, otherwise the decorator just yields immediately every tick (no masking). Operators set cadence based on their measured chain native rate + buffer. Recommended starting point: cadence = 2 × measured native rate. Audit-prep §7.13 documents this calibration step.

### 3.3 Tier C dispatch-layer gate update

The `_dispatch_sharded` Tier C deny in `prsm/compute/chain_rpc/server.py` was structural — it returns `TIER_GATE` regardless of decorator config. **Phase 3.x.11.q does NOT lift this deny on the per-stage server**. Instead, the executor-side decorator wraps the chain-level streaming; per-stage servers continue to receive only Tier A/B dispatches.

This is intentional: per-stage servers shouldn't need to know about Tier C cadence policy. The executor is the trust-policy boundary, not each stage. The stage-level deny stays as a defense-in-depth gate (a misconfigured executor that tries to send Tier C to stages still gets rejected).

**Operator config in production:**

```python
chain_executor = RpcChainExecutor(...)
tier_c_executor = FixedRateShardedExecutor(
    inner=chain_executor, cadence_seconds=0.5,
)

# Routing layer chooses based on content_tier:
def route_executor(request: InferenceRequest) -> ChainExecutor:
    if request.content_tier == ContentTier.C:
        return tier_c_executor
    return chain_executor
```

Where the routing layer lives — the existing `ParallaxScheduledExecutor` is one candidate; the HTTP endpoint / MCP shim is another. **Defer the wiring detail to Task 5** (production wiring).

### 3.4 Honest scope

- **Per-stage wire still leaks.** Documented above and in §3.6 of the threat-model addendum (1.3 revision). Operators wanting full-network masking compose with per-stage cadence wrappers (3.x.11.q.x deferral).
- **Total stream duration leaks total token count under M1.** Same as 3.x.10.y: cadence × frame count = duration. Mitigated by capping `max_tokens`.
- **Total response size leaks under M2.** A single big frame at the end leaks the total output length to the byte counter. Mitigated by padding the joined text to a max length (operator-configurable). **Document this; defer the padding to 3.x.11.q.x.**
- **Speculation under Tier C still denied.** The executor-side `use_stochastic` gate combined with the chain-level decorator doesn't compose cleanly without the 3.x.11.q.y constant-time-speculation work. v1 of this slice keeps speculation = Tier A/B only.

## 4. Tasks

### Task 1 — BatchedTrailingShardedExecutor (M2)
**Scope:** new decorator class implementing `ChainExecutor` Protocol. Pure-Python; no transport changes; no wire-format changes.
**Tests:** ≥6 — Protocol conformance; happy path joins inner tokens into one; finish_reason propagates from last token; empty inner stream emits nothing; result-only inner stream passes ChainExecutionResult unchanged; composes with RpcChainExecutor via mocked transport (proves the decorator doesn't break the chain).

### Task 2 — FixedRateShardedExecutor (M1)
**Scope:** new decorator class with cadence-driven yield. Inject clock + sleep via constructor for testability.
**Tests:** ≥7 — Protocol conformance; per-token emission paced at cadence (mocked clock); cadence honored when chain runs faster than cadence; cadence not artificially slowed when chain runs slower than cadence; ChainExecutionResult passes through immediately (no cadence delay on terminal); cadence_seconds <= 0 raises at construction; composes with RpcChainExecutor via mocked transport.

### Task 3 — Module exports + factory plumbing
**Scope:** export both decorators from `prsm.compute.chain_rpc`; add `make_tier_c_sharded_executor(inner, mode='m2'|'m1', cadence_seconds=...)` factory. Mirrors Phase 3.x.10.y's `make_tier_c_streaming_decorator` pattern.
**Tests:** ≥3 — factory returns correct class per mode; cadence required for m1, rejected for m2; misconfig (unknown mode) raises ValueError.

### Task 4 — Routing-layer integration
**Scope:** identify the right place to wire the Tier C executor. Candidates: `ParallaxScheduledExecutor` (request-routing layer that already knows content_tier), HTTP `/compute/inference/stream` handler (per-request dispatch), MCP `_call_node_api_streaming` (least-good — already deep into transport). **Recommendation: ParallaxScheduledExecutor**, since it already has per-request `content_tier` available and is the natural trust-policy boundary. Implementation: constructor accepts `tier_c_chain_executor: Optional[ChainExecutor] = None`; `execute_streaming` routes to it when `request.content_tier == ContentTier.C`. When unset + Tier C, raises `InferenceError("Tier C streaming requires Phase 3.x.11.q decorator wired in operator config")`.
**Tests:** ≥4 — Tier A/B unchanged (uses default chain_executor); Tier C with decorator routes correctly; Tier C without decorator raises with clear operator-fixable message; both decorators (M1 + M2) integrate.

### Task 5 — E2E with mocked timing observer
**Scope:** integration test asserting executor → caller per-token timing is masked. Uses real distilgpt2 + 2-stage chain (alice 0-3, bob 3-6); the test's "observer" measures inter-frame intervals on the executor's StreamToken stream.
**Tests:** ≥3 — undecorated executor leaks per-token cadence (baseline reference); M2 decorator emits 1 frame total (trivially constant-time); M1 decorator's inter-frame intervals are within ±5% of the configured cadence (statistical assertion across 6+ tokens).

### Task 6 — Threat-model addendum §3.7 + audit-prep §7.13
**Scope:** Phase 3.x.11.q is the load-bearing closure of the threat-model addendum's "Tier C structurally denied" line in §3.4. Add §3.7 documenting (1) which threat surfaces M1 + M2 mask vs. don't, (2) the per-stage internal-observability honest-scope, (3) the cadence-calibration operator advisory. Add audit-prep §7.13 with the standard guarantees + trust seams + auditor reading path. Threat-model addendum bumps to 1.3.
