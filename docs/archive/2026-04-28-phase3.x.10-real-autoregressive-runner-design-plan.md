# Phase 3.x.10 — Real Autoregressive Runner — Design Plan

**Status:** draft
**Date:** 2026-04-28
**Author:** PRSM core
**Predecessors:** Phase 3.x.7 (cross-host ChainExecutor — `phase3.x.7-merge-ready-20260428`), Phase 3.x.7.1 (chunked activations — `phase3.x.7.1-merge-ready-20260428`), Phase 3.x.8 (streaming-token wire path — `phase3.x.8-merge-ready-20260428`), Phase 3.x.8.1 (streaming HTTP endpoint — `phase3.x.8.1-merge-ready-20260428`).

---

## 1. Why this point release exists

Phase 3.x.8 + 3.x.8.1 shipped the streaming-token wire path
end-to-end, but the actual decoder is `SyntheticStreamingRunner` —
it runs the existing one-shot `LayerSliceRunner` to completion,
decodes the resulting activation to text, splits the text into
synthetic deltas (whitespace-based), and yields them as
`StreamingChunk`s. This is **honestly documented as a placeholder
in 5+ places** (Task 2 commit, MEMORY.md, audit-prep §7.4, code
docstrings) — every shipping phase reinforces "we'll do real
decode later."

This point release closes that debt. The replacement is a real
autoregressive runner that generates one token at a time:
tokenizer encodes the prompt, model forward pass produces logits
at the last position, sampling picks the next token, KV cache
state is preserved, repeat until EOS or `max_tokens`. Token-by-
token output is yielded as it's produced — no "compute everything,
chunk after the fact" dishonesty.

After this tag, the streaming work claims "real LLM serving" with
no caveats. The `SyntheticStreamingRunner` placeholder either gets
deleted or kept as a deterministic test fixture (not as a
production code path).

## 2. Scope

### 2.1 In scope

- **`AutoregressiveStreamingRunner`** — new
  `prsm.compute.inference.autoregressive_runner` module
  implementing the existing `StreamingLayerRunner` Protocol from
  Phase 3.x.8 Task 2. Drop-in replacement for
  `SyntheticStreamingRunner`; no public-surface change to
  `LayerStageServer.handle_token_stream` or anything downstream.
- **Tokenizer integration** — runner accepts a tokenizer (HF
  `AutoTokenizer`-shaped object) at construction; uses it to
  encode the prompt context AND to incrementally decode each
  generated token's text piece. Handles BPE multi-byte boundaries
  (a UTF-8 character may span multiple tokens — the runner
  buffers + flushes whole-character chunks).
- **Model adapter** — runner accepts a model handle exposing
  either:
  - HuggingFace `transformers`-style `.generate(input_ids, ...)`
    with a `streamer=` callback, OR
  - a token-by-token forward API (`forward(input_ids,
    past_key_values=...) -> logits, new_past`).
  The default v1 adapter wraps the HF path; the second is plumbed
  for future custom-runtime integrations (e.g. vLLM, llama.cpp).
- **Sampling parameters** — wire `InferenceRequest.max_tokens` and
  `InferenceRequest.temperature` into the runner. Sensible
  defaults: `max_tokens=512`, `temperature=1.0`. Greedy decode
  when `temperature=0`; otherwise temperature-scaled sampling
  with default `top_k=50` and `top_p=0.95`.
- **Stop conditions** — EOS token detection (model-specific via
  `tokenizer.eos_token_id`), `max_tokens` cap, optional stop
  sequences from the request (deferred — see §2.2). `finish_reason`
  on the terminal `StreamingChunk` set accordingly: `"stop"`
  (EOS), `"max_tokens"` (cap), `"error"` (mid-decode crash).
- **Tail-only-decode contract enforcement** — runner asserts
  `is_final_stage=True` on every dispatch; v1 does NOT support
  sharded autoregressive (where every stage runs once per token).
  A non-tail dispatch with this runner raises a structured error
  surfaced as `INTERNAL_ERROR`. Sharded autoregressive is Phase
  3.x.11.
- **E2E integration test** with a real small HuggingFace model
  (GPT-2 or similar — small enough to run in CI without a GPU).
  Verifies token-by-token output produces a real signed receipt
  with `streamed_output=True` AND matches what the model's
  `.generate()` returns deterministically (greedy decode for
  reproducibility).
- **Threat-model revision** — document the timing side-channel
  introduced by autoregressive decode (inter-token latency
  reveals output complexity / vocabulary distribution / output
  length to network observers). Acceptable for non-Tier-C
  content; Tier C constant-time padding is Phase 3.x.10.x.
- **Module exports** — re-export `AutoregressiveStreamingRunner`
  from `prsm.compute.inference` so SDK callers can import without
  knowing the submodule path.

### 2.2 Out of scope (defer)

- **Sharded autoregressive decode.** Real production sharded LLM
  serving runs each stage once per token (not just at prefill).
  This requires KV cache state coordination ACROSS stages with
  per-token dispatch through the chain — Phase 3.x.11. v1 of
  3.x.10 requires the tail stage to host enough of the model to
  generate locally.
- **Stop sequences.** OpenAI-style `stop=["\\n\\n", "User:"]`
  detection is straightforward but adds parsing complexity. v1
  supports only EOS + max_tokens. Track as Phase 3.x.10.x.
- **Speculative decoding / draft-and-verify.** Pure perf
  optimization; out of scope.
- **Constant-time padding for Tier C content.** Documented in the
  threat-model revision; Phase 3.x.10.x.
- **vLLM / llama.cpp / TensorRT-LLM custom runtimes.** v1 ships
  the HF adapter; the second adapter slot is plumbed but its
  concrete implementations are Phase 3.x.10.x.
- **Beam search.** Sampling-only in v1; beam is rare in
  production chat UX.

### 2.3 Non-goals

- **Replacing `SyntheticStreamingRunner` everywhere.** The
  synthetic runner is kept as a deterministic test fixture (it
  satisfies the same Protocol). Production callers wire
  `AutoregressiveStreamingRunner`; the synthetic runner stays for
  scenarios that need known-deterministic output (E2E test
  fixtures, smoke tests, integration harnesses without a model
  available).
- **Model loading/management.** The runner accepts a
  pre-instantiated model + tokenizer. How operators load and
  cache models is orthogonal (Phase 3.x.2 model registry already
  handles weights distribution; the runtime model object lives
  in the operator's runtime).

## 3. Architecture

### 3.1 Mental model

```
              prompt + sampling params
                       │
                       ▼
          AutoregressiveStreamingRunner
                       │
              tokenizer.encode(prompt)
                       │
                       ▼
                  input_ids ──────────┐
                       │              │
                       ▼              │
            ┌──────────────────────┐  │
            │  decode loop:        │  │
            │   forward(ids, kv)   │  │  past_key_values
            │   sample token       │  │  cache lives here
            │   tokenizer.decode   │  │
            │   yield chunk        │  │
            │   if EOS or max:     │  │
            │     break            │  │
            └────────────┬─────────┘  │
                         │            │
                         └────────────┘
                       │
                       ▼
            yield StreamingChunk(
              sequence_index=N,
              text_delta=...,
              finish_reason="stop"|"max_tokens"|None,
              # terminal also carries:
              full_output_text=...,
              duration_seconds=...,
              tee_attestation=...,
              tee_type=...,
              epsilon_spent=...,
            )
```

### 3.2 Runner construction

```python
class AutoregressiveStreamingRunner:
    """Real autoregressive decoder. Implements
    ``StreamingLayerRunner`` Protocol.

    Constructor args:
      model              The model handle. Must expose either
                         ``.generate(input_ids, **kwargs, streamer=...)``
                         (HF path) OR
                         ``.forward(input_ids, past_key_values=...)
                         -> (logits, past_key_values)`` (custom).
      tokenizer          HF AutoTokenizer-shaped object. Must
                         expose ``encode(text) -> List[int]``,
                         ``decode(ids: List[int]) -> str``, and
                         ``eos_token_id: int``.
      tee_attestation    Bytes the stage signs over (TEE-bound
                         identity). Same shape as
                         SyntheticStreamingRunner.
      tee_type           ``TEEType``.
      max_tokens_default Default cap when request doesn't
                         specify (default 512).
      sampling_defaults  Default sampling params:
                         top_k=50, top_p=0.95, temperature=1.0.
                         Request fields override.

    Runtime contract:
      run_layer_slice_streaming() asserts ``is_final_stage=True``;
      a non-tail dispatch yields a single StreamingChunk with
      finish_reason="error" (mapped server-side to INTERNAL_ERROR).
    """

    def __init__(
        self,
        *,
        model: Any,
        tokenizer: Any,
        tee_attestation: bytes,
        tee_type: TEEType,
        max_tokens_default: int = 512,
        sampling_defaults: Optional[SamplingDefaults] = None,
    ) -> None:
        ...
```

### 3.3 Decode loop

```python
def run_layer_slice_streaming(
    self,
    *,
    model: Any,
    layer_range: Tuple[int, int],
    activation: np.ndarray,
    privacy_tier: PrivacyLevel,
    is_final_stage: bool,
) -> Iterator[StreamingChunk]:
    if not is_final_stage:
        yield StreamingChunk(
            sequence_index=0, text_delta="",
            finish_reason="error",
            # ... aggregate fields with attestation but
            # full_output_text="" and duration_seconds=0
        )
        return

    # Decode the prefill activation back to the prompt's input_ids.
    # The prefill stage(s) ran before this; the tail receives the
    # final hidden state but for autoregressive decode we re-derive
    # the input_ids from the request's prompt (the activation is
    # used to seed the KV cache via the model's forward — covered
    # in §3.5 below).
    input_ids = self._tokenizer.encode(self._current_prompt)

    sequence_index = 0
    deltas: List[str] = []
    decoded_so_far = ""
    finish_reason: Optional[str] = None
    start_ts = time.time()

    # HF generate with a streamer callback. The callback fires on
    # every produced token; we yield from a queue to bridge the
    # callback's synchronous emission into the generator's pull
    # model.
    queue: List[Tuple[int, str]] = []

    class _ChunkCollector:
        def __init__(self):
            self.tokens_emitted = 0
        def put(self, token_text: str):
            queue.append((self.tokens_emitted, token_text))
            self.tokens_emitted += 1
        def end(self):
            pass  # nothing to flush — caller exits the loop

    collector = _ChunkCollector()
    streamer = _HFStreamerAdapter(
        tokenizer=self._tokenizer,
        on_token=collector.put,
    )

    # Drive the model. This blocks until generation completes;
    # the `streamer` callback fires as each token is produced.
    try:
        output_ids = self._model.generate(
            input_ids=input_ids,
            max_new_tokens=self._effective_max_tokens(request),
            temperature=self._effective_temperature(request),
            top_k=self._sampling_defaults.top_k,
            top_p=self._sampling_defaults.top_p,
            do_sample=(temperature > 0),
            streamer=streamer,
            eos_token_id=self._tokenizer.eos_token_id,
        )
    except Exception as exc:
        # Mid-decode crash → terminal error chunk.
        yield self._terminal_chunk(
            sequence_index=sequence_index,
            text_delta="",
            finish_reason="error",
            full_output_text="".join(deltas),
            duration_seconds=time.time() - start_ts,
            error=str(exc),
        )
        return

    # Drain the queue + yield as StreamingChunks. (In practice the
    # streamer callback fires synchronously during .generate(), so
    # the queue is fully populated by the time we get here. v2 with
    # async callbacks will yield interleaved with generation.)
    last_idx = len(queue) - 1
    for i, (_, token_text) in enumerate(queue):
        deltas.append(token_text)
        if i < last_idx:
            yield StreamingChunk(
                sequence_index=i,
                text_delta=token_text,
            )

    # Determine finish_reason: did .generate() stop because of EOS
    # (output_ids ended with eos_token_id) or max_tokens (output
    # length hit the cap)?
    if (
        len(output_ids) > 0
        and output_ids[-1] == self._tokenizer.eos_token_id
    ):
        finish_reason = "stop"
    else:
        finish_reason = "max_tokens"

    # Terminal chunk with aggregate fields.
    full_output = "".join(deltas)
    yield self._terminal_chunk(
        sequence_index=last_idx,
        text_delta=queue[-1][1] if queue else "",
        finish_reason=finish_reason,
        full_output_text=full_output,
        duration_seconds=time.time() - start_ts,
    )
```

### 3.4 Activation seed → KV cache priming

The tail stage receives an `activation` ndarray from the previous
stage's prefill output. For autoregressive decode at the tail,
this activation is the model's hidden state at the LAST input
position — which is what the next-token prediction operates on.

Two paths depending on the model adapter:

- **HF `.generate()`** — the model's `prepare_inputs_for_generation`
  + `past_key_values` machinery handles this internally when the
  full prompt's `input_ids` are passed in. The runner re-derives
  `input_ids` from `request.prompt` via the tokenizer, then runs
  generate from scratch. The prefill stage's activation is
  effectively redundant in this path (the model re-runs the
  prefill internally). Documented as a v1 inefficiency.

- **Custom forward adapter** — the runner takes the prefill
  activation directly as the seed `past_key_values` and runs
  token-by-token forward, preserving the prefill compute. This is
  the production-perf path; v1 ships the HF path because it's the
  smallest correct slice.

The choice is per-runner-instance via a constructor flag; the
default is HF for compat.

### 3.5 Tokenizer multi-byte handling

UTF-8 characters can span multiple tokens. A naive
`tokenizer.decode([new_id])` per step may emit invalid UTF-8 mid-
character. The runner uses HF's `TextIteratorStreamer` pattern:
buffer raw bytes from each token, flush only on whole-character
boundaries.

```python
class _HFStreamerAdapter:
    """Wraps the HF TextStreamer pattern. Buffers token bytes
    until a UTF-8 character boundary; flushes whole characters
    via on_token callback."""

    def __init__(self, *, tokenizer, on_token):
        self._tokenizer = tokenizer
        self._on_token = on_token
        self._buffer: List[int] = []
        self._byte_buffer = bytearray()

    def put(self, value):
        # Called by HF generate() with each new token's id.
        self._buffer.append(int(value))
        # Decode the accumulated buffer; if a whole character
        # emerged, flush it and reset the buffer pointer.
        text = self._tokenizer.decode(self._buffer, skip_special_tokens=True)
        if text.endswith("\\ufffd"):
            # Replacement character — incomplete UTF-8. Wait.
            return
        self._on_token(text)
        self._buffer = []

    def end(self):
        # Flush any remaining buffer at end-of-stream.
        if self._buffer:
            text = self._tokenizer.decode(
                self._buffer, skip_special_tokens=True,
            )
            if text:
                self._on_token(text)
            self._buffer = []
```

### 3.6 Joined-text invariant preservation

Phase 3.x.8 Task 2 enforces server-side that
`"".join(text_deltas) == terminal_chunk.full_output_text`.
`AutoregressiveStreamingRunner` MUST satisfy this. The
implementation does:
1. Append each yielded `text_delta` to a local `deltas` list.
2. Compute `full_output_text = "".join(deltas)` exactly.
3. Pass it on the terminal chunk.

The `_HFStreamerAdapter`'s buffer-and-flush guarantees that
`text_delta`s only contain complete UTF-8 characters, so the
joined string never has split surrogates.

### 3.7 Stop conditions + finish_reason

| Condition | finish_reason |
|---|---|
| Last generated token == `tokenizer.eos_token_id` | `"stop"` |
| Generated token count == `max_tokens` cap | `"max_tokens"` |
| Mid-decode model exception | `"error"` |
| Caller cancellation (Phase 3.x.8 Task 6 mechanism) | runner doesn't see this — Python's GeneratorExit unwinds before the runner can emit a terminal frame |

The runner does NOT detect cancellation directly; the consumer
(server's `_dispatch_token_stream`) handles it via try/finally
cleanup per Phase 3.x.8 Task 6.

## 5. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| HF `transformers` dependency adds ~500MB to operator install | High | Medium | Make `transformers` an optional install (`pip install prsm[inference]`); the synthetic runner stays as the no-deps default. Operators advertising streaming-capable nodes opt into the heavy install. |
| HF `.generate()` blocks the event loop during decode | High | Medium | v1 documented as a synchronous bottleneck (the streamer callback fires synchronously during generate). Production async wrapping is Phase 3.x.10.x. The current MCP/HTTP/SSE stack is async-compatible because the runner is consumed inside an existing async context with a thread-pool executor wrapping the sync call. |
| Multi-byte UTF-8 split across token boundaries breaks joined-text invariant | Medium | High | `_HFStreamerAdapter` buffer-and-flush — implementation directly mirrors HF's `TextIteratorStreamer`. Tested with emoji + CJK characters spanning multiple tokens. |
| Greedy decode determinism varies across torch versions / hardware | Medium | Low | E2E test pins torch version + uses CPU-only inference for reproducibility. Document expected variance bound in the threat model. |
| Operator wires `AutoregressiveStreamingRunner` to a NON-tail stage | Low | Medium | Runner's `is_final_stage=True` assertion yields a structured terminal error chunk; server maps to INTERNAL_ERROR. Tested. |
| Timing side-channel exposed by per-token latency | High | Low (non-Tier-C) | Documented in Phase 3.x.10 threat-model memo. Tier C constant-time padding deferred to Phase 3.x.10.x. |
| Tail node hosts only partial model (sharded autoregressive expected) | Medium | High | v1 explicitly rejects sharded autoregressive; tail must host enough of the model to generate locally. Phase 3.x.11 lifts this. Documented in §2.2 + runner docstring. |
| `request.max_tokens=None` causes runaway generation | Medium | Medium | `max_tokens_default=512` at runner construction; runner uses `max(request.max_tokens, max_tokens_default)` if request specifies, else default. Hard cap enforced regardless. |
