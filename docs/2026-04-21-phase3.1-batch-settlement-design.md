## 1. Context & Goals

Phase 3 ships with **local-ledger settlement only** (§3.5 of phase3-marketplace-design). Every FTNS payment — escrow create, release, refund — flows through in-memory `PaymentEscrow` state and the SQLite-backed `LocalLedger`. This is simple and fast, but it means:

1. **No cross-node reconciliation.** Two nodes' ledgers can diverge silently.
2. **No on-chain audit trail** of payments between independent parties.
3. **No programmatic recourse** if a provider disputes a payment after the fact.
4. **No fiat-aligned settlement** — operators can't convert FTNS earnings to on-chain balances without a manual off-protocol step.

The naive fix — post every single dispatch on-chain — is gas-prohibitive. At Base's current ~$0.01–$0.10 per simple ERC-20 transfer, a Phase 3 meganode executing 150M shards/year at per-dispatch settlement would burn $1.5M–$15M/year in gas alone. This kills the economics.

**Phase 3.1 solution: Lightning-style batched settlement.** Providers accumulate signed receipts locally (already produced by Phase 2); periodically, they post a single on-chain transaction representing a Merkle-committed batch of receipts; a challenge window allows requesters or third parties to contest invalid receipts; after the window closes, FTNS transfers execute on-chain for the batch's net amounts.

**Gas amortization example:**
- 1,000 receipts in one batch.
- On-chain cost: ~0.5M gas (Merkle-root commit + batch-level ERC-20 transfers).
- Equivalent single-dispatch cost: 21K × 1000 = 21M gas.
- **Savings: ~40×.** At $0.01/1000 gas units, batch = $0.50 vs $21 single-dispatch.

### 1.1 Non-goals for Phase 3.1

- **Not a rollup.** We do not operate a separate L3 or aggregated execution layer. Batching is receipt-aggregation only; the execution is on Base.
- **Not real-time settlement.** Batches post on a schedule (hourly / daily / 1000-receipt thresholds, configurable). Providers wait for batches.
- **Not cross-chain.** Phase 3.1 ships on Base. Multi-chain settlement is Phase 3.2+.
- **Not slashing.** Fraud proofs inside the challenge window refund wrongly-claimed receipts but do not slash the offending provider. Slashing is Phase 7.
- **Not automatic dispute resolution.** Challenges must be manually submitted within the window; there is no automated dispute arbiter.

### 1.2 Backwards compatibility

Phase 3.1 is **additive**. Phase 3 local-ledger settlement continues to work unchanged for operators who don't enable batch settlement. Operators opt in by configuring their `MarketplaceOrchestrator` with a `batch_settlement_client` dependency; absent that configuration, Phase 3 behavior is preserved.

## 2. Settlement flow

The canonical Phase 3.1 flow, from single-dispatch receipt to on-chain settlement:

### 2.1 Receipt accumulation (provider-side, off-chain)

1. Provider's `ComputeProvider` executes a shard, emits a `ShardExecutionReceipt` (Phase 2 format, already Ed25519-signed, keccak256-canonical).
2. Provider's new `ReceiptAccumulator` stores the receipt in a local queue indexed by settlement counterparty (requester wallet address).
3. Accumulator buffers until one of three batch triggers fires:
   - **Count threshold** — N receipts accumulated (default: 1,000).
   - **Time threshold** — T seconds since last batch (default: 3,600 seconds = 1 hour).
   - **Value threshold** — cumulative FTNS value ≥ V (default: 100 FTNS).

### 2.2 Batch commitment (provider-side, on-chain)

4. Accumulator builds a Merkle tree over the accumulated receipts. Leaf = keccak256 of the canonical receipt bytes. Root = 32-byte hash.
5. Provider calls `BatchSettlementRegistry.commitBatch(merkleRoot, receiptCount, totalValueFTNS, metadata)`. Transaction costs ~100K gas regardless of batch size.
6. The contract emits `BatchCommitted(batchId, provider, merkleRoot, receiptCount, totalValueFTNS, commitTimestamp)`.
7. Contract state: the batch enters `PENDING` state. Claim timer starts.

### 2.3 Challenge window (open to anyone, on-chain)

8. During the challenge window (default: 3 days), any party may submit a fraud proof:
   - `BatchSettlementRegistry.challengeReceipt(batchId, receiptBytes, merkleProof, reason)`.
   - Reason codes: `DOUBLE_SPEND` (receipt already in a prior batch), `INVALID_SIGNATURE` (sig doesn't verify), `NO_ESCROW` (no local-ledger escrow corresponds to this receipt), `EXPIRED` (receipt older than settlement-lookback window).
   - Contract verifies the proof: Merkle inclusion + reason-specific check (e.g., for `INVALID_SIGNATURE`, re-verify the receipt's Ed25519 signature on-chain via a precompile).
9. Successful challenge → contract marks the challenged receipt as `INVALID`; its value is excluded from the final claim.
10. Multiple challenges may target the same batch before the window closes.

### 2.4 Finalization (provider-initiated, on-chain)

11. After the challenge window expires, provider calls `BatchSettlementRegistry.finalizeBatch(batchId)`.
12. Contract computes the final payable value = `totalValueFTNS - sum(invalidated receipt values)`.
13. Contract executes ERC-20 `transfer` from requester escrow wallets to provider's address for the final amount. **This is the on-chain settlement.**
14. Contract emits `BatchFinalized(batchId, finalValueFTNS, invalidatedCount)`.
15. Batch enters `FINALIZED` state. Cannot be challenged further.

### 2.5 Reconciliation (local-ledger cleanup)

16. Provider's local `PaymentEscrow` state for the settled receipts transitions from `RELEASED` (local) to `SETTLED_ON_CHAIN`.
17. Future double-claim attempts (same receipt in a new batch) fail at challenge or batch-commit time via a receipt-index dictionary maintained by the contract.

## 3. Data model

### 3.1 On-chain data structures

```solidity
// prsm/contracts/src/BatchSettlementRegistry.sol (Phase 3.1)

enum BatchStatus { PENDING, CHALLENGED, FINALIZED, VOIDED }

struct Batch {
    address provider;           // who submitted
    bytes32 merkleRoot;         // root of the receipt tree
    uint256 receiptCount;
    uint256 totalValueFTNS;     // sum before challenges
    uint256 invalidatedValueFTNS;  // sum of challenged+proven receipts
    uint256 commitTimestamp;
    BatchStatus status;
    string metadataURI;         // IPFS pointer to off-chain batch metadata (optional)
}

struct InvalidatedReceipt {
    bytes32 batchId;
    bytes32 receiptHash;        // keccak256 of receipt bytes
    address challenger;
    uint8 reasonCode;           // enum; see §2.3
    uint256 valueFTNS;
}

mapping(bytes32 batchId => Batch) public batches;
mapping(bytes32 receiptHash => bool) public receiptClaimed;  // global replay guard
mapping(bytes32 batchId => mapping(bytes32 receiptHash => InvalidatedReceipt)) public invalidated;
```

### 3.2 Contract interface

```solidity
interface IBatchSettlementRegistry {
    // Provider-submitted batch commit
    function commitBatch(
        bytes32 merkleRoot,
        uint256 receiptCount,
        uint256 totalValueFTNS,
        string calldata metadataURI
    ) external returns (bytes32 batchId);

    // Fraud proof against a specific receipt in a pending batch
    function challengeReceipt(
        bytes32 batchId,
        bytes calldata receiptBytes,
        bytes32[] calldata merkleProof,
        uint8 reasonCode,
        bytes calldata auxData  // e.g., for DOUBLE_SPEND, the conflicting batch's ID
    ) external;

    // Provider claim after challenge window closes
    function finalizeBatch(bytes32 batchId) external;

    // Queries
    function getBatch(bytes32 batchId) external view returns (Batch memory);
    function isReceiptClaimed(bytes32 receiptHash) external view returns (bool);
    function challengeWindowSeconds() external view returns (uint256);
}
```

### 3.3 Off-chain data structures (Python)

```python
# prsm/settlement/batch.py (Phase 3.1)

@dataclass(frozen=True)
class BatchedReceipt:
    """A ShardExecutionReceipt wrapped with the data needed for batching."""
    receipt: ShardExecutionReceipt   # from Phase 2
    requester_address: str            # on-chain wallet receiving the invoice
    provider_address: str             # on-chain wallet claiming the payment
    value_ftns: Decimal               # the quoted price from Phase 3 price handshake
    local_escrow_id: str              # pointer back to local PaymentEscrow state


@dataclass
class PendingBatch:
    """Accumulator state for a batch being built."""
    receipts: List[BatchedReceipt]
    started_at_unix: int
    triggered_by: Optional[str]      # None until a trigger fires, then "count"/"time"/"value"


class BatchSettlementClient:
    """Off-chain orchestrator that drives the Phase 3.1 flow."""

    async def accumulate(self, receipt: BatchedReceipt) -> None: ...
    async def maybe_commit_batch(self) -> Optional[str]: ...  # returns batchId if committed
    async def finalize_pending_batches(self) -> List[str]: ...  # called periodically
    async def handle_challenge(self, batchId: str, challenger: str) -> None: ...
```

## 4. Gas economics

### 4.1 Per-batch gas budget

Approximate gas costs on Base (optimistic-rollup, modest gas price ~$0.0001/unit):

| Operation | Gas | USD at $0.0001/gas |
|---|---|---|
| `commitBatch` (fixed) | 100K | $0.010 |
| `challengeReceipt` (per challenge) | 150K | $0.015 |
| `finalizeBatch` (fixed overhead) | 80K | $0.008 |
| ERC-20 transfer inside finalize (per requester → provider pair) | 50K | $0.005 |
| **Total for 1000-receipt batch from 10 unique requesters, no challenges** | **~680K** | **~$0.068** |

Amortized: **$0.068 / 1000 receipts = $0.000068 per dispatch.** Phase 3's per-dispatch cost without batching: $0.01–$0.10 → Phase 3.1 batching is **100-1000× cheaper** per dispatch.

### 4.2 Break-even batch size

At what batch size does batching break even vs direct settlement?

- Direct settlement per receipt: ~21K gas = ~$0.002.
- Batch commit fixed: 100K gas = ~$0.010.
- Break-even: `100K gas / 21K gas ≈ 5 receipts`.

**Batches ≥ 10 receipts are always cheaper than direct settlement.** Default threshold of 1,000 is conservative and leaves massive headroom.

### 4.3 Impact on Prismatica economics (from ECON-WP-1)

Phase 3.1 batching removes the per-dispatch gas overhead entirely. Revisiting §4 of PRSM-ECON-WP-1:

- Without batching: 150M shards/year × $0.002/shard = **$300K/year/chip** in gas.
- With Phase 3.1 batching: 150M shards/year × $0.000068/shard = **~$10K/year/chip** in gas.
- **Savings: ~$290K/year/chip.** For a 500-chip Prismatica meganode: ~$145M/year saved vs non-batched.

Phase 3.1 is not just an operational nicety — it is **material to the unit economics** modeled in PRSM-ECON-WP-1.

## 5. Dispute mechanism

### 5.1 Challenge-window duration

Default: **3 days (259,200 seconds).** Configurable via Foundation governance per PRSM-GOV-1 §4.2 item 2.

Rationale for 3-day default:
- Long enough for a requester to notice an invalid receipt and submit a proof.
- Short enough that provider settlement isn't delayed unreasonably.
- Matches optimistic-rollup dispute-window norms (Optimism, Arbitrum use 7 days; Base derivative choices suggest 3 days is safe for our smaller scope).

### 5.2 Reason codes

| Code | Name | Verification logic |
|---|---|---|
| 0 | `DOUBLE_SPEND` | `auxData` = conflicting batchId; contract checks `invalidated[thatBatch][receiptHash]` OR `receiptClaimed[receiptHash]`. |
| 1 | `INVALID_SIGNATURE` | On-chain Ed25519 precompile verifies receipt signature against receipt's `provider_pubkey_b64`. Base supports Ed25519 via precompile since Cancun fork. |
| 2 | `NO_ESCROW` | Challenger provides local-ledger-equivalent escrow record signed by the requester; contract compares addresses. (Challenger = requester in this case.) |
| 3 | `EXPIRED` | Receipt's `executed_at_unix` > `settlementLookbackWindow` ago. Default lookback: 30 days. |
| 4 | `MALFORMED` | Receipt bytes fail canonical-form decode. |

### 5.3 Challenger incentives

A successful challenge **does not** pay the challenger directly (would create grief incentives). Instead:
- Challenged value is subtracted from provider's final claim.
- Challenger's ONLY reward is the gas cost of the challenge transaction (paid by a small bounty from the Foundation operational reserve).
- This is consistent with PRSM-GOV-1 §4.2 item 6 (Foundation research/operational grants); Foundation board must ratify the bounty budget.

Rationale: direct challenger rewards would incentivize spurious challenges. The structure here preserves honest challenging (requester detects invalid receipt, submits proof at minimal cost) while not creating adversarial behavior.

### 5.4 What happens to an invalidated receipt's FTNS

The invalidated receipt's FTNS value is:
1. **Not paid to the provider.** Excluded from the final `finalizeBatch` transfer amount.
2. **Held in the contract** until a Foundation-governance resolution specifies disposition.
3. **Default disposition** (Foundation pre-ratified): invalidated amounts flow to the Foundation's operational reserve after a 90-day quarantine. Rationale: the value was originally escrowed from the requester; returning it to the requester is complex (the requester may be the challenger, the original escrow holder, or neither); pooling in the Foundation reserve funds the challenge-bounty program itself (§5.3).

### 5.5 Malicious provider patterns defeated

- **Receipt reuse across batches.** `receiptClaimed[receiptHash]` is a global one-shot guard set at finalization. Second batch containing the same receipt fails on-chain verification.
- **Forged receipts.** `INVALID_SIGNATURE` challenge defeats. Ed25519 precompile makes this cheap.
- **Timing abuse.** `EXPIRED` challenge defeats receipts older than 30 days.
- **Value inflation.** Provider claims `value_ftns` higher than what the requester actually authorized. Requester catches this by checking the receipt's recorded value vs the price-handshake record (off-chain). Challenge: `NO_ESCROW` with auxData = signed quote showing correct value.

## 6. Integration with Phase 2/3 primitives

### 6.1 Receipt format — NO CHANGE

Phase 2's `ShardExecutionReceipt` is the unit of settlement. No changes to the Phase 2 format. Phase 3.1 adds **metadata around** receipts (counterparty addresses, local escrow ID) rather than modifying the receipt itself. Backwards compatibility with existing signed receipts is preserved.

### 6.2 `MarketplaceOrchestrator` hook

`MarketplaceOrchestrator` receives an optional `batch_settlement_client` dependency in `__init__`. After a successful dispatch:

```python
# prsm/marketplace/orchestrator.py — Phase 3.1 addition
if self.batch_settlement_client is not None:
    await self.batch_settlement_client.accumulate(BatchedReceipt(
        receipt=receipt,
        requester_address=self.identity.ethereum_address,
        provider_address=listing.provider_ethereum_address,
        value_ftns=quote.quoted_price_ftns,
        local_escrow_id=escrow_job_id,
    ))
```

If `batch_settlement_client is None`, Phase 3 local-ledger behavior is preserved (default).

### 6.3 `PaymentEscrow` — extended state

`PaymentEscrow` adds a new status `SETTLED_ON_CHAIN` (beyond RELEASED / REFUNDED). A local-ledger escrow in `RELEASED` state can transition to `SETTLED_ON_CHAIN` when the corresponding receipt is finalized on-chain. This prevents double-counting: once on-chain-settled, the local-ledger tab no longer contributes to the provider's local balance calculation.

### 6.4 NodeIdentity — ethereum_address

`NodeIdentity` gains an `ethereum_address` attribute. For Phase 2/3 nodes this is None (backwards-compatible). For Phase 3.1-enabled nodes, it's derived from the node's Ed25519 pubkey via a deterministic KDF → secp256k1 keygen, OR manually set at node bootstrap.

The KDF-derivation path keeps a single key-material source (Ed25519 seed) but is cryptographically distinct from the operator's personal Ethereum wallets. An operator may alternately configure their NodeIdentity with a preferred external Ethereum address.

### 6.5 FTNS transfer authorization

The BatchSettlementRegistry needs authorization to execute FTNS transfers from requester escrow wallets. Two options:

**Option A: ERC-20 approve pattern.** Requester's Phase 3 `PaymentEscrow.create_escrow` transitions FTNS from requester wallet to an on-chain escrow wallet (requires per-requester wallet + gas). On finalization, BatchSettlementRegistry pulls from escrow wallet to provider.

**Option B: Escrow-holding contract.** A single `EscrowPool` contract holds FTNS from all requesters, tracked per-requester. BatchSettlementRegistry calls `EscrowPool.settleFromRequester(...)` during finalization.

**Recommendation: Option B.** Single deployment, single approve transaction per requester, cleaner gas profile, reuses a pattern already well-understood from Uniswap V4 hooks and similar. Contract: `prsm/contracts/src/EscrowPool.sol`.

## 10. Open issues

### 10.1 Ed25519 precompile availability on Base

PRSM-CIS-1 attestation and Phase 2 receipts use Ed25519. Base inherits Ethereum's precompiles and the Cancun fork added Ed25519 support via EIP-7212 (secp256r1) but NOT EIP-665 (Ed25519). **Action required:** verify current Base mainnet precompile set; if Ed25519 is unavailable, implement on-chain verification via an audited Solidity Ed25519 library (gas cost: ~2M, significant but acceptable for challenge path only).

**Status:** needs immediate verification; may affect Task 3 feasibility.

### 10.2 Challenge-window duration

3-day default is a guess. Operational experience post-mainnet will reveal whether it's appropriate. Governance-adjustable per PRSM-GOV-1 §4.2.

### 10.3 Bounty budget for challengers

§5.3 proposes Foundation-funded challenge bounties. Budget size is a Foundation operational decision. Initial proposal: $100K/year, reviewed quarterly.

### 10.4 Gas-price volatility on Base

Batched settlement economics depend on Base gas price remaining low. A sustained spike would compress the amortization advantage. Mitigation: batch-size thresholds are configurable per-node; operators can raise thresholds during gas spikes.

### 10.5 Cross-requester batches

The design assumes a provider's batch can include receipts from multiple requesters (one batch, many requester→provider settlements). This works but requires multiple ERC-20 transfers in `finalizeBatch`. Gas analysis in §4.1 already accounts for this. Worth validating empirically on testnet.

### 10.6 Finalization ordering dependence

If two batches from different providers both reference the same receipt (via requester collusion or implementation bug), only one can finalize successfully. The other fails at `finalizeBatch` due to the `receiptClaimed` guard. This is correct behavior but creates a race; providers SHOULD use off-chain coordination to avoid conflicting claims. Documented as a provider-side best practice.

### 10.7 Metadata off-chain storage

The `metadataURI` field on Batch allows providers to attach rich metadata (off-chain batch-detail documents). We deliberately don't enforce a format. If widespread use emerges, a companion `PRSM-BATCH-META-1` standard may specify expected content. Not blocking.
