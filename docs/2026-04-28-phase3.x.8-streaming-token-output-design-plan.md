**Status:** draft
**Date:** 2026-04-28
**Author:** PRSM core
**Predecessors:** Phase 3.x.7 (cross-host ChainExecutor вҖ” `phase3.x.7-merge-ready-20260428` tag at `56c39557`), Phase 3.x.7.1 (chunked activation streaming вҖ” `phase3.x.7.1-merge-ready-20260428` tag at `339957ee`), Phase 3.x.1 Task 8 (MCP streaming context for `prsm_inference`).

## 1. Why this point release exists

Phase 3.x.7's executor returns `ChainExecutionResult` only after the
ENTIRE chain has finished its forward pass. For a 4096-token output on
a 5-stage chain that's roughly 5Г— per-token-decode latency before the
caller sees a single character. A user typing into the canonical
`prsm_inference` MCP tool sees nothing вҶ’ waits ~10вҖ“60 seconds вҶ’ full
response materializes. That UX rules out chat-style interactive use,
which is the dominant LLM access pattern in 2026.

Streaming-token output is the most user-visible scaling lever we can
ship without a multi-week architecture overhaul. The receipt + tier
gate + multi-stage attestation invariants from 3.x.7 + 3.x.7.1 stay
unchanged; this point release adds an additive incremental-output
pipe from the tail stage back through the executor and out the MCP
tool surface.

The trade-off this release makes is honest: PRSM's tail-stage runner
does NOT yet do real autoregressive LLM decode. The current
`TensorParallelInferenceExecutor` runs one forward pass and emits a
single output. Phase 3.x.8 ships the streaming WIRE PATH + EXECUTOR
API + MCP INTEGRATION with a synthetic tail-decode adapter (split
the one-shot output into chunks). When real autoregressive decode
lands as a follow-up runner replacement, the public surface is
already in place вҖ” no contract changes downstream.

## 2. Scope

### 2.1 In scope

- **Wire-format extension** вҖ” two new message types on protocol v2
  (no version bump; additive):
  - `TokenFrame` вҖ” incremental token chunk emitted by tail stage.
  - `StreamFinalFrame` вҖ” terminal frame carrying the signed
    `RunLayerSliceResponse` (full attestation + final aggregate).
- **Streaming flag on Request** вҖ” `RunLayerSliceRequest.streaming:
  bool = False`. When `True`:
  - Non-tail stages reject with `MALFORMED_REQUEST` (only tail
    decodes).
  - Tail stage emits a TokenFrame stream followed by exactly one
    StreamFinalFrame.
- **Server tail-stream handler** вҖ” `LayerStageServer.handle_token_stream(
  request_bytes) -> Iterator[bytes]` mirrors the 8-step validation
  order; yields TokenFrame frames as the tail decoder produces them;
  finalizes with one StreamFinalFrame. Same "never raises" invariant.
- **Streaming runner Protocol** вҖ” `StreamingLayerRunner` extends
  `LayerRunner`. v1 ships a synthetic adapter that wraps the existing
  one-shot runner and splits the joined output into N TokenFrames.
- **Executor streaming API** вҖ” `RpcChainExecutor.execute_chain_streaming(
  request, chain) -> Iterator[Union[StreamToken, ChainExecutionResult]]`.
  Drives non-tail stages first (one-shot prefill via existing
  `_dispatch_stage`), dispatches tail with `streaming=True`, forwards
  tokens upstream, terminates with a `ChainExecutionResult`.
- **Receipt extension** вҖ” `InferenceReceipt` gains a
  `streamed_output: bool` flag. Signing payload commits to the joined
  output text (same hash) regardless of streamed vs. non-streamed
  mode.
- **MCP integration** вҖ” `prsm_inference` tool detects streaming-
  capable client context (per 3.x.1 Task 8); emits progress events
  per token; falls back to buffer-and-return for non-streaming
  callers.
- **Graceful cancellation** вҖ” caller-initiated cancellation
  propagates to the tail stage; tail stops decode, emits
  StreamFinalFrame with `finish_reason="cancelled"`, partial output,
  signed receipt over partial.
- **E2E test** вҖ” single-stage + 2-stage + 3-stage chains stream
  token-by-token; concatenated output bit-identical to single-shot
  reference; receipt verifies under the anchor.

### 2.2 Out of scope (defer)

- **Real autoregressive decode loop in the tail stage runner.** v1
  ships the streaming wire path; the runner-side decode loop is a
  follow-up that plugs into the same `StreamingLayerRunner`
  Protocol without changing the public surface. Tracked as a Phase
  3.x.8.x runner upgrade, not a 3.x.8 task.
- **Per-token chain redispatch** (true sharded autoregressive). v1
  does prefill once and decodes at the tail; cross-stage KV cache
  state coordination is multi-week and deferred.
- **Per-token signatures.** Final receipt commits to the joined
  output bytes. Per-token signing wastes ~100Вөs/token Ed25519 cost
  for no auditor-visible value. Auditors verify the final receipt.
- **Speculative decoding / draft-and-verify.** Out of scope; pure
  perf optimization.
- **Streaming inputs** (long-context prompt streaming). Out of
  scope; prompt arrives in one shot.

### 2.3 Non-goals

- Replacing the unary `execute_chain()` API. Streaming is additive.
  Existing callers continue to work unchanged.
- Sharded model autoregressive coordination. v1 assumes the tail
  stage has enough of the model to decode at all (true for the
  current single-node `TensorParallelInferenceExecutor`; future
  sharded runners plug in via the Protocol).

### 3.2 Phase split (prefill вҶ’ decode)

Streaming operates on the realistic LLM serving split:

- **Prefill phase.** All non-tail stages run their one-shot forward
  pass via the existing unary `_dispatch_stage` path (inline OR
  3.x.7.1-streamed activations both supported). The activation
  threading is unchanged.
- **Decode phase.** ONLY the tail stage runs the streaming token
  loop. The tail receives the upstream activation as its input,
  decodes incrementally, and emits TokenFrames.

This sidesteps cross-stage KV cache coordination entirely. v1 of
3.x.8 is correct for any chain where the tail has full decode
capability for its layer slice. Sharded autoregressive (where every
stage runs once per token) is out of scope.

### 3.3 Wire format

```python
@dataclass(frozen=True)
class TokenFrame:
    """Incremental output frame emitted by the tail stage during
    decode. Multiple TokenFrames per stream; ordered strictly by
    sequence_index. The terminal frame is always StreamFinalFrame."""
    request_id: str
    sequence_index: int   # 0-indexed; strictly increasing
    text_delta: str       # text added by this frame
    token_id: Optional[int] = None  # vocab id if available
    finish_reason: Optional[str] = None  # set on stream tail-end
    protocol_version: int = CHAIN_RPC_PROTOCOL_VERSION


@dataclass(frozen=True)
class StreamFinalFrame:
    """Terminal frame on a token stream. Carries the signed
    RunLayerSliceResponse over the FULL joined output (text bytes
    used as activation_blob in the existing signing payload)."""
    response: RunLayerSliceResponse
    protocol_version: int = CHAIN_RPC_PROTOCOL_VERSIONS
```

`TokenFrame.text_delta` is the unit of incremental output. The
joined `text_delta` across all frames equals the full output that
non-streaming mode would have returned. Token boundaries are at
the runner's discretion (whole tokens for production, word-level
or fixed-byte splits for the v1 synthetic adapter).

`finish_reason вҲҲ {None, "stop", "max_tokens", "cancelled", "error"}`.
Set on the LAST TokenFrame; StreamFinalFrame.response carries the
signed final state.

### 3.4 Streaming flag on RunLayerSliceRequest

```python
@dataclass(frozen=True)
class RunLayerSliceRequest:
    # ... existing fields ...
    streaming: bool = False  # NEW
    # ... existing fields ...
```

Server-side validation (new gate):

- If `streaming=True` AND `is_final_stage=False` вҶ’ reject with
  `MALFORMED_REQUEST` ("only the tail stage may decode").
- If `streaming=True` AND `is_final_stage=True` вҶ’ route to
  `handle_token_stream`.
- If `streaming=False` вҶ’ existing unary `handle()` path unchanged.

v1вҶ”v2 forward-compat: when `streaming=False`, the `streaming` field
is OMITTED from the canonical JSON entirely (mirroring the 3.x.7.1
`activation_manifest` conditional encoding pattern). Existing v1вҶ”v2
inline byte-equivalence preserved for non-streamed messages.

### 3.5 Executor streaming API

```python
class RpcChainExecutor:
    # ... existing execute_chain unchanged ...

    def execute_chain_streaming(
        self,
        *,
        request: InferenceRequest,
        chain: GPUChain,
    ) -> Iterator[Union[StreamToken, ChainExecutionResult]]:
        """Streaming counterpart to execute_chain. Yields
        StreamToken objects as the tail stage produces them; the
        LAST yielded item is a ChainExecutionResult carrying the
        signed multi-stage receipt.

        Caller MUST exhaust the iterator (or call its close())
        otherwise the tail stage's decode loop leaks. Use as:

            for item in executor.execute_chain_streaming(...):
                if isinstance(item, ChainExecutionResult):
                    final = item
                else:
                    handle_token(item)

        On caller cancellation (iterator close before tail-end),
        the executor sends a cancel signal upstream; tail stage
        emits StreamFinalFrame with finish_reason='cancelled'.
        """
```

`StreamToken` is the executor-facing dataclass that wraps
TokenFrame's `text_delta` + `sequence_index` + optional `token_id`
+ optional `finish_reason`. The wire-format type stays internal;
callers see a clean dataclass.

### 3.6 Server streaming handler

```python
class LayerStageServer:
    # ... existing handle, handle_streamed unchanged ...

    def handle_token_stream(
        self,
        request_bytes: bytes,
    ) -> Iterator[bytes]:
        """Streaming-decode path. ONLY invoked for tail stages with
        streaming=True. Mirrors the validation gates of handle();
        yields TokenFrame wire frames incrementally; finalizes with
        exactly one StreamFinalFrame.

        Never raises through the wire boundary. Failures encode a
        StageError as the SOLE yielded frame and the iterator
        terminates.

        Cancellation: if the consumer stops iterating, the runner's
        generator is .close()'d (Python protocol). Runner is
        expected to clean up + emit a final `cancelled` frame
        synchronously on close, but the server doesn't block on it.
        """
```

Validation gates that fire BEFORE any decode work begins:
1. parse + version check (existing)
2. `streaming=True` AND `is_final_stage=True` (new gate)
3. token verify (existing)
4. deadline check (existing)
5. registry lookup (existing)
6. shard coverage (existing)
7. tier gate (existing)
8. activation decode (existing)

Failure at any gate yields ONE encoded StageError frame and the
iterator terminates. No partial token output on rejected requests.

### 3.7 Streaming runner Protocol

```python
class StreamingLayerRunner(Protocol):
    """Tail-stage streaming decoder. Wraps a model that can produce
    incremental output."""

    def run_layer_slice_streaming(
        self,
        *,
        model: ModelHandle,
        layer_range: Tuple[int, int],
        activation: np.ndarray,
        privacy_tier: PrivacyLevel,
        is_final_stage: bool,  # always True on this path
    ) -> Iterator[StreamingChunk]:
        """Yields StreamingChunk objects incrementally; LAST chunk
        has finish_reason set. Final chunk carries the full
        joined output for receipt-signing."""


@dataclass(frozen=True)
class StreamingChunk:
    sequence_index: int
    text_delta: str
    token_id: Optional[int]
    finish_reason: Optional[str]  # set on last chunk
    # On the last chunk only вҖ” full state for receipt signing:
    full_output_text: Optional[str] = None
    duration_seconds: Optional[float] = None
    tee_attestation: Optional[bytes] = None
    tee_type: Optional[TEEType] = None
    epsilon_spent: Optional[float] = None
```

v1 ships a `SyntheticStreamingRunner` that wraps the existing
one-shot `LayerRunner`: runs the full forward pass + decode, then
chunks the resulting text into N TokenFrames (default: split on
spaces, fallback to fixed 16-byte chunks). Real autoregressive
decode replaces this with a runner that emits tokens AS the
underlying model produces them вҖ” same Protocol, no surface change.

### 3.8 Receipt extension

```python
@dataclass(frozen=True)
class InferenceReceipt:
    # ... existing fields ...
    streamed_output: bool = False  # NEW

    def signing_payload(self) -> bytes:
        # For streamed + non-streamed: the payload commits to the
        # full joined output text bytes. The streamed_output flag
        # is part of the signed payload so a relay can't downgrade
        # a streamed receipt to a non-streamed one (or vice versa).
        ...
```

Non-streamed receipts encode `streamed_output: False` (or omit the
field for v1вҶ”v2 byte-equivalence вҖ” same conditional pattern as
3.x.7.1 and the streaming flag). Streamed receipts encode it
explicitly. The `verify_with_anchor` path is unchanged.

### 3.9 MCP integration

The `prsm_inference` MCP tool gains streaming-aware output. Per
Phase 3.x.1 Task 8, MCP context already exposes streaming
capability detection. The flow:

```python
async def prsm_inference(ctx: MCPContext, args: PrsmInferenceArgs):
    request = build_inference_request(args)
    chain = router.allocate_chain(request)

    if ctx.supports_streaming:
        async for item in executor.execute_chain_streaming(
            request=request, chain=chain
        ):
            if isinstance(item, ChainExecutionResult):
                receipt = item
            else:
                await ctx.emit_progress(text=item.text_delta)
        return PrsmInferenceResult(
            text=joined_output,
            receipt=receipt.to_serializable(),
            streamed=True,
        )
    else:
        # Non-streaming client: buffer and return.
        result = executor.execute_chain(request=request, chain=chain)
        return PrsmInferenceResult(
            text=result.output,
            receipt=result.to_serializable(),
            streamed=False,
        )
```

Existing non-streaming MCP clients continue to work unchanged.
Streaming-capable clients see incremental text via standard MCP
progress events.
