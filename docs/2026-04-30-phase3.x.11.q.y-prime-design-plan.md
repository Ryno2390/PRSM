## 1. Problem statement

Phase 3.x.11.q.y closes 2.5 of the 3 Â§3.6 wire surfaces but leaves two carry-forward residuals from the v1 scope:

### 1.1 `RollbackCacheRequest.n_positions_to_drop` leaks `K - accepted_count`

When the speculation loop's rejection sampler determines `accepted_count < K`, the executor emits a `RollbackCacheRequest` to each stage (alice + bob) carrying `n_positions_to_drop = K - accepted_count` so each stage can truncate its KV-cache to the accepted prefix length. A passive wire observer who sees the `RollbackCacheRequest` body learns `K - accepted_count` directly â€” defeating the constant-K commitment and `flat_k_mode` masking from sprint-7. The leak ceiling per VERIFY round is `log2(K+1)` bits of acceptance information; over a 10-VERIFY-round generation at K=4, this leaks ~23 bits of cumulative content-correlated information.

This is the load-bearing v1 honest-scope item from Â§3.8. The E2E test pinning the leak's wire presence (`test_e3_residual_rollback_leak_is_documented`) was structured deliberately so that any closure attempt forces a coordinated update to threat-model Â§3.8 in the same PR.

### 1.2 PSK distribution is operator-managed; no on-wire DH negotiation

Phase 3.x.11.q.y's `ProbsCipher` derives an AES-256 key via HKDF-SHA256 from an operator-distributed pre-shared key (PSK). The PSK MUST reach both the executor and every wired `LayerStageServer` out-of-band â€” environment variable read at server startup, KMS pull, secrets-manager fetch, etc.

Operationally this is workable for tightly-coupled deployments (single-operator multi-region) but creates friction for cross-operator chains where alice and bob are run by different parties. PSK rotation = redeploy. There's no key-compromise recovery path short of full PSK rotation across all wired servers.

The Phase 3.x.11.y.x signing-payload commitment chain already covers per-request authentication via `HandoffToken`. Extending that envelope with an ECDH ephemeral key per request would let executor and tail negotiate a fresh symmetric key per request without operator-managed PSK distribution.

## 2. Scope

In scope:

1. **Always-rollback-K wire-side protocol.** The executor emits a `RollbackCacheRequest` with `n_positions_to_drop = K` on **every** VERIFY round (not just rejection rounds). Combined with a follow-up `replay_accepted_prefix=True` field that re-runs the accepted prefix through the chain to repopulate the cache to the correct state. Wire observer sees a constant-K rollback per round â€” no acceptance-correlated information.

2. **`replay_accepted_prefix` wire field on `RollbackCacheRequest`.** New `Optional[Tuple[int, ...]] = None` field carrying the accepted token prefix bytes. When set, the receiving stage re-runs the prefix through `ShardedAutoregressiveRunner.forward_verify` to rebuild the cache. When `None`, the receiver treats the rollback as a v1-style truncation (preserves backwards-compat for non-Tier-C requests).

3. **`always_rollback_k` mode on `RpcChainExecutor`.** New constructor kwarg defaulting to `False`. When set, every VERIFY round triggers the always-rollback-K protocol regardless of `accepted_count`. Routing-layer extension: `tier_c_speculation_enabled=True` requires `always_rollback_k=True` (else surfaces structured failure).

4. **ECDH per-request key negotiation envelope on `HandoffToken`.** New `ephemeral_pubkey: Optional[bytes] = None` field carrying an X25519 ephemeral public key bytes (32 bytes). When set, the receiving stage performs ECDH against its own X25519 private key (signed by the anchor under the existing publisher-key infrastructure) to derive a per-request shared secret; HKDF-SHA256 over the shared secret + `(request_id, stage_index)` info â†’ AES-256 cipher key for THAT request only.

5. **`X25519AnchoredCipher` factory** alongside the existing `ProbsCipher`. Signature: `X25519AnchoredCipher(local_x25519_privkey, anchor=...)`; encrypt/decrypt API surface-compatible with `ProbsCipher` so the existing `RpcChainExecutor`/`LayerStageServer` integration paths flow through unchanged. The cipher does the per-request ECDH internally based on the `request.upstream_token.ephemeral_pubkey`.

6. **PSK fallback path.** Operators who prefer PSK distribution continue to wire `ProbsCipher` directly (no behavior change from sprint-7). The new `X25519AnchoredCipher` is opt-in via the same `encrypted_probs_cipher=` kwarg â€” the field is duck-typed (Protocol with `encrypt`/`decrypt`), so swapping is a drop-in change.

7. **E2E test transition.** `test_e3_residual_rollback_leak_is_documented` flips from "ASSERT leak presence" to "ASSERT leak absence" (specifically: `n_positions_to_drop == K` on every observed RollbackCacheRequest under `always_rollback_k=True`). The test name evolves to `test_e3_constant_k_rollback_pin`. New test: `test_e5_x25519_per_request_key_rotation` â€” confirms the ephemeral_pubkey is fresh per request and decryption fails when an old request's pubkey is replayed into a new request's envelope.

Out of scope (defer):

- **Forward-secrecy across requests for the SAME key pair.** v1 ECDH per request gives forward secrecy across requests but not across compromise of the long-term anchor key. Post-quantum upgrade (X-Wing or Kyber-768 hybrid) deferred to R6 trigger-watch when post-quantum standards firm up.
- **Replay-attack window.** Even with per-request ECDH, a relay adversary could replay an entire prior request body (all encrypted blobs + ephemeral_pubkey). Mitigation via `request.deadline_unix` + per-stage nonce cache is orthogonal; defer to Phase 3.x.11.q.y'' if telemetry shows replay attempts.
- **Multi-key tail (anchor key rotation mid-stream).** Operators rotate anchor keys at deployment cadence; mid-stream rotation requires re-issuing the chain. Defer.
- **Replay-accepted-prefix optimization.** v1 always-rollback-K replays the prefix per round (perf cost: 1 extra prefix forward per VERIFY round). A streamlined version that batches the replay with the next round's PREFILL exists; defer to perf-tuning sprint.
- **Speculative cache snapshots.** Closing the rollback channel via cache snapshotting (rather than always-rollback) would avoid the replay cost but doubles cache memory. Defer.

## 3. Design

### 3.1 Always-rollback-K wire protocol (closes residual leak 1)

The v1 rollback path:

```
executor: rejection_sampler returns accepted_count=2 (out of K=4)
executor: send RollbackCacheRequest(n_positions_to_drop=2) to each stage
stage:    KVCacheManager.rollback(handle, n=2)
```

Wire observer sees `n_positions_to_drop=2` â†’ infers `accepted_count = K - 2 = 2`.

The Phase 3.x.11.q.y' path:

```
executor: rejection_sampler returns accepted_count=2 (out of K=4)
executor: send RollbackCacheRequest(n_positions_to_drop=K=4,
                                    replay_accepted_prefix=accepted_token_ids[:2])
                  to each stage
stage:    KVCacheManager.rollback(handle, n=4)  # always K
stage:    ShardedAutoregressiveRunner.forward_verify(prefix=replay_accepted_prefix)
          # Rebuilds cache to length len(prefix) by re-running it forward
          # through this stage's layer slice. Required because the rollback
          # truncated past the accepted prefix.
```

Wire observer sees:
- `n_positions_to_drop = K` (constant byte per round; tells nothing about `accepted_count`)
- `replay_accepted_prefix` length on the wire (TELLS `accepted_count`!)

Latter leak is closed by encrypting `replay_accepted_prefix` under the same per-request cipher as `proposed_token_probs`. New `encrypted_replay_accepted_prefix: Optional[bytes]` field on `RollbackCacheRequest`, mutual-exclusion with the plaintext field. AAD = `request_id_utf8 || b"|" || stage_index_byte || b"|" || b"rollback"` â€” distinct from probs AAD to defend against cross-message replay.

### 3.2 ECDH per-request key negotiation (closes residual leak 2)

`HandoffToken` already carries `(request_id, settler_node_id, chain_stage_index, chain_total_stages, deadline_unix, signature)`. The Phase 3.x.7 signing payload covers all fields except `signature` itself. Adding `ephemeral_pubkey: Optional[bytes]` as a covered field extends the same Ed25519 signature envelope:

```python
HandoffToken(
    request_id="req-1",
    settler_node_id="settler-pub-...",
    chain_stage_index=1,
    chain_total_stages=2,
    deadline_unix=1746998400.0,
    ephemeral_pubkey=b"<32-byte X25519 pubkey>",  # NEW
    signature=b"<Ed25519 signature over signing_payload>",
)
```

Receiving stage (alice for stage_index=1):

```python
# Verify signature (existing; unchanged)
anchor.verify(handoff_token.upstream_pubkey, handoff_token.signing_payload(), handoff_token.signature)

# Derive per-request key (NEW)
shared_secret = X25519PrivateKey(local_privkey).exchange(
    X25519PublicKey.from_public_bytes(handoff_token.ephemeral_pubkey)
)
aes_key = HKDF(
    algorithm=SHA256(),
    length=32,
    salt=b"prsm-3.x.11.q.y-prime-x25519-aead",
    info=request_id.encode("utf-8") + b"|" + bytes([stage_index]),
).derive(shared_secret)
cipher = AESGCM(aes_key)
```

Each stage holds a long-term X25519 keypair signed by the anchor (publisher-key extension). The executor generates a fresh ephemeral X25519 keypair per request, ships the public half in the handoff token, derives a separate shared secret per stage. Per-stage uniqueness comes from the `(request_id, stage_index)` info input to HKDF.

### 3.3 Backwards-compat path

Both new wire fields (`replay_accepted_prefix` on `RollbackCacheRequest`, `ephemeral_pubkey` on `HandoffToken`) default to `None`. Pre-3.x.11.q.y' clients/servers continue to work; servers that decode a `None` `ephemeral_pubkey` fall back to the operator-distributed PSK path. Operators upgrade end-to-end at their own cadence.

The `RpcChainExecutor`'s `always_rollback_k` flag stays opt-in; setting `tier_c_speculation_enabled=True` AND `flat_k_mode=True` AND `constant_k_commitment=True` AND `always_rollback_k=False` continues to work but logs a WARNING because the residual leak from Â§3.8 is still present. Setting `always_rollback_k=True` upgrades the deployment to fully-closed Tier C speculation.

## 5. Open questions for the round-1 review

1. **HKDF `info` input shape.** Should the `info` include the `chain_total_stages` to defend against chain-length forgery? Current proposal uses `(request_id, stage_index)`; adding `chain_total_stages` is cheap.
2. **Ephemeral pubkey replay across `deadline_unix` window.** A relay could replay the entire request envelope (including `ephemeral_pubkey`) within the deadline. Per-stage nonce cache is the standard mitigation; do we want it now or defer?
3. **`X25519AnchoredCipher` surface â€” is the Protocol shape from `ProbsCipher` actually compatible?** The latter takes a 32-byte key at construction; the former derives the key per call from `(local_privkey, request.upstream_token.ephemeral_pubkey)`. Is the `encrypt(probs, request_id, stage_index) â†’ bytes` signature still correct, or does the cipher need access to the full `RunLayerSliceRequest` to read `upstream_token.ephemeral_pubkey`?
4. **Replay-prefix encoding.** Tuple of int token IDs is fine for the tail; non-tail stages don't need the prefix tokens (they hold cache by KVCacheHandle). Should `replay_accepted_prefix` be wire-included only when the receiving stage is the tail?
