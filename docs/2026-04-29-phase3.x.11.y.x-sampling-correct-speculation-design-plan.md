## 1. Problem statement

Phase 3.x.11.y ships **greedy-only speculation**: the executor's speculation loop entry rejects `request.temperature > 0` with `PROMPT_ENCODE_ERROR`. Under greedy (`temperature == 0`), the verifier's argmax-per-position is deterministic; the standard "compare verified[i] to proposed[i] for longest prefix match" acceptance algorithm is correct because the verifier's output equals what a non-speculative single-host greedy decode would have produced.

**Under `temperature > 0`, that equivalence breaks.** The verifier's tail samples from a temperature-scaled distribution (with optional top-k / top-p filtering); the draft model samples from its own distribution. Naive prefix-matching against argmaxes would either:
1. Reject every draft (drafts sampled from a stochastic distribution rarely match the verifier's stochastic argmax) — degenerates to single-token decode, no perf win.
2. Accept everything (just trust the draft) — output distribution drifts from the verifier's distribution, breaking the "speculation is a perf optimization, not a sampling change" invariant.

**Phase 3.x.11.y.x closes this gap** by implementing the [Leviathan et al. 2023] *Fast Inference from Transformers via Speculative Decoding* rejection-sampling correction. Under the correction:
- Each draft token is accepted with probability `min(1, p_target(t) / p_draft(t))` where `p_target` is the verifier's distribution and `p_draft` is the draft model's distribution at that position.
- On rejection, sample a correction token from the residual distribution `(p_target - p_draft)+` (clipped to non-negative + renormalized).
- **The output distribution of accepted+correction tokens equals the verifier's distribution exactly.** Speculation remains a perf optimization, not a sampling change.

After this slice, operators can serve sharded inference at `temperature > 0` with full speculation perf benefits and provably-correct sampling.

## 2. Scope

**In scope.**
- **Wire-format extension** — VERIFY response carries the verifier's per-position **probabilities** (or top-K probability tuples; full vocab is too expensive to wire). Probabilities-of-emitted-only is the practical fan-out (we only need p_target(d_i) for the i-th draft we proposed; full distributions for residual sampling are tail-only).
- **DraftModel Protocol extension** — `propose_with_probs(...) → (List[int], List[float])` returns the per-token probability the draft sampled each token at. Greedy path (temperature=0) returns probability 1.0 per token (degenerate); stochastic path returns the actual `softmax(logits)[token_id]` per position.
- **ShardedAutoregressiveRunner tail extension** — `apply_lm_head_and_sample_batch_with_probs(...)` returns K+1 tuples `(sampled_token_id, p_target_at_token_id)`. The K-th position (the bonus) returns its own argmax as before; the rejection check only applies to positions 0..K-1.
- **Executor rejection-sampling loop** — for each draft i in 0..K-1: compute `accept_prob = min(1, p_target(d_i) / p_draft(d_i))`; sample uniform u ~ U(0,1); accept iff u < accept_prob. On rejection at position i: sample correction from residual distribution `(p_target - p_draft)+` (requires more wire data; design choice below) and stop accepting.
- **Adaptive K tuning** (bundled here; cheap once probability-flow is wired) — track per-request rolling accept-rate; if it drops below a threshold, halve K for the next round; if it stays high, double K (capped at MAX_VERIFY_BATCH_TOKENS - 1).
- **Composition** — sampling-correct speculation must compose cleanly with all existing v1 wire surfaces (signing payload commitments, eviction broadcast, IterationAttestation envelope).
- **E2E with real distilgpt2 + small draft** at `temperature > 0` (e.g., 0.7 / 0.9). Verify: (a) accept-rate is realistic (not 0%, not 100%); (b) output distribution converges to single-host generated under the same temperature (sample many runs, compare distribution moments); (c) per-iteration receipt records accept-rate.

**Out of scope (deferred).**
- **Authenticated rollback envelope** — sprint-4 plan called this for the same slice if telemetry warranted. Defer to **Phase 3.x.11.y.x'** (apostrophe) until production telemetry shows DoS exploitation.
- **Multi-draft consensus** — Phase 3.x.11.y' (research direction).
- **Cross-request draft caching** — Phase 3.x.11.y''.
- **Tier C compatibility** — speculation's accept-rate timing surface remains structurally denied for Tier C; combined Tier C + sampling-correct speculation is Phase 3.x.11.q.y.
- **Cache swap-out / paging** — Phase 3.x.11.z.
- **Mid-stream re-routing** — Phase 3.x.12.

## 3. Architecture

### 3.1 Leviathan-2023 rejection sampling — refresher

The standard algorithm (paraphrased from §2.2 of the paper):

```
For draft tokens d_1...d_K with draft probs q(d_i) and target probs p(d_i):
  for i in 0..K-1:
    sample u ~ U(0, 1)
    if u < min(1, p(d_i) / q(d_i)):
      accept d_i
    else:
      reject d_i
      # sample correction from residual distribution
      r(t) = max(0, p(t) - q(t))
      r_normalized(t) = r(t) / sum(r(t))
      sample correction ~ r_normalized
      emit accepted_prefix + correction
      stop
  # all K accepted: emit accepted_prefix + sample one bonus token from p
```

**Key invariant.** The marginal distribution of the emitted tokens equals the target distribution `p`. Proof in Leviathan §2.2 — the construction is an unbiased single-step rejection sampler over the joint distribution.

**v1 greedy specialization.** When `temperature == 0`, both `p` and `q` are point masses on their respective argmaxes. `p(d_i) / q(d_i)` is 1 if argmaxes match, 0 otherwise. The accept condition reduces to argmax-match (= our v1 longest-prefix-match algorithm). v1 is a strict specialization of Leviathan; this slice generalizes correctly.

### 3.2 Probability data on the wire

The design point: **how much probability data does the response carry?**

**Option A — Per-emitted-position probability only.** Tail returns `(sampled_token_id, p_target_at_token_id)` per position. Wire cost: K+1 floats. **Sufficient for the accept check** (rejection sampling needs `p(d_i)` per draft) but **NOT sufficient for residual sampling** on rejection (the executor needs `p` over the entire vocab to sample a correction).

**Option B — Top-K probability tuple per position.** Tail returns `(top_M_token_ids, top_M_probs)` per position for M ≤ vocab. Wire cost: (K+1) × M × (int + float). Compromise: M=50 captures ~99% of probability mass for typical LLM distributions and lets the executor approximate residual sampling locally.

**Option C — Tail computes residual sampling.** Tail receives `(d_i, q(d_i))` from the executor for each draft; on rejection, tail samples from residual locally and returns `(corrections, accept_count)`. Wire cost: K floats outbound + 1 token + 1 int inbound. **Cleanest separation of concerns** (residual sampling is on the model side, where the full distribution lives) but requires the tail to know `q` — needs new wire field.

**Recommendation: Option C.** Lowest wire cost, keeps the residual-sampling math on the model side (where it can use the actual `p` distribution without compression). The draft probabilities `q(d_i)` come from the draft model's `propose_with_probs` and ride out on the request as a new `proposed_token_probs` Tuple field (parallel to the existing `proposed_token_ids`). The tail returns the post-rejection-sampling result directly; the executor's loop just emits what the tail tells it to.

This makes the executor side near-identical to v1 (just emit verified[:accepted_count + 1]) while moving the rejection-sampling math to the runner. **Trade-off**: changes the model-side Protocol — `apply_lm_head_and_sample_batch` becomes `apply_lm_head_and_sample_batch_with_rejection`. Operators wiring custom model adapters for v2 must implement the new method. v1 greedy-only path stays unchanged (the greedy adapter raises on `temperature > 0` per existing gate; this slice's gate adjustment lifts that for `apply_lm_head_and_sample_batch_with_rejection`-capable models).

### 3.4 Tail rejection-sampling Protocol

```python
class ShardedLayerForward(Protocol):
    # existing methods unchanged

    def apply_lm_head_and_sample_batch_with_rejection(
        self,
        *,
        hidden_state_batch: np.ndarray,    # [K+1, hidden]
        proposed_token_ids: List[int],     # K drafts
        proposed_token_probs: List[float], # q(d_i) per draft
        temperature: float,
        top_k: int,
        top_p: float,
    ) -> Tuple[List[int], int]:
        """v2 rejection-sampling tail.

        Returns ``(verified_token_ids, accepted_count)``:
          - For i in 0..accepted_count-1: verified_token_ids[i] ==
            proposed_token_ids[i] (accepted draft).
          - verified_token_ids[accepted_count] is either:
              - the verifier's correction sampled from residual
                distribution (rejection at position accepted_count), OR
              - the bonus token sampled from p (if accepted_count == K).
          - len(verified_token_ids) == accepted_count + 1.

        Algorithm:
          for i in 0..K-1:
            p_i = softmax(ln_f(hidden_state[i]) @ lm_head / temperature)
            (apply top-k / top-p filtering to p_i if specified)
            u ~ U(0, 1)
            if u < min(1, p_i[d_i] / q(d_i)):
              accept d_i; continue
            else:
              residual_t = max(0, p_i - q_i_distribution)
              residual_normalized = residual_t / sum(residual_t)
              correction ~ residual_normalized
              return (accepted_prefix + [correction], i)
          # all K accepted
          p_K = softmax(ln_f(hidden_state[K]) @ lm_head / temperature)
          bonus ~ p_K
          return (accepted_prefix + [bonus], K)
        """
        ...
```

**Note on q_i_distribution.** The residual sampling step needs `q_i` over the full vocab, not just `q(d_i)` (which is just one scalar). Two options:
- **Option C.1**: tail uses the convention that `q_i_distribution` is a point mass on `d_i` with probability `q(d_i)` (i.e., the draft only ever samples `d_i` with probability 1.0 in the greedy specialization). Under this approximation, `residual_t = p_i except residual_t[d_i] = max(0, p_i[d_i] - q(d_i))`. **Mathematically valid only when the draft is greedy** — under stochastic drafting, the draft also has nonzero probability mass on other tokens.
- **Option C.2**: tail receives the draft's full distribution `q_i` per position. Wire cost: K × vocab_size floats per VERIFY round. Prohibitive for vocab > 50K.
- **Option C.3** (**recommended**): tail receives draft's top-M distribution `q_i_top_m` per position. Wire cost: K × M × (int + float). M=50 captures the same ~99% mass argument as Option B.

**Decision.** Ship Option C.1 in v1 of this slice as an **approximation** under the assumption that drafts are typically near-greedy (low draft temperature). Document the approximation in §3.6 honest scope; full Option C.3 is a follow-up if real-world traffic shows the approximation drifts from target distribution at typical draft temperatures.

### 3.5 Executor changes

The executor's speculation loop becomes:

```python
def _execute_chain_streaming_sharded_speculative(self, *, request, chain):
    # PREFILL same as v1
    ...
    while not is_terminal and tokens_emitted < max_tokens:
        # v2: propose with probs (greedy or stochastic)
        proposed_ids, proposed_probs = self._draft_model.propose_with_probs(
            request_id=...,
            parent_token_id=next_token_id,
            k=self._speculation_depth,
            temperature=request_temp,  # threaded from request
        )
        # Chain VERIFY with proposed_token_probs on the wire
        verified, accepted_count, is_terminal, outcomes = (
            self._run_chain_iteration_sharded_verify(
                request=request, chain=chain,
                parent_token_id=next_token_id,
                proposed_token_ids=proposed_ids,
                proposed_token_probs=proposed_probs,  # NEW
                deadline_unix=deadline_unix,
            )
        )
        # accepted_count + 1 tokens already include the
        # rejection-sampled correction or bonus — emit as-is.
        emitted = list(verified)
        # ... rest of the loop as v1: emit, commit, rollback, continue
```

**The executor's role narrows**: it no longer runs prefix-match logic (the tail did the rejection sampling). It just emits what the tail returns and broadcasts rollback for the rejected suffix.

### 3.6 Honest scope (carries forward)

**v2 ships honest about three limitations:**

1. **Draft distribution approximation (§3.4 Option C.1).** Residual sampling treats the draft as a point mass on `d_i`. This is mathematically equivalent to greedy drafting; under stochastic drafting, the residual is computed as if the draft had no mass on non-`d_i` tokens. Empirically the approximation produces output close to but not identical to the target distribution under realistic draft temperatures (≤ 0.5). Phase 3.x.11.y.x' (apostrophe) closes the gap with full Option C.3 if production telemetry shows divergence beyond a measurable bound.

2. **Adaptive K tuning is greedy.** The default policy halves K on accept-rate < 25%, doubles K on accept-rate > 75%. No exploration / regret bound; just a simple ramp. Operators can override with a fixed K. Phase 3.x.11.y.x' could add a multi-armed-bandit policy if production data warrants.

3. **Accept-rate timing surface is amplified.** Sprint-4 threat-model addendum §3.5 documents the per-iteration accept-rate channel under greedy. Under stochastic, the accept-rate distribution becomes wider (more variance per round); a network observer learns the SAME information per round but with finer-grained signal. Tier C remains structurally denied; operator advisory unchanged. Phase 3.x.11.q.y bundle still covers Tier-C-compatible speculation.

## 4. Tasks

### Task 1 — Wire-format extension: proposed_token_probs + bonus_or_correction
**Scope:** add `RunLayerSliceRequest.proposed_token_probs: Optional[Tuple[float, ...]]`. Co-set with `proposed_token_ids` (both set together, both validated against `MAX_VERIFY_BATCH_TOKENS - 1` cap). Each prob in [0, 1]. `RunLayerSliceResponse.verified_token_ids` semantic narrows: now strictly `accepted_count + 1` tokens (not K+1); the last entry is the rejection-sampled correction OR the bonus token. Signing payload extended to commit proposed_token_probs (mirrors existing verify wire pattern).

**Tests:** ≥6 — round-trip + omit-when-default byte-equivalence; co-set invariant; range validation; cap; signing-payload coverage; v1 (no probs) → v2 server backwards-compat.

### Task 2 — DraftModel Protocol + HFDraftModel.propose_with_probs
**Scope:** new `propose_with_probs` returning `(proposed_ids, proposed_probs)`. HF impl uses `model.generate(..., output_scores=True)` and computes `softmax(scores[i])[proposed_ids[i]]` per step. v1 `propose` becomes a thin wrapper that discards the probs. Greedy path returns probs == 1.0 (degenerate).

**Tests:** ≥4 — greedy returns 1.0s; stochastic returns realistic probs in (0, 1); K mismatch handling; deterministic given seed.

### Task 3 — ShardedLayerForward.apply_lm_head_and_sample_batch_with_rejection
**Scope:** new tail Protocol method. Implements the §3.4 algorithm with Option C.1 residual sampling. Uses the existing top-k / top-p filtering machinery from `apply_lm_head_and_sample`. Numerical stability: clamp `min(1, p / q)` away from 1.0 + epsilon (avoid floating-point edge).

**Tests:** ≥6 — perfect-match accept (greedy specialization equivalent to v1); rejection at position 0 (correction sampled from residual); rejection at position K-1 (accept K-1, correct); all-accept K (bonus from p); top-k filtering applied correctly; numerical edge (q = 0).

### Task 4 — ShardedAutoregressiveRunner VERIFY routing
**Scope:** runner detects `temperature > 0 + proposed_token_probs is not None` → routes to `apply_lm_head_and_sample_batch_with_rejection`. Greedy path (`temperature == 0` or no probs) keeps using v1 `apply_lm_head_and_sample_batch` for backwards-compat. Result: tail's `verified_token_ids` length == `accepted_count + 1` (narrowed semantic from v1's K+1).

**Tests:** ≥4 — v1 greedy path unchanged; v2 stochastic path returns `accepted_count + 1` tokens; rejection at boundary; the "v1 client (no probs) hits v2 server" backwards-compat case routes greedy.

### Task 5 — Executor speculation loop v2 + adaptive K
**Scope:** lift the executor's `temperature > 0` gate when `draft_model` exposes `propose_with_probs` AND the chain's tail server advertises v2 capability (lazy detection: first VERIFY response with stochastic temperature confirms tail support; fail loud if not). Adaptive K state per request: rolling `accepted_count / K` over last 4 rounds; halve K on < 25%, double on > 75%. Cap K at `MAX_VERIFY_BATCH_TOKENS - 1`.

**Tests:** ≥6 — temp > 0 path emits accepted+1 tokens correctly; greedy-equivalence under temp == 0 (regression: v1 path still works); adaptive K halves on bad accept-rate; doubles on good; max_tokens cap correct under stochastic; cancellation cleanup correct.

### Task 6 — Server VERIFY routing v1↔v2 backwards-compat
**Scope:** server-side dispatch in `LayerStageServer._dispatch_sharded` detects whether `request.proposed_token_probs is None` (v1 client) or set (v2 client) and routes to the appropriate runner method. v1 server with v2 client returns `MALFORMED_REQUEST` on probs being set (server lacks v2 capability); v2 server is fully backwards-compatible with v1 clients.

**Tests:** ≥4 — v1 client + v1 server (greedy works); v1 client + v2 server (greedy still works); v2 client + v2 server (stochastic works); v2 client + v1 server (returns malformed_request gracefully).

### Task 7 — E2E with real distilgpt2 at temperature > 0
**Scope:** integration test (`@pytest.mark.slow`). Drives 16-token speculative decode with K=4 at temperatures 0.7 and 0.9. Verifies:
  - (a) accept-rate is in a reasonable range (10% < x < 90%) — proves rejection sampling actually fires.
  - (b) over 100 sample runs at fixed prompt + temperature, the output distribution moments (token-id histogram entropy, mean, variance) converge to single-host `model.generate(do_sample=True, temperature=...)` distribution within tolerance. Statistical correctness check.
  - (c) per-iteration receipt records accept-rate AND the receipt's tee_attestation envelope decodes correctly with K=4 VERIFY iterations + corrections.
  - (d) cancellation mid-speculation cleanly rolls back + evicts.

**Tests:** ≥4.

~1-1.5 weeks engineering, 9 tasks. The big-lift items are:
- Task 3 (rejection-sampling tail with numerical-edge correctness — needs careful FP analysis).
- Task 5 (adaptive K state machine + lazy v2-capability detection).
- Task 7 (statistical-correctness E2E — moments-converge test needs decent sample size and fixed-seed determinism).

Risk areas: (a) numerical instability in `min(1, p/q)` when q is small; (b) statistical-correctness E2E flake under low sample sizes; (c) v1↔v2 backwards-compat edge cases (especially v2 client + v1 server, which must fail loud).
