## 1. Problem statement

Every streaming-inference phase since 3.x.8 has shipped with a **tail-only contract**: only the final chain stage runs the autoregressive decode. The tail stage must host enough of the model to generate locally; non-tail stages run a single forward pass and hand off activations. This is the load-bearing limitation that has accumulated as "Phase 3.x.11 deferral" in 5+ docstrings, two design plans, and three audit-prep sections.

The limitation is operationally expensive. A node hosting a 70B-parameter model needs ≥140GB of GPU memory at fp16, which forces operators with smaller GPUs out of the network entirely. The whole point of PRSM's tensor-parallel sharding (Phase 2) was to let multiple nodes pool memory + compute for one inference; the tail-only autoregressive runner discards that capability for the autoregressive case.

**Phase 3.x.11 closes this gap.** Each chain stage runs its layers exactly once per generated token. KV-cache state stays local on each stage between tokens (per-layer attention cache survives across iterations); activations move between stages over the wire each token; the tail stage samples the next token and the executor feeds it back to Stage 1 for the next iteration.

After this phase, an operator with a single 24GB GPU can participate in serving a 70B-parameter inference by hosting one or two layers of the chain — the load-bearing capability for the network's distributed-inference value prop.

## 2. Scope

**In scope.**
- New **decode-mode field** on `RunLayerSliceRequest` (`prefill` vs `incremental`) signaling whether this dispatch is the initial full-prompt forward or a per-token incremental step.
- New **`next_token_id` field** on `RunLayerSliceResponse` (tail-only; populated when the tail samples the next token from its logits).
- **Server-side KV-cache manager** keyed on `request_id`; lifecycle (allocate on prefill, mutate on incremental, evict on terminal/TTL).
- **`ShardedAutoregressiveRunner`** — sibling to `AutoregressiveStreamingRunner` that runs only its layer range with KV-cache management. Tail variant additionally samples and emits `next_token_id`.
- **Executor refactor**: `RpcChainExecutor.execute_chain_streaming` extended with per-token chain loop. Prefill: one chain pass with `decode_mode=prefill`. Decode: per-token chain pass with `decode_mode=incremental` and the previous tail's `next_token_id` as input. Loop until tail returns EOS or `max_tokens` reached.
- **Eviction policy**: cache evicted on terminal (EOS / max_tokens / error), on TTL expiry, and on operator-config cap (max-concurrent-cached-requests).
- **Composition tests** with multi-stage chain producing genuinely-distinct autoregressive output across stages.
- **E2E with real distilgpt2 split across 2 stages** — proves the architecture works end-to-end with a real model.

**Out of scope (deferred).**
- **Pipelined per-token decoding** — overlap forward passes for consecutive tokens to amortize network latency. Phase 3.x.11.x perf upgrade.
- **Speculative decoding** (draft-and-verify across stages). Phase 3.x.11.y.
- **KV-cache compression** for cross-host bandwidth optimization. Tracked via R7.
- **Cache swap-out / paging** — for v1 the cache stays in memory; long-context workloads that exceed per-stage memory caps are rejected. Phase 3.x.11.z.
- **Mid-stream re-routing** — once a request's chain is committed, all incremental dispatches go to the same stages. Phase 3.x.12.
- **Tier C compatibility** — sharded decode is Tier-A/B-eligible only at v1; the per-token wire dispatch creates new timing-side-channel surfaces that the Phase 3.x.10.y constant-time padding decorators don't yet cover. Tier C support tracked via Phase 3.x.11.q (separate scope; depends on a fresh threat-model memo).

## 3. Architecture

### 3.1 Per-token chain flow

```
Per-token wire flow (T-stage chain):

  Executor                Stage 1            Stage 2     ...     Stage T (tail)
  ----------              ---------          ---------           --------------
  prefill request    →    [run prefill]
                          [emit hidden]  →   [run prefill]
                                             [emit hidden]  →    [run prefill]
                                                                 [sample next_token_id]
                                                                 [emit response]
                          ←  ←  ←  next_token_id, full_output_text
  if not done (no EOS, < max_tokens):
    incremental request (input=next_token_id) →
                          [decode 1 step]
                          [emit hidden]  →   [decode 1 step]
                                             [emit hidden]  →    [decode 1 step]
                                                                 [sample next_token_id]
                          ←  ←  ←  next_token_id
  ... loop until EOS or max_tokens ...
  evict request_id's cache on terminal
```

Each stage holds its own KV-cache for the layers it serves. Activations move between stages over the wire each token; the cache stays local.

### 3.2 Wire-format extension

`RunLayerSliceRequest` gains:

```python
@dataclass(frozen=True)
class RunLayerSliceRequest:
    # ... existing fields ...
    # Phase 3.x.11 — sharded autoregressive decode mode.
    # ``prefill`` = first dispatch in a request, full prompt
    # forward, fresh KV-cache allocation. ``incremental`` =
    # per-token dispatch, single-position forward, mutates
    # existing KV-cache. Default ``prefill`` preserves
    # byte-equivalence with pre-3.x.11 messages.
    decode_mode: DecodeMode = DecodeMode.PREFILL
```

`RunLayerSliceResponse` gains (tail stages only):

```python
@dataclass(frozen=True)
class RunLayerSliceResponse:
    # ... existing fields ...
    # Phase 3.x.11 — next sampled token id (populated by the
    # chain tail; non-tail stages leave it None). Carries the
    # token from the tail back to the executor for feedback
    # to Stage 1 in the next chain iteration.
    next_token_id: Optional[int] = None
    # Tail-stage signal: this response represents the final
    # token in the stream (EOS reached or max_tokens hit).
    is_terminal: bool = False
```

Both fields are optional with defaults that preserve byte-equivalence with pre-3.x.11 signed bytes (omit-when-default canonical encoding pattern from 3.x.10.x).

### 3.3 Server-side KV-cache manager

New module `prsm/compute/chain_rpc/kv_cache.py`:

```python
class KVCacheManager:
    """Per-request KV-cache lifecycle. Keyed on request_id;
    survives across incremental dispatches; evicted on
    terminal / TTL / cap.

    Operator config:
      max_cached_requests   Default 64. LRU evict beyond this.
      ttl_seconds           Default 300s (5min). Idle caches
                            evict to bound long-lived
                            zombie-request memory.
    """

    def __init__(
        self, *, max_cached_requests: int = 64,
        ttl_seconds: float = 300.0,
    ) -> None: ...

    def allocate(
        self, request_id: str, n_layers: int,
    ) -> KVCacheHandle:
        """Called on prefill — fresh cache."""

    def get(self, request_id: str) -> Optional[KVCacheHandle]:
        """Called on incremental — existing cache."""

    def evict(self, request_id: str) -> None:
        """Called on terminal / explicit cleanup."""

    def evict_idle(self) -> List[str]:
        """Periodic call — evicts caches past TTL.
        Returns list of evicted request_ids."""
```

The cache handle wraps the underlying tensor structure (per-layer K + V tensors of shape `[batch=1, n_heads, seq_len, head_dim]`) the inner runner produces during its forward pass. The runner manages the actual tensor allocation; the manager manages the *handles* (lifecycle metadata + LRU bookkeeping).

### 3.4 ShardedAutoregressiveRunner

Sibling to `AutoregressiveStreamingRunner` in `prsm/compute/inference/sharded_runner.py`:

```python
class ShardedAutoregressiveRunner:
    """Runs ``layer_range`` with KV-cache management. Two
    dispatch modes:

      - prefill: full input_ids forward; allocates cache;
        emits hidden state at the layer-range boundary
        (or, for the tail, samples next_token_id).
      - incremental: single-token forward with cached KV;
        emits hidden state (or sampled token).

    Tail variant additionally:
      - Applies LM head to final hidden state.
      - Samples next_token_id per request sampling params.
      - Emits is_terminal=True if EOS reached or max_tokens hit.
    """

    def run_layer_slice_unary(
        self, *,
        model, layer_range, activation_or_input_ids,
        kv_cache_manager, request_id, decode_mode,
        is_final_stage, request,
    ) -> LayerSliceResult: ...
```

Note this is the **unary** path — sharded decode does NOT use the streaming wire path. Each per-token dispatch is a unary `RunLayerSliceRequest`; the executor handles the streaming-to-user surface separately by emitting `StreamToken` events as each token arrives from the tail.

### 3.5 Executor per-token chain loop

`RpcChainExecutor.execute_chain_streaming` extended:

```python
def execute_chain_streaming(
    self, *, request, chain,
) -> Iterator[Union[StreamToken, ChainExecutionResult]]:
    if not self._sharded_decode_enabled:
        # Existing 3.x.10.x tail-only path.
        yield from self._tail_only_streaming(request, chain)
        return

    # Phase 3.x.11 sharded path.
    # Step 1: prefill the full prompt across all stages.
    next_token_id, tail_attestation = self._run_chain_iteration(
        request=request, chain=chain,
        decode_mode=DecodeMode.PREFILL,
        input_ids=self._tokenizer.encode(request.prompt),
    )
    yield StreamToken(text_delta=self._decode_text(next_token_id))

    # Step 2: per-token decode loop.
    for _ in range(request.max_tokens or DEFAULT_MAX_TOKENS):
        next_token_id, attest, is_terminal = self._run_chain_iteration(
            request=request, chain=chain,
            decode_mode=DecodeMode.INCREMENTAL,
            input_ids=[next_token_id],
        )
        yield StreamToken(text_delta=self._decode_text(next_token_id))
        if is_terminal:
            break

    # Step 3: emit terminal frame + signed receipt.
    yield ChainExecutionResult(...)

    # Step 4: cleanup — send eviction signals to all stages.
    self._evict_cache_on_stages(chain, request.request_id)
```

The cache lives on the stages, but the executor is responsible for issuing eviction signals at the end of the request (and on cancellation). The eviction signal is a new lightweight wire message: `EvictCacheRequest(request_id)`.

### 3.6 Eviction policy

Three eviction triggers:

1. **Explicit eviction from executor**: end of request (terminal token / `max_tokens`), or executor receives `GeneratorExit`. Sends `EvictCacheRequest(request_id)` to all stages.
2. **TTL expiry on the stage**: each stage runs a periodic sweeper that evicts caches whose last-touch time exceeds `ttl_seconds`. Defends against zombie requests where the executor crashed without eviction.
3. **LRU cap**: each stage has a `max_cached_requests` config. When a new prefill arrives and the cache pool is full, the LRU cache is evicted to make room.

All three paths are tested.

### 3.7 Honest scope — performance characteristics

Sharded decode trades memory pressure (no longer concentrated on the tail) for **wire latency per token**. For a 3-stage chain over WAN with 50ms RTT:
- Per-token wire latency: 3 hops × 50ms = 150ms minimum.
- Token throughput: ~6-7 tokens/second (before model decode time).

This is significantly slower than tail-only autoregressive on a single host. The trade is "any operator can participate" vs "decode is fast." Pipelining (Phase 3.x.11.x) overlaps forward passes across consecutive tokens to bring effective per-token latency down to roughly the slowest stage's decode time.

For LAN deployments (datacenter chains), per-hop RTT is ~0.5-2ms, making the wire-latency tax negligible. v1 ships honest about the WAN limitation; operators serving over WAN should disable sharded decode (`enable_sharded_decode=False`).

## 4. Tasks

### Task 1 — Wire-format extension
**Scope:** add `decode_mode: DecodeMode` (Enum: PREFILL/INCREMENTAL) to `RunLayerSliceRequest`; add `next_token_id: Optional[int]` + `is_terminal: bool` to `RunLayerSliceResponse`. Omit-when-default canonical encoding for byte-equivalence. Validation: `decode_mode` must be a real DecodeMode (bool-rejection); `next_token_id` must be non-negative int when set.
**Tests:** ≥6 — golden-bytes byte-equivalence (mirrors 3.x.10.x M-TEST-1 pattern); both fields round-trip; bool-rejection on decode_mode; type-rejection on next_token_id; pre-3.x.11 wire bytes (default values) parse cleanly under new code.

### Task 2 — Server-side KV-cache manager
**Scope:** new `prsm/compute/chain_rpc/kv_cache.py` with `KVCacheManager` + `KVCacheHandle`. Allocate / get / evict / evict_idle. LRU eviction on cap; TTL sweeper.
**Tests:** ≥8 — allocate creates new handle; get returns existing; double-allocate-same-id raises; evict releases; LRU eviction order; TTL eviction past expiry; concurrent-allocation thread safety; evict_idle returns evicted ids list.

### Task 3 — `ShardedAutoregressiveRunner` (non-tail variant)
**Scope:** new `prsm/compute/inference/sharded_runner.py`. Constructor accepts model + layer_range + kv_cache_manager. `run_layer_slice_unary` dispatches prefill (full input_ids forward, fresh cache) or incremental (single-token forward, cached KV). Returns hidden state at layer-range boundary. Tail variant deferred to Task 4.
**Tests:** ≥6 — prefill allocates cache; incremental uses existing cache; incremental without prior prefill rejects with malformed-state error; layer_range respected; cache survives across multiple incremental calls; eviction on explicit signal.

### Task 4 — `ShardedAutoregressiveRunner` tail variant
**Scope:** extend the runner to detect tail (`is_final_stage=True`), apply LM head to final hidden state, sample `next_token_id` per request sampling params, populate response with `next_token_id` + `is_terminal`. Honors EOS detection + max_tokens cap from the request.
**Tests:** ≥5 — tail samples next_token_id; greedy temperature=0 deterministic; max_tokens=1 returns is_terminal=True; EOS hit returns is_terminal=True; non-tail dispatch leaves next_token_id=None.

### Task 5 — Executor per-token chain loop
**Scope:** `RpcChainExecutor.execute_chain_streaming` extended with sharded path (`enable_sharded_decode=True`). Prefill chain pass; per-token decode loop; yields `StreamToken` per token; emits final `ChainExecutionResult` with signed receipt. Eviction signals on terminal + on cancellation.
**Tests:** ≥6 — single-stage chain prefill + decode (smoke); 2-stage chain produces autoregressive output; max_tokens cap honored end-to-end; EOS reached → terminates loop; cancellation triggers eviction broadcast; sampling params from request reach the tail.

### Task 6 — Eviction signal wire message + handler
**Scope:** new `EvictCacheRequest(request_id)` wire message. Server-side handler invokes `kv_cache_manager.evict(request_id)`. Executor sends to all stages on terminal / cancellation.
**Tests:** ≥4 — message round-trip; handler evicts cache; idempotent (evict-then-evict-again); broadcast to all chain stages on executor-side terminal.

### Task 7 — E2E with real distilgpt2 split across 2 stages
**Scope:** integration test (`@pytest.mark.slow`) that loads distilgpt2, splits its 6 transformer layers into two stages (Stage 1: layers 0-2; Stage 2: layers 3-5 + LM head). Drives a 4-token sharded autoregressive decode through the executor's per-token chain loop. Verifies output is genuinely autoregressive (token_ids change per iteration) and matches single-host distilgpt2 output for the same prompt + greedy decode (bit-identical determinism check).
**Tests:** ≥3 — sharded output equals single-host output with greedy temperature=0; per-token next_token_id is real (not synthetic); cache survives across the 4-iteration loop and is evicted on terminal.

### Task 8 — Threat-model addendum + audit-prep §7.9
**Scope:** new `docs/2026-04-30-phase3.x.11-threat-model-addendum.md` covering:

- Cross-stage activation handoff per token magnifies the activation-inversion surface from R3 (more chances per request for a malicious shard to capture activations).
- Tier C is structurally INCOMPATIBLE with v1 sharded decode; gated off until a fresh memo addresses the new timing surface.

## 7. Estimated effort

~2-3 weeks engineering, ~9 tasks. The big-lift items are Task 3 (cache management with real tensor handles), Task 5 (executor refactor), and Task 7 (real-model E2E). Tasks 1+2+4+6 are roughly half-day each. Task 8 is documentation; Task 9 is the review-and-tag cycle.

The slice is significantly larger than 3.x.10.x or 3.x.10.y because it changes the fundamental dispatch model (per-token chain redispatch) and adds server-side state (KV-cache manager). Risk areas: cache lifecycle correctness (memory leaks if eviction misses a path), cross-stage activation handoff semantics (must match the model's expected layer-boundary tensor shapes), and per-token wire latency (the WAN limitation is honest scope, not a bug, but operators need to know).
