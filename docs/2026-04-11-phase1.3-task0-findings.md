## Gossip / bootstrap wiring paths checked

### Gossip subscriptions in `ContentProvider`
- `prsm/node/content_provider.py:405` — subscribes to `GOSSIP_CONTENT_ADVERTISE`
- Handler: `_on_content_advertise` at line 811

Read of `_on_content_advertise` (lines 811–825): it calls
`self.content_discovery.announce_content(...)` (a **remote-peer** discovery
map) and nothing else. **It does NOT write to `self._local_content`.**
This handler is for tracking what *other* nodes can serve, not for registering
our own content.

### CrossWire check: is there an alias or shared dict?
Grep of `content_uploader.py` for `_local_content|register_local` returned

copied or aliased into the provider.

## Is there a production path that populates `_local_content`?

**NO.**

Concrete evidence:

1. `register_local_content` has zero production callers anywhere in `prsm/`
   (the only writer method to `_local_content`).
2. `_on_content_advertise` (the provider's only gossip subscriber,
   `content_provider.py:811`) writes only to `content_discovery`, never to
   `_local_content`.
3. All three HTTP upload endpoints (`prsm/node/api.py:612`,
   `prsm/node/api.py:640`, `prsm/interface/api/content_api.py:34`) complete
   without calling any method on `content_provider`.
4. `ContentUploader.upload()` (`content_uploader.py:254`) persists to its
   own `self.uploaded_content` dict and publishes gossip, but makes no call
   to `content_provider.register_local_content`.
5. `ContentUploader._upload_with_sharding()` (`content_uploader.py:449`)
   and the sharding-fallback path (line 598) both do the same — no
   `register_local_content` call.
6. `_hydrate_from_db()` at `node.py:744` rehydrates the uploader's
   `uploaded_content` dict only; it never calls into
   `content_provider.register_local_content`.
7. `ContentIndex._on_content_advertise` (`content_index.py:89`) writes to
   its own `_records` dict; `ContentIndex._on_provenance_register`
   (line 133) writes to the local ledger. Neither touches
   `ContentProvider._local_content`.

In production, `ContentProvider._local_content` is initialized as `{}` at
`content_provider.py:377` and **remains empty for the lifetime of the node**.
Consequently:

- `_handle_content_request` at line 510 will return `not_found` for **every**
  inbound CID request (the `content_info = self._local_content.get(cid)`
  lookup will always be `None`).
- The local fast-path in `request_content` at line 651
  (`if cid in self._local_content`) is dead code in production.
- `content_economy.process_content_access` never fires from the serve path
  (the serve path is gated by `_local_content` and the not-found branch does
  not invoke payment processing — see `content_provider.py:511–516`).
- `get_stats()` always reports `local_content_count=0`.

The Phase 1 on-chain royalty flow, end-to-end, is **inoperable in
production** despite passing unit and integration tests, because the tests
manually call `register_local_content` to stage the fixture.

## If NO: proposed fix location

The fix belongs in `ContentUploader.upload()` and its two sibling code
paths (`_upload_with_sharding` and the sharding fallback). The uploader
already holds all the data needed (`cid`, `size_bytes`, `content_hash`,
`filename`, `metadata`, `royalty_rate`, `parents`, `provenance_hash_hex`,
`creator_id`). The wiring just needs to also invoke the provider.

### Option A: inject `content_provider` into `ContentUploader`

1. Extend `ContentUploader.__init__` (`content_uploader.py:181`) with an
   optional `content_provider: Optional[Any] = None` kwarg, stored as
   `self.content_provider`.
2. In `node.py:342–366`, after both are constructed, add:
   ```python
   self.content_uploader.content_provider = self.content_provider
   ```
   (matches the existing late-wiring pattern at `node.py:719, 837`.)
3. In `content_uploader.py:396` (monolithic upload), right after
   `self.uploaded_content[cid] = uploaded` and before the gossip publish on
   line 404, add:
   ```python
   if self.content_provider is not None:
       self.content_provider.register_local_content(
           cid=cid,
           size_bytes=size_bytes,
           content_hash=content_hash,
           filename=filename,
           metadata={
               **(metadata or {}),
               "creator_id": self.identity.node_id,
               "royalty_rate": rate,
               "parent_cids": parents,
               "provenance_hash": provenance_hash_hex,
           },
       )
   ```
4. Mirror the same call in `_upload_with_sharding()` at
   `content_uploader.py:541` (right after
   `self.uploaded_content[manifest_cid] = uploaded`, before the gossip
   publish at line 549).
5. Mirror the same call in the sharding fallback at
   `content_uploader.py:638` (right after
   `self.uploaded_content[cid] = uploaded`, before line 642).
6. Also mirror in `_hydrate_from_db()` around `content_uploader.py:1356`
   so provenance records restored from DB repopulate the provider's
   `_local_content` at node start, not just the uploader's `uploaded_content`.

**Critical:** the metadata dict must carry `provenance_hash`,
`royalty_rate`, `creator_id`, and `parent_cids` because
`_fire_payment_for_test` (line 887) and, by extension, the real serve path
need them to call `content_economy.process_content_access` correctly.
`register_local_content`'s current signature stores arbitrary metadata
under `self._local_content[cid]["metadata"]`, so this works without a
schema change. (Phase 1.2's `_fire_payment_for_test` already reads
`provenance_hash` from either the top-level dict or nested metadata — line
891–894 — so either placement works.)

### Option B: register from the HTTP handlers instead

Not recommended. `prsm/interface/api/content_api.py:34` has no handle on a
running node's `content_provider` (it's a plain FastAPI router using
`ContentStore` directly) and `prsm/node/api.py:612` could reach
`node.content_provider`, but splitting the registration across uploader
and HTTP handler duplicates the logic and misses the CLI/SDK path. The
single source-of-truth fix is in the uploader.

### Additional observation (not strictly Task 0, but worth flagging)

The serve path at `content_provider.py:493–516` gates the response on
`_local_content` but does **not** invoke `content_economy.process_content_access`
on the happy path. Read lines 518–599 of the file carefully during Task 1:
after confirming `content_info` and retrieving bytes, the response is sent,
but there's no `await self.content_economy.process_content_access(...)` call
in the serve flow that I can see in the 330–915 range I read. The only
`process_content_access` call in production code paths lives in the test
seam `_fire_payment_for_test` (line 895). Task 1 may need to fix both
(a) populating `_local_content` and (b) wiring the actual serve path to
fire the payment — otherwise the royalty still won't flow end-to-end even
after the wiring gap is closed. **Verify this before closing Task 1.**

## If YES: what path was it?

N/A — no production path populates `_local_content`.

## Decision recommendation

**REAL_BUG_PROCEED_WITH_TASK_1**

Evidence is unambiguous:
1. Zero production callers of `register_local_content`.
2. No gossip handler writes to `_local_content`.
3. No bootstrap code wires uploader → provider for local content registration.
4. All three upload entrypoints (`node/api.py`, `interface/api/content_api.py`,
   `cli.py` → HTTP) complete without touching `content_provider`.
5. `_hydrate_from_db()` only restores the uploader's own dict, not the
   provider's.

Phase 1.2's scan did not miss a path — the path genuinely does not exist.
Task 1 should proceed to add the wiring per Option A above. Strongly
recommend also verifying the serve-path payment trigger during Task 1 (see
"Additional observation").
