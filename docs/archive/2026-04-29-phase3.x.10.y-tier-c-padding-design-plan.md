## 1. Problem statement

Phase 3.x.10.x's full-stack E2E surfaced two open issues that need to land together:

**Tier C streaming is structurally blocked.** Per the timing-sidechannel memo §4, the runner ships Tier-A + Tier-B-eligible only because the per-token inter-token-latency leak is unacceptable for Tier C semantics (encrypted/private inference where the threat model includes a passive on-path observer). Operators serving Tier C content currently fall back to the unary path. This memo §5 enumerated two mitigation candidates (M1 fixed-rate cadence preferred, M2 batched-trailing as toggle); 3.x.10.y implements both.

**HF prompt-echo on first wire chunk.** With GPT-2-family byte-level BPE tokenizers, HuggingFace's TextStreamer convention passes the FULL token sequence (`input_ids + first generated token`) to `streamer.put()` on the first call. Our `_HFStreamerAdapter` cumulatively decodes whatever tokens it accumulates, so the first wire chunk includes the prompt text. The Phase 3.x.10.x full-stack E2E worked around this by asserting cap propagation via `finish_reason="max_tokens"` rather than chunk count, but the actual behavior is observable to end users — they see their own prompt echoed before the model's response. Phase 3.x.10's L1 review-finding noted a stale `_prompt_id_count` comment referencing exactly this fix.

Both are runner-internals concerns; bundling them keeps the slice coherent.

## 2. Scope

**In scope.**
- HF prompt-echo fix: `_HFStreamerAdapter` skips the first `prompt_id_count` tokens passed via `put()`. Runner constructs the adapter with `prompt_id_count=len(input_ids)` before calling `model.generate(streamer=...)`.
- **M2 batched-trailing decorator** — `BatchedTrailingStreamingRunner(inner)` wraps any `StreamingLayerRunner` and consumes the full inner stream before emitting a SINGLE terminal `StreamingChunk` carrying the joined output. Zero per-token timing observable to a wire observer; sacrifices streaming UX for Tier C.
- **M1 fixed-rate decorator** — `FixedRateStreamingRunner(inner, cadence_seconds=0.05)` wraps any `StreamingLayerRunner` and emits chunks at a fixed wall-clock cadence. Buffers tokens internally; pads with no-op frames when the model hasn't produced a token by the next tick. Preserves streaming UX while masking per-token latency.
- **Tier C dispatch-layer gate** — when the operator opts in to streaming for Tier C content, the dispatch layer (Phase 3.x.1 Task 3 content-tier gate) applies one of the two decorators per operator config. Default-deny: a Tier C streaming dispatch with no decorator configured falls back to the unary path with a clear warning.
- E2E tests with mocked timing observer asserting the no-leak invariant for both decorators.

**Out of scope (deferred).**
- **Async-during-generate runner** — the HF generate buffering means tokens reach the wire only after `.generate()` returns. Async streaming during generate is a Phase 3.x.10.z perf upgrade.
- **Sharded autoregressive** — Phase 3.x.11.
- **Stop sequences** — separate scope.
- **Per-request seeding** — Phase 3.x.10.z if requested.
- **Cadence auto-tuning** — operators set the M1 cadence manually in v1; auto-tuning based on model decode rate is a 3.x.10.z follow-up.

## 3. Design

### 3.1 Prompt-echo fix

`_HFStreamerAdapter` gains a constructor kwarg `prompt_id_count: int = 0`. Inside `put()`, the adapter tracks total tokens received vs. `prompt_id_count`; the first `prompt_id_count` tokens are recorded (so cumulative-decode positions stay correct) but `_print_offset` is initialized to the prompt's decoded text length AFTER the prompt is fully accumulated. This ensures the runner's `_HFStreamerAdapter._maybe_flush()` only emits text from positions BEYOND the prompt.

Implementation note: HF's TextStreamer has a `skip_prompt: bool = False` kwarg that does exactly this — we mirror its semantics in our adapter. The adapter's contract becomes: "emit text deltas for token ids strictly after the first `prompt_id_count` ids."

`AutoregressiveStreamingRunner.run_layer_slice_streaming` constructs the adapter with `prompt_id_count=len(input_ids)` (where `input_ids` is the encoded prompt) before passing it to `model.generate(streamer=adapter)`.

### 3.2 BatchedTrailingStreamingRunner (M2)

```python
class BatchedTrailingStreamingRunner:
    def __init__(self, inner: StreamingLayerRunner):
        self._inner = inner

    def run_layer_slice_streaming(
        self, *, model, layer_range, activation, privacy_tier,
        is_final_stage, request=None,
    ) -> Iterator[StreamingChunk]:
        chunks = list(self._inner.run_layer_slice_streaming(
            model=model, layer_range=layer_range,
            activation=activation, privacy_tier=privacy_tier,
            is_final_stage=is_final_stage, request=request,
        ))
        if not chunks:
            return
        terminal = chunks[-1]
        joined = "".join(c.text_delta for c in chunks)
        yield StreamingChunk(
            sequence_index=0,
            text_delta=joined,
            finish_reason=terminal.finish_reason,
            full_output_text=terminal.full_output_text,
            duration_seconds=terminal.duration_seconds,
            tee_attestation=terminal.tee_attestation,
            tee_type=terminal.tee_type,
            epsilon_spent=terminal.epsilon_spent,
        )
```

Single-frame emission means no per-token timing observable. The downside: user sees no streaming UX — output appears all at once. The Tier C trade.

The decorator is `StreamingLayerRunner`-Protocol-compatible by construction, so it composes with any inner runner (Synthetic, Autoregressive, future variants). The server's `handle_token_stream` consumes it identically.

### 3.3 FixedRateStreamingRunner (M1)

```python
class FixedRateStreamingRunner:
    def __init__(
        self, inner: StreamingLayerRunner,
        cadence_seconds: float = 0.05,
    ):
        self._inner = inner
        self._cadence = cadence_seconds
```

Implementation: drive the inner runner in a background thread (or asyncio task), buffer chunks as they arrive. Main loop emits one chunk per `cadence_seconds` tick; if the buffer is empty, emits a no-op frame (zero-length `text_delta` with no terminal flag). When the inner runner produces its terminal chunk, drain the buffer respecting the cadence, then emit the terminal.

The cadence MUST be operator-uniform across all Tier C dispatches — observable variance defeats the purpose. Operators set this once in their node config; the cadence itself becomes a public commitment.

**v1 trade-off:** the no-op-frame approach inflates total wire frame count by `total_decode_seconds / cadence`. For a 10-second decode at 50ms cadence, that's 200 frames — manageable but worth flagging in the audit-prep §7.x. A future optimization could batch consecutive no-op frames.

### 3.4 Tier C dispatch-layer gate

`make_layer_stage_server` gains a new optional kwarg `tier_c_streaming_decorator: Optional[Callable[[StreamingLayerRunner], StreamingLayerRunner]]`. When set + the dispatched request is Tier C, the server wraps the streaming runner before invocation. When unset + Tier C, the server returns `StageError(INTERNAL_ERROR, "Tier C streaming requires constant-time padding decorator")`.

The decorator is a callable rather than a class instance because it's applied per-request (different requests may have different sampling overrides; the decorator wraps the runner+request pair). Operator config in production:

```python
server = make_layer_stage_server(
    ..., streaming_runner=runner,
    tier_c_streaming_decorator=lambda inner: BatchedTrailingStreamingRunner(inner),
    # or:
    tier_c_streaming_decorator=lambda inner: FixedRateStreamingRunner(inner, cadence_seconds=0.05),
)
```

### 3.5 Honest scope

The decorator approach is a "leak masking" not "leak elimination" — even with M1 fixed-rate cadence, total stream duration leaks total token count (because the cadence × frame count = duration). Operators mitigate this by capping `max_tokens` per Tier C request; the leak ceiling becomes `max_tokens × cadence`. M2 reduces leak to a single duration value (output length leaks but per-token latency does not).

This is documented in the timing-sidechannel memo update and audit-prep §7.8.
