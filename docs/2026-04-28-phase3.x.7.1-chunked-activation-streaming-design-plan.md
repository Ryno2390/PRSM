# Phase 3.x.7.1 — Chunked Activation Streaming — Design Plan

**Status:** draft
**Date:** 2026-04-28
**Author:** PRSM core
**Predecessors:** Phase 3.x.7 (cross-host ChainExecutor — `phase3.x.7-merge-ready-20260428` tag at `56c39557`), Phase 6 Task 6b (`ShardChunker` / `ShardAssembler` already shipped).

---

## 1. Why this point release exists

Phase 3.x.7's inline path tops out at ~25 MiB raw activation (64 MiB
`MAX_HANDSHAKE_BYTES` ÷ ~2.5× hex+JSON overhead). That fits a single
2048-token × 4096-dim float16 forward pass but rejects:

- 4096-token contexts (= 32 MiB raw, exceeds inline cap)
- batched serving (batch=4 × 16 MiB = 64 MiB raw, ≈ 160 MiB hex+JSON)
- larger hidden dims (5120, 6144, 8192 — common in ≥30B parameter
  models)

Without chunked streaming, the cross-host inference path can demo
single-user short-context inference but cannot scale to production
serving regimes. The codec for chunked encoding is already shipped
(Phase 3.x.7 Task 3 — `chunk_activation` / `reassemble_chunked`
wrapping the Phase 6 Task 6b `ShardChunker`); only the transport-side
wiring is missing.

This point release closes that gap with a bounded scope change: add
an opt-in streaming wire path; auto-pick inline vs streamed based on
the existing `should_chunk` threshold; preserve back-compat with v1
inline-only peers.

---

## 2. Scope

### 2.1 In scope

- **Wire-format extension** — extend `RunLayerSliceRequest` and
  `RunLayerSliceResponse` with optional `activation_manifest`
  (`ShardManifest`) field. When present, `activation_blob` is empty
  and the chunks ride out-of-band on the streaming transport.
  Bump `CHAIN_RPC_PROTOCOL_VERSION` to `2` (v1 peers continue to
  send inline; v2 peers can send either form).
- **`ActivationChunk` wire type** — minimal envelope around the
  Phase 6 `ShardChunk` for cross-host transport.
- **`StreamedSendMessage` Protocol** — sibling to the existing
  unary `SendMessage`. Production wires to Phase 6 gRPC bidi-
  streaming RPC method. Tests inject a fake.
- **Client-side streaming dispatch** — `RpcChainExecutor._dispatch_stage`
  branches on `should_chunk(blob, threshold=CHUNK_THRESHOLD_BYTES)`.
  Inline path unchanged; streamed path mints the manifest, sends via
  `streamed_send_message`, assembles the response chunks via
  `ShardAssembler`.
- **Server-side streaming receive** — `LayerStageServer` gains a
  `handle_streamed(manifest_bytes, chunks_iter)` method that mirrors
  the 8-step validation, assembles the activation from chunks, runs
  the layer slice, returns response manifest + response chunks.
- **E2E test** — multi-MB activation scenario in
  `test_chain_rpc_e2e.py`; output bit-identical to inline reference.
- **Wiring** — `make_rpc_chain_executor` / `make_layer_stage_server`
  factories accept the optional streaming transport; defaults to
  unary-only (callers without streaming-capable transport keep
  working).

### 2.2 Out of scope (defer)

- **Streaming-token output** (chat-style incremental decode) — still
  Phase 3.x.7.x.
- **Asymmetric inline-or-streamed direction** — v2.0 keeps it simple:
  if the request was streamed, the response is also streamed. Mixed
  modes are a follow-up if measurements show value.
- **Backpressure / flow control** — relies on Phase 6 gRPC's existing
  flow control. No PRSM-layer credit windows.
- **Compression of chunks** — R7 research track.

### 2.3 Non-goals

- Replacing Phase 6 `ShardChunker` / `ShardAssembler`. The existing
  chunker is dual-use; Phase 3.x.7.1 just wires it to the chain-RPC
  surface.
- Replacing the unary path. Inline activations stay inline; this is
  additive.

## 3. Architecture

### 3.1 Mental model

```
                        ┌─────────────────────────┐
                        │ RpcChainExecutor        │
                        │ ._dispatch_stage(...)   │
                        └──────────┬──────────────┘
                                   │
                       ┌───────────┴────────────┐
            blob ≤ THRESHOLD          blob > THRESHOLD
                       │                        │
                       ▼                        ▼
        ┌─────────────────────┐  ┌──────────────────────────┐
        │ inline path (v1)    │  │ streamed path (v2)       │
        │ send_message()      │  │ streamed_send_message()  │
        │ unary → bytes       │  │ manifest + chunks_iter   │
        │                     │  │  → manifest + chunks_iter│
        └─────────────────────┘  └──────────────────────────┘
                       │                        │
                       └───────────┬────────────┘
                                   │
                                   ▼
                       ┌─────────────────────────┐
                       │ Phase 6 transport       │
                       │ (unary OR gRPC bidi)    │
                       └─────────────────────────┘
                                   │
                                   ▼
                       ┌─────────────────────────┐
                       │ LayerStageServer        │
                       │ .handle()               │
                       │ .handle_streamed()      │
                       └─────────────────────────┘
```

### 3.2 Wire format additions

```python
# protocol.py — extend existing RunLayerSliceRequest + Response.

@dataclass(frozen=True)
class RunLayerSliceRequest:
    # ... existing fields unchanged ...

    # NEW (v2): when present, activation_blob is empty (b"") and the
    # bytes ride out-of-band as ActivationChunk frames over the
    # streaming transport. The manifest's payload_sha256 commits
    # the request body to the to-be-assembled bytes — the server
    # MUST verify the assembled bytes match payload_sha256 before
    # decoding the activation.
    activation_manifest: Optional[ShardManifest] = None

    # When activation_manifest is present, activation_shape +
    # activation_dtype still describe the post-assembly tensor
    # (so the server can decode without an extra round-trip).


@dataclass(frozen=True)
class ActivationChunk:
    """Per-chunk frame for the streaming path. Wraps a Phase 6
    ShardChunk for chain-RPC transport."""

    request_id: str   # correlator binding chunk to its parent request
    sequence: int
    data: bytes
    chunk_sha256: str
    protocol_version: int = CHAIN_RPC_PROTOCOL_VERSION


@dataclass(frozen=True)
class RunLayerSliceResponse:
    # ... existing fields unchanged ...

    # NEW (v2): symmetric to request. When present, activation_blob
    # is empty and chunks ride out-of-band.
    activation_manifest: Optional[ShardManifest] = None
```

`ActivationChunk.from_dict` / `to_dict` round-trip through the
existing `parse_message` / `encode_message` codec; chunks are
small (≤1 MiB by default — well under `MAX_HANDSHAKE_BYTES`).

### 3.3 Streaming transport Protocol

```python
# client.py — sibling to SendMessage.

StreamedSendMessage = Callable[
    [str, bytes, Iterable[bytes]],
    Tuple[bytes, Iterable[bytes]],
]
"""Streaming transport: ``(address, manifest_bytes, chunk_bytes_iter)
→ (response_manifest_bytes, response_chunk_bytes_iter)``.

Production wires this to a Phase 6 gRPC bidi-streaming RPC method.
The transport sends the manifest as the first frame, then iterates
the chunks; the server processes them; the server's response manifest
arrives as the first reply frame, followed by response chunks.

Tests inject a fake that simulates the round-trip in-process."""
```

### 3.4 Client-side branching

```python
def _dispatch_stage(self, ..., activation: np.ndarray, ...) -> RunLayerSliceResponse:
    blob, shape, dtype_str = encode_activation(activation)

    if not should_chunk(blob):
        # v1 inline path — unchanged.
        return self._dispatch_inline(blob, shape, dtype_str, ...)

    if self._streamed_send is None:
        # Streamed path requires the streaming transport. Caller
        # didn't wire one → structured failure.
        raise ChainExecutionError(
            stage_index=stage_index,
            stage_node_id=stage_node_id,
            code=ExecutorErrorCode.ACTIVATION_TOO_LARGE,
            message=(
                f"activation blob {len(blob)} bytes exceeds inline "
                f"threshold {CHUNK_THRESHOLD_BYTES}; streaming "
                f"transport not wired (pass streamed_send_message= "
                f"to the executor / factory)"
            ),
        )

    return self._dispatch_streamed(blob, shape, dtype_str, ...)
```

`_dispatch_streamed` mints the activation manifest + chunks via
`chunk_activation`, builds the `RunLayerSliceRequest` with
`activation_manifest=manifest, activation_blob=b""`, sends via
`streamed_send_message`, parses the response manifest + assembles
response chunks via `ShardAssembler`. The signature verification
contract from Phase 3.x.7 H2 (require `expected_stage_node_id`) is
preserved unchanged — the response signature commits to the
post-assembly bytes via `payload_sha256` chained through
`signing_payload`.

### 3.5 Server-side handling

```python
class LayerStageServer:
    # ... existing handle(bytes) → bytes unchanged ...

    def handle_streamed(
        self,
        manifest_bytes: bytes,
        chunk_iter: Iterable[bytes],
    ) -> Tuple[bytes, Iterable[bytes]]:
        """Streamed-path sibling to handle(). Same 8-step validation
        order; chunks consumed AFTER token-anchor-verify + deadline
        check fire so we don't waste assembly cost on rejected
        requests. Returns (response_manifest_bytes,
        response_chunk_bytes_iter).

        Never raises through the wire boundary — failures map to
        StageError encoded as the manifest_bytes return with empty
        chunk_iter."""
```

Validation order matters: token-verify and deadline check fire
BEFORE chunk consumption. A peer that exhausts the stream without
a valid token wastes only the manifest parse cost.

### 3.6 Signing payload preservation

`RunLayerSliceResponse.signing_payload` already includes
`activation_blob` hex. For the streamed path, `activation_blob` is
empty AND `activation_manifest.payload_sha256` is the cryptographic
commitment to the post-assembly bytes. The signing payload extends
to include the manifest's `payload_sha256` so the stage signature
binds the same way as the inline path:

```python
@staticmethod
def signing_payload(
    request_id: str,
    activation_blob: bytes,                  # b"" if streamed
    activation_manifest: Optional[ShardManifest],
    activation_shape: Tuple[int, ...],
    activation_dtype: str,
    duration_seconds: float,
    tee_attestation: bytes,
    tee_type: TEEType,
    epsilon_spent: float,
    stage_node_id: str,
) -> bytes:
    payload = {
        "request_id": request_id,
        "activation_blob_hex": bytes(activation_blob).hex(),
        "activation_manifest_sha": (
            activation_manifest.payload_sha256
            if activation_manifest is not None else ""
        ),
        # ... existing fields unchanged ...
    }
    return json.dumps(payload, sort_keys=True).encode("utf-8")
```

This is a v2 signing payload — v1 verifiers running against v2
payloads would mismatch. Acceptable: protocol version is bumped, so
v1 ↔ v2 cross-version verification is not expected to work for
streamed responses. Inline responses on v2 still verify under v1's
payload (the new `activation_manifest_sha` field is empty string and
doesn't change the bytes). Need to verify this carefully in tests.

Actually safer move: only add the manifest_sha field to the payload
when the manifest is present. Otherwise the payload is byte-identical
to v1. Then v1 ↔ v2 inline interop works; only streamed responses
require v2 on both sides, which is enforced by the protocol_version
check anyway.

### Task 3 — Server streaming receive
**Scope:** `server.py`. Add `handle_streamed(manifest_bytes, chunk_iter)

**Tests:** Happy path streamed end-to-end via the in-process pair

## 5. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Signing-payload v1↔v2 cross-version mismatch | Medium | High | Conditional manifest_sha encoding (empty string when absent) makes v2 byte-identical to v1 for inline messages. Verify with explicit cross-version test. |
| Phase 6 gRPC bidi streaming behaviorally differs from in-process fake | Medium | Medium | E2E test exercises the in-process pair; document a follow-up "live transport" test as a Phase 3.x.7.1.x item. |
| Server allocation under hostile chunk flooding | Medium | High | `ShardAssembler` already fails-fast on out-of-order / bad-checksum chunks. Server caps total assembled bytes via existing `MAX_HANDSHAKE_BYTES` × scaling factor. Document the bound. |
| Backward-incompat protocol bump breaks v1 peers | Low | Medium | v1 peers already reject protocol_version != 1 with `ChainRpcVersionMismatchError`. v2 peers know how to talk to v1 (don't send streamed messages). Dual-version coexistence works because servers can advertise capabilities. |
| Stream cancellation mid-flight | Medium | Low | Phase 6 gRPC already handles cancellation; PRSM layer just propagates failures. Client gets `TRANSPORT_ERROR`. |
