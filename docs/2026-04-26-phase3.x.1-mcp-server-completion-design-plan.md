## 1. Context & Goals

The MCP server is PRSM's canonical user-facing surface. Per `PRSM_Vision.md` §1: *"users continue to interact with Claude, Gemini, or GPT as their conversational interface; PRSM plugs in underneath via MCP as a retrieval and heavy-compute substrate. The user never chooses — the model routes."* Every adoption KPI passes through the MCP server.

Today's MCP server (`prsm/mcp_server.py`, 1070 LOC) ships 16 tools covering query analysis, dataset browsing, agent dispatch, settlement stats, privacy budget, and training pipeline status. Per `2026-04-22-phase3-status-and-forward-plan.md` §2.3, the core tool surface is **delivered** — but five completion gaps remain before MCP can carry serious adoption load:

1. **`prsm_inference` private-inference tool** — privacy tier enforcement not fully wired to Phase 2 TEE attestation + Phase 7 content tier gating
2. **Per-tool FTNS settlement** — billing visible to MCP client; PaymentEscrow integration in `/compute/forge` exists but isn't surfaced through MCP per-tool
3. **Distribution** — npm + Python + Homebrew one-liner installers; currently must clone repo and `pip install -e .[dev]`
4. **Hosted MCP server decision** — Foundation-run "PRSM-as-a-service" vs. fully self-hosted-only; deferred decision
5. **Streaming response semantics** — partial implementation; inference streaming not fully tested through MCP path

Phase 3.x.1 closes all five gaps. This is engineering work plus one decision memo (Gap #4).

### 1.1 Why now

Promotion triggers from §3.1 of the Phase 3 status doc:

| Trigger | Status as of 2026-04-26 |
|---|---|
| T1 — adoption floor (≥20 active providers, ≥50 daily dispatches) | **NOT MET** — pre-mainnet |
| T2 — dependencies green (Phase 2 TEE + Phase 7 content tier LIVE) | **MET** — both shipped per memory |
| T3 — partner ask (MCP-client ecosystem partner expresses interest) | **NOT MET** — outreach not started |

T1 and T3 are adoption-side; T2 is the engineering blocker. T2 is now met, which means engineering can proceed ahead of demand. This is "build the canonical UX surface so it's ready when adoption arrives" — the work doesn't get easier by waiting.

### 1.2 Non-goals for Phase 3.x.1

- **Not a tool-surface expansion beyond the 5 named gaps.** New tools (e.g., `prsm_train`, `prsm_governance_vote`, `prsm_recovery`) are out of scope; this phase fills existing gaps.
- **Not a redesign of the existing 16 tools.** Existing tools work; this phase adds and completes, doesn't refactor.
- **Not a new vendor relationship.** Distribution uses existing channels (PyPI already live, npm and Homebrew are first-time but standard).
- **Not a self-improving / reflective MCP server.** Standard tool definitions, not generative.
- **Not a hosted-MCP-server implementation.** Decision memo only; if go-decision, that's a separate Phase 3.x.2 work item.

### 1.3 Backwards compatibility

- All 16 existing tools' interfaces preserved unchanged
- New `prsm_inference` is additive (does not modify `prsm_analyze`)
- Distribution channels are additive (PyPI install path remains canonical)
- Streaming is opt-in (existing clients receive non-streaming responses by default)

### 3.2 Privacy tier mapping

Per `PRSM_Vision.md` §7, two layers of privacy:
- **Layer 1 (data-layer confidentiality):** content tier A / B / C — encryption status of data being queried
- **Layer 2 (inference-layer confidentiality):** TEE attestation + DP noise on activations

The `prsm_inference` tool exposes both via two parameters:

| Parameter | Values | Effect |
|---|---|---|
| `content_tier` | `A` (default) | Public content; node operators see plaintext shards |
| | `B` | Encrypted-before-sharding; ciphertext-only on operators; key released on payment |
| | `C` | Tier B + erasure coding + Shamir-split keys; reconstructs only with K-of-N + M-of-N quorum |
| `privacy_tier` | `none` | No DP noise; `ε=∞`; standard inference |
| | `standard` (default) | DP noise; `ε=8.0` |
| | `high` | Strong DP; `ε=4.0` |
| | `maximum` | Tightest DP; `ε=1.0`; significant accuracy cost at scale |

Privacy budget is tracked per-node per `PrivacyBudget` (existing infrastructure); `prsm_privacy_status` already surfaces this.

### 3.3 TEE attestation receipt

Each inference returns a receipt with:

```python
@dataclass(frozen=True)
class InferenceReceipt:
    job_id: str                    # globally unique
    model_id: str                  # which model was executed
    content_tier: str              # A / B / C
    privacy_tier: str              # none / standard / high / maximum
    epsilon_spent: float           # actual DP epsilon consumed
    tee_type: str                  # SGX / TDX / SEV-SNP / TrustZone / Software
    tee_attestation: bytes         # platform-specific attestation blob
    output_hash: bytes             # SHA-256 of inference output
    duration_seconds: float
    cost_ftns: Decimal             # actual FTNS settled
    settler_signature: bytes       # Ed25519 sig from settling node identity
```

Verification flow:
1. Caller receives receipt + output
2. Caller verifies `settler_signature` against settling node identity
3. Caller verifies `tee_attestation` against platform vendor's attestation service (Intel ASP for SGX/TDX, AMD KDS for SEV-SNP)
4. Caller confirms `output_hash` matches received output

This converts MCP inference from "trust the provider" to "verifiable inference" — a Phase 7 §A1 commitment in the auditor briefing.

### 3.4 Per-tool billing visibility

Currently MCP responses include text only. Extension: structured response embedding cost reconciliation:

```
PRSM Inference Result
=====================

[inference output here]

—————————————————————————————————————————————————————
Job ID: forge-a1b2c3d4
Cost: 2.34 FTNS ($4.80 USD-equivalent at $2.05 FTNS)
Privacy: standard (ε=8.0); 7.2 ε remaining in budget
Content tier: A (public)
TEE: SGX (hardware-attested)
Duration: 1.7s
—————————————————————————————————————————————————————
```

The MCP client (LLM) sees this; the LLM can surface cost info to the user contextually. New `prsm_billing_status(job_id)` tool lets the LLM query historical job cost.

### 3.5 Distribution channel architecture

**npm package** (`prsm-mcp`):
- Wraps Python MCP server in Node.js entrypoint
- `npx prsm-mcp` installs and runs locally
- Auto-detects Python via `python3` or `py` (Windows)
- Auto-installs `prsm-network` Python package on first run
- Why: every MCP client (Claude Desktop, Cursor, etc.) speaks `npx`; lowest friction

**Python package** (`prsm-network`, already on PyPI):
- Existing path: `pip install prsm-network` then `prsm mcp-server`
- Continue maintaining; npm wraps this

**Homebrew tap** (`prsm/homebrew-tap`):
- `brew install prsm/tap/prsm`
- Bottles for macOS Intel + Apple Silicon
- Linux via Linuxbrew

### 3.6 Streaming semantics

MCP supports partial responses via `Server.send_progress_notification()` and progressive `TextContent` chunks. Streaming wins are largest for:
- `prsm_inference` — inference can stream tokens as model generates
- `prsm_analyze` — agent dispatch can stream intermediate results from each shard
- Long-running tools (`prsm_dispatch_agent`)

Tools that return small fixed-size results (e.g., `prsm_revenue_split`, `prsm_quote`) don't need streaming.

Implementation approach:
- Tool handlers that benefit from streaming: change return type to `AsyncIterator[str]`
- MCP server dispatcher: detects iterator return, emits each chunk as progress notification
- Non-streaming clients (older MCP clients) receive concatenated final output (transparent fallback)

## 6. Open Issues

### 6.1 Model registry: where do `model_id`s come from?

`prsm_inference(prompt, model_id, ...)` requires a model registry. Options:
- **A.** Foundation-curated registry (small, hand-picked open-weight models like Llama 3, Mistral, Phi)
- **B.** Community-uploaded registry (any model can be sharded + registered)
- **C.** Both — Foundation seeds with curated set, community can extend

Recommendation: Start with **(A)** for Phase 3.x.1; promote to **(C)** once usage proves demand. Avoid **(B)** as default to prevent malicious-model dispatch.

Decision deferred to Task 4 design review.

### 6.2 Billing: pre-pay vs. post-pay

Current `/compute/forge` locks escrow up-front. For inference, this works but creates UX friction (user must specify budget; if budget exceeds actual cost, refund happens but takes a transaction).

Alternative: post-pay (run inference, settle on completion). Lower friction but exposes provider to walk-aways.

Recommendation: keep pre-pay for Phase 3.x.1; add post-pay as Phase 4 wallet-SDK feature (delegated authority to settle from user wallet without per-call lock).

### 6.3 Hosted MCP server: defer or now?

Gap #4 (decision memo) addresses this. Strong recommendation: defer to post-mainnet, revisit after 6 months of self-hosted MCP usage data. Hosted service implies Foundation custody of user FTNS for billing, which complicates Phase 5 KYC scope.

### 6.4 npm package distribution risk

Distributing via npm exposes PRSM brand to potential package-spoofing attacks. Mitigations:
- Reserve `prsm-mcp`, `prsm`, `@prsm/mcp` namespaces immediately
- Sign packages with foundation signing key
- Document official package names prominently

Open: who maintains package signing keys? (Pre-board: founder. Post-board: foundation council multi-sig per Foundation Officer JDs.)

### 6.5 TEE software fallback for nodes without hardware TEE

Software TEE (just process isolation) is a weaker guarantee than hardware TEE (SGX/TDX/SEV-SNP/TrustZone). Should `prsm_inference` accept software-TEE-only nodes?

Recommendation: `privacy_tier=none` accepts software-TEE; `privacy_tier=standard/high/maximum` requires hardware TEE. Receipt clearly indicates which was used.

## 8. Vendor / Approach Decisions

### 8.1 Embedded Python in npm package vs. shell-out

**Decision:** Shell-out to user-installed Python (`python3 -m prsm.mcp_server`).

**Rationale:**
- Embedding Python in npm package adds ~50-100MB per-platform; auto-update friction
- PRSM users will have Python anyway (PyPI install is canonical)
- npm wrapper just provides convenience for the MCP client integration

### 8.2 Streaming protocol: MCP native vs. SSE bridge

**Decision:** MCP native streaming via `progress_notification`.

**Rationale:**
- MCP has progress-notification primitive built-in
- SSE adds dependency for a feature MCP supports natively
- Backwards-compat: non-streaming clients ignore notifications and use final response

### 8.3 Model registry custody

**Decision (per §6.1):** Foundation-curated initial set; community-extensible later.

**Rationale:**
- Reduces malicious-model dispatch risk during bootstrap
- Foundation curation = ~5-10 known-good open-weight models (Llama 3 8B, Mistral 7B, Phi-3, Whisper, etc.)
- Community extension via governance proposal once registry is live
