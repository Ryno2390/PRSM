On 2026-05-08, the Foundation shipped 42 commits across 31 merge-ready tags + 14 cumulative audit-prep tags spanning three distinct architectural arcs:

- **Morning arc** (`project_phase78_operator_surface_2026_05_08.md` per founder memory): 4 modules in `prsm/economy/web3/` (CompensationDistributorClient + StorageSlashingClient + HeartbeatScheduler + PullAndDistributeScheduler) + initial Node wiring + audit-prep §7.20.
- **Afternoon arc** (`project_phase78_afternoon_arc_2026_05_08.md`): 3 event watchers + completing Node wiring + fleet kill-switch scoping doc + Vision/Tokenomics doc updates for Aerodrome + Coinbase CDP architecture + audit-prep §7.21 + §7.22.
- **Evening arc** (`project_phase5_mcp_composers_evening_arc_2026_05_08.md`): 2 MCP composer tools (`prsm_balance_check` + `coinbase_offramp_initiate`) + 2 backend endpoints + system-level integration tests + PARTICIPANT_GUIDE refresh + audit-prep §7.23 / §7.24 / §7.25 / §7.26.

End-of-day cumulative: **Vision §13 Phase 5 cash-out flow end-to-end at MCP composer scope; Phase 7-storage + Phase 8 operator surface end-to-end (contract INVOCATION + OBSERVATION); fleet kill-switch design-ratified-trigger-gated; documentation surfaces (OPERATOR_GUIDE + PARTICIPANT_GUIDE + Vision/Tokenomics) all reflect mainnet-deployed reality.**

### 2.2 Why a single ratification doc

Each arc's individual deliverables fall within the founder's existing operational authority under the Foundation governance charter (engineering work, doc updates). What requires explicit council ratification at end-of-day is:

1. **Coherence of the wiring** — three arcs separately would not require ratification, but together they constitute a meaningful architectural commitment (the cash-out flow end-to-end commits PRSM publicly to the Aerodrome + Coinbase CDP path).
2. **Three regression-discipline rules** that emerged from the day's audit-prep entries — these need to be on the formal record, not just inline in audit-prep §§7.24 / 7.26 / 7.21.
3. **Two trigger-gated implementation paths** (fleet kill-switch Phase F1 + `coinbase_offramp_initiate` execute path) where pre-commission ratification of the framework — but NOT pre-authorization of the actual implementation — bounds future founder discretion.
4. **One commission-gated funding decision** (Aerodrome pool seeding source: Foundation treasury vs Prismatica balance sheet) where today's docs publicly commit the Foundation to the Vision §13 Phase 5 architecture; this resolution flags that as a future decision requiring its own ratification when the gantt date approaches.

### 2.3 Context — today's mainnet posture

As of this resolution:

- All Phase 7-storage + Phase 8 contracts are live on Base mainnet (per PRSM-CR-2026-05-06-3 + 2026-05-07 deploy ceremony per memory `project_mainnet_full_stack_live_2026_05_07.md`).
- The operator-side Python surface (clients + daemons + watchers) is end-to-end at composer scope as of today's three arcs.
- The user-facing MCP composer surface (`prsm_balance_check` + `coinbase_offramp_initiate`) is shipped at composer scope; the latter explicitly returns `status: "PENDING_COMMISSION"` and does NOT initiate any on-chain or fiat-side action.
- The Aerodrome USDC-FTNS pool is NOT yet seeded (gantt 2026-06-15); the `PRSM_FTNS_USD_RATE` env var serves as a static USD-rate placeholder.
- Coinbase CDP production credentials are NOT yet commissioned.

Tags: `prsm-balance-check-mcp-v1-merge-ready-20260508` + `coinbase-offramp-initiate-v1-merge-ready-20260508` + `phase78-mcp-integration-tests-merge-ready-20260508`.

**RESOLVED 4 — Adopt the fleet kill-switch design (trigger-gated, NOT pre-authorize implementation).** The council adopts FLEET-KILL-SWITCH-SCOPING-1 (`docs/2026-05-08-fleet-kill-switch-scoping.md`) as the **ratified design framework** for future implementation. The recommended phased C → A migration, the four-tier issuance authority schedule (§6.1: P0 3-of-5 / P1 single-actor / deactivation 3-of-5 / list-expansion 4-of-5 + auditor), the granularity bound at the 7 monitored subsystems catalogued in `EXPLOIT_RESPONSE_PLAYBOOK_ANNEX_2026_05.md` §4, the seven explicit non-goals in §9, and the six named §7 promotion triggers are all **incorporated by reference**. **Implementation is NOT pre-authorized**; when any §7 trigger fires, the implementation work itself requires its own founder action under the operational authority granted by the governance charter.

**RESOLVED 5 — Ratify the doc/Tokenomics updates.** Today's updates to `PRSM_Vision.md` §13 Phase 5 (Aerodrome + Coinbase CDP + MCP architecture) and `PRSM_Tokenomics.md` (§3.5 reframe + §3.7 Aerodrome pool architecture + §4.10 USD-denominated pricing thesis + §5.5 Coinbase CDP regulatory framing) are **ratified as foundation positions**. The OPERATOR_GUIDE and PARTICIPANT_GUIDE refreshes (commits `36784694` + `9c26f07d`) are **ratified as the canonical operator-facing and end-user-facing surfaces** for the mainnet-deployed reality.

**RESOLVED 6 — Adopt three regression-discipline rules emerging from today's audit-prep entries.**

> **Rule R-2026-05-08-1 (Composer-only invariant on `coinbase_offramp_initiate`).** Per audit-prep §7.24 + §7.25, introducing an actual-execution path on `coinbase_offramp_initiate` (e.g., adding an `execute=true` argument that triggers on-chain swap or fiat off-ramp) without explicit council ratification of the commission completion + audit-trail mechanism = **P0 audit finding**. The MCP tool's `status: "PENDING_COMMISSION"` envelope is the load-bearing claim users see; introducing execution silently would deceive users that their funds aren't moving when they are. CI must pin the composer-only invariant until the commission-completion ratification doc supersedes it.
>
> **Rule R-2026-05-08-2 (Tokenomics ↔ PARTICIPANT_GUIDE coherence).** Per audit-prep §7.26, when `PRSM_Tokenomics.md` is updated, ALSO run a freshness check on `docs/PARTICIPANT_GUIDE.md` to catch contradicting wording. The participant-facing doc surfaces tokenomic claims for end-users; investors compare narratives across both surfaces. Without this discipline, future Tokenomics edits can introduce silent contradictions that auditors / investors pick up via doc-walking. Future Tokenomics-update commits SHOULD include a `PARTICIPANT_GUIDE.md` freshness verification (or explicit no-op rationale) in the commit message.
>
> **Rule R-2026-05-08-3 (Fleet kill-switch authority preservation).** Per FLEET-KILL-SWITCH-SCOPING-1 §6.1, the four-tier issuance authority schedule (P0 3-of-5 / P1 single-actor / deactivation 3-of-5 / list-expansion 4-of-5 + auditor) is the binding council commitment for any future implementation. Future implementation work that introduces alternative authority structures requires a new council resolution superseding this commitment.

**RESOLVED 7 — Pre-commit to two trigger-gated implementation paths.** When the §7 triggers from FLEET-KILL-SWITCH-SCOPING-1 fire, the council pre-commits to the recommended **phased C → A migration**: Phase 1 (HTTPS-pull architecture, ~1-2 weeks) ships first; Phase 2 (on-chain `FleetKillSwitch.sol` + external audit + 30-day Sepolia bake-in, ~4-6 weeks) ships once audit clears; Phase 3 retires HTTPS-pull 90 days after on-chain mainnet activation. Architecture B (DHT-broadcast) is documented-but-rejected per the scoping doc and is NOT pre-authorized as an alternative path.

**RESOLVED 8 — Flag commission-gated funding decision for future ratification.** The Aerodrome USDC-FTNS pool seeding source (Foundation treasury vs Prismatica balance sheet) is a **deferred decision** requiring its own council resolution when the Vision §13 Phase 5 gantt date (2026-06-15) approaches. Today's doc updates publicly commit the Foundation to the Aerodrome + Coinbase CDP architecture, but explicitly do NOT pre-authorize either funding source. The seed amount range ($250K-$1M USDC + matched FTNS per Tokenomics §3.7) is the ratified envelope; specific source + amount require their own resolution.

**RESOLVED 9 — Annual review.** This resolution is calendared for annual review on 2027-05-08. Trigger-event review is immediate upon any of: (a) §7 fleet kill-switch promotion trigger fires, (b) Aerodrome pool seeding date approaches, (c) Coinbase CDP commissioning completes, (d) any of the three R-2026-05-08-* regression-discipline rules is breached or considered for amendment.

## 5. Explicit non-scope

This resolution does NOT:

- **Authorize the actual fleet kill-switch implementation.** §7 trigger-gated; founder executes when triggers fire; Phase F1 itself is operational founder work, not pre-authorized by this resolution.
- **Authorize the `coinbase_offramp_initiate` execute path.** Composer-only invariant ratified; execute path requires its own resolution post-commission.
- **Authorize Aerodrome pool seeding from any specific funding source.** Architecture and seed-amount envelope ratified; source decision deferred per RESOLVED 8.
- **Authorize Coinbase CDP production credential commissioning.** That's an external-vendor-engagement event; when it occurs, today's composer-only tools gain the execute path, but the commissioning itself is operational work.
- **Modify the FTNS economic parameters** (`MAX_SUPPLY`, `NETWORK_FEE_BPS`, the 10/2/88 split, or the halving schedule).
- **Modify the Foundation Safe's signer set or signing threshold.**
- **Authorize any minting beyond `INITIAL_SUPPLY = 100M`** (already issued).
- **Authorize any treasury disbursement** (governed by PRSM-POL-1).
- **Modify the FTNS role-admin state** established in PRSM-CR-2026-05-06-3.
- **Waive the L8 Track B gate** for any future Prismatica investor solicitation. (PRSM-CR-2026-05-06-5 retains L8 Track B; this resolution does not affect that.)
- **Permanently retire any of the three R-2026-05-08-* regression-discipline rules.** They may be amended by future council resolution, but the §9 annual review trigger requires explicit consideration before any amendment.
