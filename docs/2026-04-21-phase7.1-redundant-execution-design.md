**Date:** 2026-04-21
**Depends on:** `phase7-merge-ready-20260421` (Tier C stake + slashing primitives)
**Blocks:** completion of the verification triad (Tier A receipt-only, Tier B redundant execution, Tier C single-provider + stake-slash)
**Mainnet coupling:** wants to land in the same auditor engagement as Phase 3.1 Task 10 + Phase 7 Task 9. Engineering-only work; no hardware dependency.

---

## 1. Context & Goals

Phase 7 closed the stake-and-slash gap: a premium or critical job runs on one provider, and if that provider misbehaves (double-spends a receipt, forges a signature), they forfeit stake. **That only works when misbehavior is provable on-chain from a single receipt.**

Some misbehavior isn't. A provider can silently return wrong outputs — the signature verifies, no receipt is double-spent, nothing on-chain flags the error. The only way to catch a wrong-answer cheater is **run the same shard on k providers and compare outputs**. That's Tier B.

**Phase 7.1 scope:**

- `k`-of-`n` redundant execution (default k=3) for jobs that opt in via `DispatchPolicy.consensus_mode`.
- Per-shard output-hash consensus: all k providers execute the same shard on identical inputs; the requester compares their `output_hash` values; the majority hash wins.
- On disagreement: minority providers are challenged on-chain with a new `CONSENSUS_MISMATCH` reason code. Their stake is slashed via the existing Phase 7 slash path.
- Per-provider receipts stay individually signed — minority receipts are the cryptographic proof consumed by the on-chain challenge handler.

### 1.1 Non-goals for Phase 7.1

- **Byzantine fault tolerance (3f+1).** MAJORITY mode ships; BYZANTINE mode designed but deferred to Phase 7.1x.
- **Floating-point tolerance in output comparison.** Hash equality stays the contract (consistent with Phase 2's float64 bitwise-reproduction invariant). Tolerance comparison lives in `prsm/node/result_consensus.py` for *job-level* results; shard-level stays hash-exact.
- **TEE-backed execution proofs.** Phase 2 Ring 9 TEE attestation already exists; it's orthogonal and composes (a Tier-B job can also require TEE).
- **Cross-shard consensus.** Each shard consenses independently. Full-job consensus is the product of per-shard consensus.

### 1.2 Backwards compatibility

- `DispatchPolicy.consensus_mode` defaults to `None` → single-provider dispatch exactly as Phase 7 behaves.
- `RemoteShardDispatcher` untouched. MultiDispatcher wraps it.
- Existing `ShardExecutionReceipt` format unchanged. New `ConsensusShardReceipt` is a container around k of them.
- `BatchSettlementRegistry` gets a new `ReasonCode.CONSENSUS_MISMATCH` (enum extension — safe per Phase 3.1 design §7).

Out of scope (deferred):

- Adaptive k (raise k on provider disagreement trends, lower k when the pool is consistently honest).

## 3. Protocol

### 3.1 Consensus modes

Reuse `ConsensusMode` from `prsm/node/result_consensus.py`:

```python
class ConsensusMode(str, Enum):
    SINGLE = "single"         # Phase 7 behavior — k=1, no consensus
    MAJORITY = "majority"     # Phase 7.1 default — k=3, need 2 agreeing
    UNANIMOUS = "unanimous"   # k=3, all 3 must agree (strictest; rarely used)
    BYZANTINE = "byzantine"   # Phase 7.1x — 3f+1 for f faults
```

For Phase 7.1 MVP: MAJORITY with k=3 is the shipping default. UNANIMOUS is wired for tests but not the default (one honest disagreement shouldn't fail the job). BYZANTINE is a stub that raises `NotImplementedError` until 7.1x.

### 3.2 k-of-n selection

From the policy-eligible pool, MultiDispatcher picks `k` providers. Selection rule:

**Top-k by composite score = reputation × (1 / advertised price_per_shard_ftns).**

Ties broken by lexicographic provider_id (deterministic — the requester can audit why a given set was picked).

Rejection: if the eligible pool has fewer than `k` providers, fail the job immediately with `InsufficientEligibleProvidersError` (don't quietly degrade to k-1; the requester asked for a specific consensus mode).

### 3.3 Parallel dispatch

MultiDispatcher calls `RemoteShardDispatcher.dispatch_with_receipt` for all `k` providers **concurrently** via `asyncio.gather`. The escrow contribution per provider is `quoted_price_ftns / k` (equal split), drawn from the requester's single escrow.

Partial failures: if `m < k` providers return success and `k - m` timeout / preempt / error, the consensus group is the `m` who responded. The majority-hash rule applies to `m` (not to the full `k`). If `m < ceil(k/2)` — i.e., not enough responses to establish a majority — raise `InsufficientResponsesError` and refund escrow from the responders.

### 3.4 Output-hash consensus

Each successful response carries a `ShardExecutionReceipt` with an `output_hash` (SHA256 of the raw output bytes per Phase 2's `build_receipt_signing_payload`). MultiDispatcher groups receipts by `output_hash` and finds the largest group:

- **MAJORITY mode:** largest group size ≥ `ceil(k/2) + 1` (so 2 of 3, 3 of 5) — accept; return the group's shared output.
- **UNANIMOUS mode:** largest group size = `k` — accept; otherwise reject.
- **BYZANTINE mode (7.1x):** largest group size ≥ `2f + 1` where `f = (k-1)/3`.

On acceptance, the "minority" is `{provider_id}` for all provider receipts NOT in the accepted group. Minority receipts are captured for downstream slashing.

### 3.5 ConsensusShardReceipt

```python
@dataclass(frozen=True)
class ConsensusShardReceipt:
    job_id: str
    shard_index: int
    consensus_mode: ConsensusMode
    k: int                                  # intended k
    responded: int                          # actual responses
    agreed_output_hash: str                 # winning hash
    majority_receipts: list[ShardExecutionReceipt]   # all receipts in accepted group
    minority_receipts: list[ShardExecutionReceipt]   # disagreeing receipts (may be empty)
    consensus_reached_unix: int
```

Minority being empty is the happy case — all k providers agreed. When minority is non-empty, the orchestrator's post-dispatch hook feeds minority receipts into the challenge pipeline.

### 3.6 Challenge path

New `ReasonCode.CONSENSUS_MISMATCH` (value = 5; appends to the enum per §7 of Phase 3.1 design). **Cross-batch structure mirrors DOUBLE_SPEND's.** In Phase 7.1's k-of-n model each provider commits their own single-receipt batch of their own output — the challenge against a minority provider uses a majority provider's (different) batch as proof of disagreement, and the slash naturally hits `b.provider` (the committer of the minority batch).

auxData layout:

```solidity
abi.encode(
  bytes32 conflictingBatchId,   // the majority provider's batch
  bytes32[] majorityProof,      // proof for the majority leaf
  ReceiptLeaf majorityLeaf      // the majority receipt
)
```

Contract verifies:
- `msg.sender == minorityBatch.requester` — MVP authorization.
- `conflictingBatchId != batchId` — distinct batches (a provider cannot disagree with themselves inside one batch; blocks self-referential challenges).
- `leaf.jobIdHash == majorityLeaf.jobIdHash` — same job.
- `leaf.shardIndex == majorityLeaf.shardIndex` — same shard (otherwise "disagreement" is meaningless).
- `leaf.outputHash != majorityLeaf.outputHash` — genuinely different outputs.
- `batches[conflictingBatchId]` exists.
- `MerkleProof.verify(majorityProof, batches[conflictingBatchId].merkleRoot, hash(majorityLeaf))` — the majority leaf is provable in the conflicting batch.

If verified, `_handleConsensusMismatch` returns true, the minority receipt is invalidated, and the slash hook fires against `b.provider` (= minority's committer) with their bond-time `tier_slash_rate_bps` snapshot.

**Why the requester is the only allowed challenger** (MVP): only the requester knows both receipts came from their single k-of-n job. If we open this to any third party, a provider could forge two batches with different outputs for the same shard and trigger a self-slash challenge to mine the 70% bounty. The Phase 7.1x refinement is to make the batches carry a `consensus_group_id` that binds k receipts together so a third party can verify membership — deferred.

> **Task 7 design-gap note (2026-04-21):** the initial Task 3 implementation verified the majority proof against the minority batch's own root, which only works if a single provider commits BOTH leaves in one batch — but k-of-n has k different providers, each committing their own batch. During the Task 7 E2E we refactored `_handleConsensusMismatch` to the cross-batch pattern above (mirroring DOUBLE_SPEND's structure), which is how `b.provider` ends up pointing at the minority's stake naturally. The Task 3 test file was rewritten accordingly (see `contracts/test/BatchSettlementConsensus.test.js`).

If we open this to any third party, a provider could forge two batches with different outputs for the same shard and trigger a self-slash challenge to mine the 70% bounty. ### 3.7 Slashed FTNS routing

Reuses Phase 7 §3.4 exactly. Requester challenges a minority provider → 70% bounty to requester, 30% to Foundation. Self-slash protection (`challenger == provider`) still applies.

Pure function — no I/O, no async. Unit-testable with synthetic receipts.

### 4.2 `prsm/compute/multi_dispatcher.py` (new)

```python
class MultiShardDispatcher:
    def __init__(
        self,
        single_dispatcher: RemoteShardDispatcher,
        consensus: ShardConsensus,
    ): ...

    async def dispatch_with_consensus(
        self,
        shard: ModelShard,
        input_tensor: np.ndarray,
```

Errors raised:
- `InsufficientResponsesError` — fewer responses than consensus threshold.
- `ConsensusFailedError` — threshold met in count but no agreeing group of sufficient size (e.g., k=3 and all 3 returned different hashes).

### 4.3 `DispatchPolicy` extension

When `consensus_mode is None`, orchestrator uses Phase 7's single-provider path. When set, MultiDispatcher is invoked.

### 4.4 `BatchSettlementRegistry.sol` extension

```solidity
enum ReasonCode {
    DOUBLE_SPEND,
    INVALID_SIGNATURE,
    NO_ESCROW,
    EXPIRED,
    MALFORMED,
    CONSENSUS_MISMATCH    // Phase 7.1 — appended; value = 5
}

function _handleConsensusMismatch(
    ReceiptLeaf calldata minorityLeaf,
    bytes calldata auxData
) internal view returns (bool) {
    (bytes32 majorityAgreedHash,
     bytes32 minorityClaimedHash,
     bytes32[] memory majorityProof,
     ReceiptLeaf memory majorityLeaf) =
        abi.decode(auxData, (bytes32, bytes32, bytes32[], ReceiptLeaf));

    // Bind the passed hashes to the leaves.
    if (keccak256(abi.encode(minorityLeaf.outputHash)) != minorityClaimedHash) return false;
    if (keccak256(abi.encode(majorityLeaf.outputHash)) != majorityAgreedHash) return false;

    // Genuinely different outputs.
    if (majorityLeaf.outputHash == minorityLeaf.outputHash) return false;

    // Majority leaf must be in the same batch.
    Batch storage b = batches[/* derived from caller context */];
    if (!MerkleProof.verify(majorityProof, b.merkleRoot, _hashLeaf(majorityLeaf))) {
        return false;
    }

    // Caller must be the batch requester (MVP).
    if (msg.sender != b.requester) return false;

    return true;
}
```

The hook into `challengeReceipt` adds one more branch (append to the existing if/else chain in the handler dispatch at line 398). Slashing check gains `reason == ReasonCode.CONSENSUS_MISMATCH` in the slashable set.

## 5. Integration points

### 5.1 MarketplaceOrchestrator

New branch in `_dispatch_one_shard`:

```python
if policy.consensus_mode is not None and policy.consensus_mode != "single":
    # Phase 7.1 path
    selected = self._select_top_k(eligible, policy.consensus_k)
    if len(selected) < policy.consensus_k:
        raise InsufficientEligibleProvidersError(...)
    consensus_receipt = await self.multi_dispatcher.dispatch_with_consensus(
        shard=shard, input_tensor=input_tensor,
        node_ids=[l.provider_id for l in selected],
        job_id=job_id, ...
    )
    # Feed minority receipts to the challenge pipeline (if any).
    if consensus_receipt.minority_receipts:
        await self._queue_consensus_challenges(consensus_receipt, selected)
    return consensus_receipt.agreed_output_hash_as_ndarray
```

The consensus path is invisible to callers who don't set `consensus_mode`.

### 5.2 ReputationTracker

No code changes. `record_slash` already accepts arbitrary reason strings. The E2E test asserts `CONSENSUS_MISMATCH` flows through `has_been_slashed` / `get_slash_events` like any other reason.

### 5.3 BatchSettlementRegistry — new challenge helper

Phase 3.1 Task 3 established the challenge handler dispatch pattern. Phase 7.1 adds one more handler (§4.4 above). The existing slash hook already runs on any slashable reason via:

```solidity
if (... && (reason == ReasonCode.DOUBLE_SPEND || reason == ReasonCode.INVALID_SIGNATURE))
```

This becomes:

```solidity
if (... && (reason == ReasonCode.DOUBLE_SPEND
            || reason == ReasonCode.INVALID_SIGNATURE
            || reason == ReasonCode.CONSENSUS_MISMATCH))
```

One-line change.

### 5.4 Escrow accounting

Phase 3.1's `EscrowPool` operates in FTNS base units and doesn't care about per-provider allocation — the batch total covers all committed receipts. For Phase 7.1, the requester's escrow is still a single deposit; the batch's `totalValueFTNS` now includes k receipts per shard instead of 1. MultiDispatcher passes `escrow_amount_per_provider_ftns = quoted_price / k` to each single-dispatch call internally, but the registry sees a normal batch.

## 7. Acceptance criterion

A single end-to-end scenario demonstrates Phase 7.1:

> **Given** a requester policy with `consensus_mode="majority"` and `consensus_k=3`, and three providers A/B/C each bonded at `critical`, **when** the orchestrator dispatches shard N across all three and providers A+B return hash H₁ while provider C returns hash H₂ ≠ H₁, **then** (1) the orchestrator returns the H₁ output, (2) the ConsensusShardReceipt's minority list contains C's receipt, (3) the requester's challenge with `CONSENSUS_MISMATCH` succeeds on-chain, (4) C's stake is slashed, (5) the requester receives the 70% bounty after `claim_bounty()`, and (6) `ReputationTracker.record_slash(C, reason="CONSENSUS_MISMATCH")` records the event.

## 8. Open issues

### 8.1 Adaptive k

Running k=3 on every job is expensive (3× the compute cost). A smarter policy would raise k only for jobs above a value threshold, or when the pool's observed agreement rate drops. Phase 7.1 MVP: fixed k per job, set by the requester. Phase 7.1x: an advisory service that recommends k based on reputation distribution of the pool.

### 8.2 Provider knowledge of redundancy

**Open security question:** should providers know they're on a k-of-n job? Arguments for transparency: providers can price differently if they know. Arguments for opacity: providers who know they're being verified perform better, which masks the true honesty signal we want to measure. Phase 7.1 MVP: providers do not know (no flag in the dispatch message). Revisit if the asymmetry causes price-discovery issues.

### 8.3 Cost allocation under partial failure

If 3-of-3 was requested but only 2 respond and agree, does the requester pay the third provider? Current plan: no — unresponsive providers forfeit their escrow share (Phase 2's existing timeout semantics). Worth flagging to the auditor because it creates a weak incentive for malicious providers to participate just enough to earn the price-handshake fee but not actually execute.

### 8.4 Consensus across non-deterministic operations

Phase 2 hardcodes float64 and specifies bitwise-identical outputs. Some compute kernels (e.g., CUDA matrix multiplication) have non-deterministic reduction order → identical input can produce slightly different float64 output bits on different hardware. This would cause false CONSENSUS_MISMATCH. **Mitigation:** Phase 7.1 restricts redundant execution to operations that are provably deterministic under the current executor contract (today: all scalar + matrix operations via numpy's default reducers, which ARE deterministic). GPU compute paths need a separate determinism audit — deferred to Phase 7.1x or a CIS (Confidential Inference Standard) extension.

### 8.5 Challenger-authorization surface for CONSENSUS_MISMATCH

MVP restricts the challenge to the batch's requester. This loses the "anyone can watch the chain and collect a bounty" property that DOUBLE_SPEND enjoys. Phase 7.1x: introduce a `consensus_group_id` field in the batch that binds k receipts together; any third party holding all k receipts can challenge. Weighed against this: the current one-requester restriction matches NO_ESCROW's access model, so the precedent is already in the contract.

### 8.6 Consensus-minority queue persistence (from Task 8 review)

`MarketplaceOrchestrator.consensus_minority_queue` is an in-memory list, never drained and never persisted. In production, a process crash or restart between dispatch and challenge-submission drops queued minority receipts on the floor. Missing pieces for a complete production flow:

- Drain API (e.g., `pop_pending_challenges(batch_id)`) so a submitter service can pull work.
- Persistence across process restart. Phase 3.1 already has DB infrastructure (SQLite or Redis); a small table keyed on `(job_id, shard_index)` would suffice.
- Handoff hook from the settlement accumulator that delivers the `batch_id` + Merkle proof when the minority's batch commits on-chain — that's when the challenge becomes submittable.

**Disposition:** carried into Phase 7.1x as the "consensus-submitter service" deliverable. Not a merge-blocker for `phase7.1-merge-ready` because the orchestrator's in-process queue IS a clean seam with explicit design-doc notes that a downstream service owns submission. Added to the audit-prep bundle so auditors know minority receipts currently accumulate without on-chain action.

### 8.7 Sybil-requester griefing vector (from Task 8 review)

The MVP auth model (requester-only challenger) combined with 70/30 bounty split opens a griefing-but-not-farming vector:

- Attacker controls both a provider EOA P and a "requester" EOA R (distinct addresses).
- Attacker commits two batches from P for the same job/shard with different outputs. R is recorded as both batches' requester.
- R challenges one of P's batches, using the other as conflicting-batch evidence.
- StakeBond's self-slash protection (`challenger == provider`) does NOT trigger because R ≠ P, so the 70/30 split fires normally: 70% bounty to R, 30% to Foundation.

**Net attacker cost:** 30% of slashed amount + gas + the reputation crush on P. At `critical` tier this is ≈15K FTNS net loss per stake slot burned plus P is excluded from future premium/critical dispatches. Negative EV by construction — this is griefing (costs the attacker real money to harass the Foundation's reserve), not farming (the attacker cannot net-profit).

**Disposition:** acknowledged; Phase 7.1x's `consensus_group_id` binding (§8.5) is the planned fix — it would require the attacker to forge TWO providers' signatures to trigger, not one provider + one requester. Added to the audit-prep bundle for auditor visibility.

### 8.8 asyncio.TimeoutError / CancelledError propagation (from Task 8 review)

`MultiShardDispatcher` classifies `ShardPreemptedError`, `MissingAttestationError`, and `ShardDispatchError` as "did not respond" (excluded from consensus) while propagating any other exception type. The single dispatcher currently wraps transport-level timeouts into `ShardDispatchError`, so `asyncio.TimeoutError` / `asyncio.CancelledError` at the gather level would indicate an outer-loop cancellation, not a single-provider failure — propagation is correct.

**Disposition:** low-priority audit item. Worth a grep pass in the audit-prep bundle confirming no timeout path escapes `RemoteShardDispatcher` without wrapping. Not a merge-blocker.
