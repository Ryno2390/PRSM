## Context

### How Phase 1.3 fits the master roadmap

The [roadmap](./2026-04-10-audit-gap-roadmap.md) defines seven strategic phases. **Phase 1 ("On-Chain Provenance & Royalty Distribution") is a single phase with three acceptance criteria:**

1. Anyone can independently verify a creator earned FTNS for content X by reading Base mainnet logs.
2. Rolling back royalty payments requires a chain reorg.
3. Existing batch settlement and local ledger remain functional under feature flag.

The phase ships with a release tag and CHANGELOG entry, and the roadmap prescribes: *"After Phase 1, re-audit before starting Phase 2 — gaps may have shifted."*

Phases 1.1 ([plan](./2026-04-10-phase1.1-codex-fixes-plan.md)) and 1.2 ([plan](./2026-04-10-phase1.2-codex-rereview-fixes-plan.md)) were **not in the original roadmap**. They were tactical code-review remediation rounds driven by independent codex audits of the Phase 1 commit. Phase 1.2 reached "SAFE TO DEPLOY" on its fourth codex pass, but that's a *code-level* verdict, not a *roadmap-level* "Phase 1 shipped" verdict. The roadmap acceptance criterion is about the live Base mainnet system, not the Python test suite.

**Phase 1.3 closes that distance.** It is not a new strategic phase — it is the final sub-phase of roadmap Phase 1.

### What's left before Phase 1 can ship

After Phase 1.2, three things still stand between the current state and "Phase 1 shipped per roadmap":

1. **Known deferred P2** — `provenance_hash` is not persisted in the `ContentProvenanceModel` SQL row. `_persist_provenance` (content_uploader.py:1177) and `_hydrate_from_db` (content_uploader.py:1338) both explicitly enumerate fields and both miss it; `ContentProvenanceModel` in `prsm/core/database.py:432` has no column. After a node restart, previously uploaded content loses its hash and silently bypasses on-chain routing.
2. **Suspected production-wiring gap** — `ContentProvider.register_local_content` is only called in tests. The uploader doesn't call it; no node bootstrap call site has been found. If that's accurate, `_local_content` is never populated in production, `ContentProvider._handle_content_request` can never fire the payment path, and the entire Phase 1 stack is inoperable end-to-end despite all tests passing. This must be verified before Sepolia deploy — the cost of a false alarm is small, the cost of skipping verification is Phase 1 shipping broken.
3. **No live deploy** — contracts have never been deployed to Base Sepolia or Base mainnet. Phase 1's acceptance criteria cannot be satisfied by mock/Hardhat tests alone. A bake-in week on Sepolia is the last chance to catch runtime bugs static review missed.

### Out of scope

- **Contract changes.** `ProvenanceRegistry.sol` and `RoyaltyDistributor.sol` are considered stable; no re-audit is triggered unless the bake-in surfaces a bug that requires a contract patch. If that happens, Phase 1.3 forks a fast-patch sub-task.
- **Backfilling pre-1.3 DB rows with `provenance_hash`.** The new column is nullable; legacy rows stay null and fall back to local royalties. Documented limitation.
- **The `parent_cids` vs `parent_content_ids` naming inconsistency** across ~30 call sites. Task 8 of Phase 1.2 caught one live bug from this; audit and normalize scheduled separately (Phase 1.4 candidate if no other phase claims it first). Phase 1.3 intentionally does not touch it — widening scope here delays mainnet.
- **Marketplace / remote compute / wallet SDK / fiat / P2P / storage hardening.** These are Phases 2-7 per the roadmap.

## Architecture Decisions

### 1. Alembic migration, not manual schema patch

`prsm/core/database.py` uses SQLAlchemy with Alembic migrations under `alembic/versions/`. The most recent migration is `014_add_federation_distillation_tables.py`. Phase 1.3 adds `015_add_provenance_hash_column.py` for consistency with the project's pattern. Hand-editing the model without a migration would drift the DB from the declared schema on fresh installs and break production upgrades.

### 2. Nullable column with no default

`provenance_hash` is nullable (`Optional[str]`). Legacy rows uploaded before Phase 1.3 have no hash; the on-chain branch in `_try_onchain_distribute` already returns `None` on a missing hash and falls back to local. Forcing a non-null default would either require synthesizing a fake hash (wrong — hashes are creator-bound) or re-registering every legacy CID (out of scope).

### 3. Investigation-before-fix for the wiring gap

Task 0 is pure investigation, no code. It establishes whether the `register_local_content` gap is real. The subsequent fix task (Task 1) is **conditional** on Task 0's finding. If Task 0 concludes the gap is a false alarm, Task 1 becomes a documentation task only. This prevents "fixing" a non-bug by inventing a wiring path that doesn't belong.

### 4. Operator-gated deploy tasks

Tasks 4, 5, 6, 8, 9 require operator action (holds the `PRIVATE_KEY`, watches Basescan, observes bake-in metrics over multiple days). The agent cannot execute them autonomously. These tasks are structured so the agent prepares everything runnable, then hands off a clear operator checklist. Each deploy task ends with "operator runs X, pastes output back, agent verifies."

### 5. Bake-in exit criteria are explicit

Phase 1 cannot ship to mainnet until Sepolia bake-in completes. "Bake-in" is fuzzy unless the exit criteria are written down. Task 6 specifies the exact metrics to watch and the exact thresholds that must hold for ≥7 days before Task 7 unlocks.

### 6. No contract redeploy after successful Sepolia

Mainnet uses the same Solidity source, the same Hardhat config, the same deploy script — only `FTNS_TOKEN_ADDRESS` and `NETWORK_TREASURY` env vars change. Task 7 includes a bytecode diff check to enforce that invariant. If mainnet bytecode differs from Sepolia bytecode, something broke in the build and the deploy is blocked.

```python
"""Add provenance_hash column to content_provenance table.

Phase 1.3 Task 2. Closes the deferred P2 from the Phase 1.2 codex
re-review: after a node restart the in-memory provenance_hash was
lost because _persist_provenance didn't write it and
_hydrate_from_db didn't read it. This migration adds the column
(nullable, 0x-prefixed hex, max 66 chars) so the round-trip survives.

Revision ID: 015_add_provenance_hash
Revises: 014_add_federation_distillation_tables
Create Date: 2026-04-11
"""
from alembic import op
import sqlalchemy as sa


# revision identifiers, used by Alembic.
revision = "015_add_provenance_hash"
down_revision = "014_add_federation_distillation_tables"
branch_labels = None
depends_on = None


def upgrade() -> None:
    """Add provenance_hash column. Nullable so legacy rows stay null."""
    with op.batch_alter_table("content_provenance") as batch_op:
        batch_op.add_column(
            sa.Column(
                "provenance_hash",
                sa.String(length=66),  # "0x" + 64 hex chars
                nullable=True,
                comment=(
                    "Canonical creator-bound provenance hash "
                    "(keccak256(creator_address || sha3_256(file_bytes))). "
                    "Set at upload time when the node is configured with a "
                    "creator 0x address; None otherwise. Used by "
                    "_try_onchain_distribute to route royalties through "
                    "the Base RoyaltyDistributor."
                ),
            )
        )


def downgrade() -> None:
    """Drop the column on downgrade."""
    with op.batch_alter_table("content_provenance") as batch_op:
        batch_op.drop_column("provenance_hash")
```

> Note: verify the `down_revision` string matches whatever `014_add_federation_distillation_tables.py` declares as its `revision`. If the existing style uses short hashes, match that.
