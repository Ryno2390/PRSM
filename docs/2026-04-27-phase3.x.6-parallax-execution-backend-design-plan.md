## 1. Context & Goals

PRSM's Phase 2 Ring 8 ships `TensorParallelExecutor` — a single-host tensor-parallel runtime that splits one model across multiple local GPUs. Phase 3.x.5 lets manifests propagate cross-node. Phase 7.1 gives us redundant-execution consensus check. What's still missing: **a scheduler that, given a heterogeneous pool of cross-host volunteer GPUs (different cards, different regions, different bandwidth), efficiently runs pipeline-parallel inference across them.**

Naive approaches fail. Round-robin layer placement creates stragglers. Latency-greedy placement creates load imbalance. Cross-region placement creates pipeline-stage stalls because inter-region bandwidth is 1-2 orders of magnitude lower than intra-region. The joint placement-and-routing problem is NP-hard under realistic heterogeneity.

GradientHQ published `Parallax: Efficient LLM Inference Service over Decentralized Environment` (arXiv 2509.26182, Sep 2025) which solves this problem with a two-phase scheduler: Phase-1 model allocation (region-bounded DP + water-filling rebalance) and Phase-2 per-request DAG routing (single-pass DP over a layer-indexed DAG). Empirically: 1.36×-3.2× lower tail latency, 1.58×-3.6× higher throughput vs. HexGen baseline at scales up to 256 GPUs. Apache 2.0; reference implementation at `github.com/GradientHQ/parallax`.

This phase **vendors Parallax's algorithmic core** — `src/scheduling/` — into PRSM's tree under Apache 2.0 attribution, and layers PRSM's trust + economic stack on top via four named adapters.

### 1.1 Why now (vs. why later)

- **The capability gap is real.** PS5-as-a-node / Mac-as-a-node narrative requires cross-host scheduling. Without it, we ship "decentralized model storage but single-host execution" — half a story.
- **The vendor moment is good.** Parallax repo is actively maintained (commits today), 1.3k stars, real adoption signal. Vendoring now captures the mature snapshot; if we wait, we either re-port a divergent codebase or commit to upstream tracking.
- **The paper validates the algorithm**, but **PRSM owns the trust model.** Parallax's allocator trusts profile values verbatim from the DHT — fine in cooperative academic environments, broken under adversarial volunteer GPUs. Layering PRSM's stake/slash on top is the contribution this phase actually adds beyond what's published.
- **Apache 2.0 makes vendor-with-attribution legally clean.** No license-compatibility hazard to manage; the alternative paths (reimplement vs. runtime depend) both have downsides we avoid.

### 1.2 Why vendor and not runtime depend

This is the load-bearing scope decision. **We vendor; we do not `pip install parallax`.**

| Consideration | Vendor (chosen) | Runtime depend |
|---|---|---|
| **Roadmap independence** | PRSM controls the code path forever | PRSM tracks Gradient releases |
| **Threat model alignment** | Trust adapters land directly in the modified code | Trust adapters wrap an opaque package |
| **License surprise** | Apache 2.0 obligations frozen at vendor commit | Future upstream license changes affect us |
| **Audit story** | Auditor reviews ONE codebase | Auditor reviews PRSM + a pinned third-party with its own dep tree |
| **Maintenance burden** | We own bug fixes / security patches | Upstream provides them — but on their schedule, not ours |
| **Selective adoption** | Take only `src/scheduling/`; skip runtime + comms + frontend | All-or-nothing on the package |
| **Lattica transitive dep** | NOT INCLUDED — we vendor only the algorithm | Forced — `parallax==0.1.2` requires `lattica==1.0.21` |

The maintenance-burden cost is real but bounded: we're vendoring ~6-10 files of algorithmic Python (DP search, water-filling, DAG sweep). No active GPU runtime in the vendored set; no cryptography; no external service integration. Bugs in this code are bugs in pure-functional algorithms. Manageable.

### 1.3 Non-goals for Phase 3.x.6

These are explicit deferrals; not gaps.

- **Not vendoring Parallax's runtime layer (`src/parallax/`, `src/backend/`).** PRSM has `TensorParallelExecutor` (Phase 2 Ring 8). The vendor scope is the *scheduler*, not the *runtime*.
- **Not vendoring Lattica.** PRSM has Phase 6 transport for manifest/economic/anchor traffic. The new (τ, ρ) profile DHT this phase introduces uses the same Phase 6 transport; no Lattica dependency anywhere in PRSM's tree.
- **Not vendoring their frontend.** PRSM has its own SDK (Phase 4 Task 3) + MCP surface (Phase 3.x.1).
- **Not provable optimality.** Same as Parallax paper: heuristic + DP gives "good enough." Acceptable for engineering; flagged for governance if it ever becomes a fairness question.
- **Not active rebalancing on every GPU join/leave.** Phase-1 reruns trigger only on coverage gaps or imbalance > threshold (matches Parallax §3.4 semantics). High-churn pools can thrash; documented as risk R5.
- **Not training-mode scheduling.** Inference only. Training-mode pipeline parallelism is separate research (and Echo, also from Gradient, may be the relevant prior art there — out of scope).
- **Not per-token KV-cache repacking.** Inherits Parallax's coarse-granularity assumption (whole layer slice per GPU). Sub-layer scheduling is a research track if it ever matters.

### 1.4 Backwards compatibility

- `TensorParallelExecutor` (single-host) continues to be the default `InferenceExecutor` implementation. No existing caller breaks.
- Phase 3.x.6 adds `ParallaxScheduledExecutor` as a NEW `InferenceExecutor` implementation. Operators choose at startup.
- `MarketplaceOrchestrator` (Phase 7) treats both executors as the same Protocol — no orchestrator changes.
- The new (τ, ρ) profile DHT is opt-in. Operators not running cross-host inference don't see it.

### 3.2 Two-phase scheduler (vendored algorithm)

Inherits from Parallax paper §3.2-3.4. Briefly:

**Phase-1 (model allocation)** runs once per (model, GPU pool) configuration:
- Sort GPUs by layer capacity (descending)
- DP search over (skip / extend pipeline / start new pipeline) transitions
- Maximize replication count subject to stage-count minimization
- Water-filling rebalance: redistribute layers within each pipeline proportional to per-GPU compute capacity
- Output: a set of complete pipelines, each pipeline a contiguous-layer assignment to GPUs in one region

**Phase-2 (request routing)** runs per inference request:
- Treat the allocation as a layer-indexed DAG (nodes = (layer, GPU) pairs; edges = consecutive-layer transitions)
- Single-pass DP sweep: minimum-latency path from layer 1 to layer L
- Constraint: chain must cover every layer exactly once and in order
- Output: GPU chain pinned for the request lifetime

Both phases consume the (τ_g,ℓ, ρ_g,g') profile from the DHT. Both are pure-functional given that input — easy to test, easy to audit.

### 3.3 Four trust adapters (PRSM-original layer)

This is the work the Parallax paper does NOT do. It's what makes the vendored algorithm safe to run against adversarial volunteers.

**Adapter A — Anchor-verify GPU candidates before they enter Phase-1.**
Before a GPU is added to the allocation pool, the orchestrator queries Phase 3.x.5 manifest DHT: does this GPU's claimed `node_id` resolve to a manifest signed by a publisher on the Phase 3.x.3 anchor? If not, exclude. Closes a class of "fake GPU advertises capacity to capture routing" attacks.

**Adapter B — Tier gate before Phase-2 routing.**
At request time, BEFORE the DAG sweep runs, filter the GPU set by content tier: a privacy-tier-MAXIMUM workload only routes through GPUs that have advertised a hardware-TEE attestation (Phase 2 Ring 8). Phase-2 then runs over the filtered subset. Refuses the request entirely if the filtered subset can't cover all layers.

**Adapter C — Stake-weighted profile trust.**
The naive Parallax algorithm uses (τ_g,ℓ, ρ_g,g') verbatim. Adapter C divides each GPU's advertised speed by a confidence factor derived from its current stake (Phase 7 `StakeManager`). A GPU with no stake gets 0× confidence (effectively excluded from routing); fully-staked GPUs get 1× (full advertised speed). This makes profile-lying economically irrational — to win routes you must stake; if you lie, the stake gets slashed via Adapter D.

**Adapter D — Post-route consensus-mismatch hook.**
After a chain completes, the orchestrator OPTIONALLY (config-driven) routes a small fraction of inferences to a second, independent chain for redundant execution. Output mismatch between the two chains triggers Phase 7.1's `CONSENSUS_MISMATCH` challenge, which slashes the misbehaving GPU(s). Sample rate is operator policy — defaults to 1% redundancy. This is the deterrent that makes Adapter C credible.

### 3.4 Profile DHT — wire format + delta from Parallax

Parallax's DHT publishes (τ_g,ℓ, ρ_g,g') every 1-2 seconds. Wire format isn't pinned in the paper. PRSM's profile DHT mirrors the Phase 3.x.5 manifest DHT pattern:

```
ProfileAnnouncement {
  node_id: str           # 32-char hex; resolves to manifest registration
  per_layer_latency: dict[int, float]  # {layer_idx: τ in ms}
  rtt_table: dict[str, float]          # {peer_node_id: ρ in ms, half-RTT}
  capacity_layers: int   # max layers this GPU can serve concurrently
  tier_attestation: str  # TEE attestation hash (NONE for software TEE)
  stake_amount: int      # FTNS wei staked (snapshot)
  signed_by: str         # Ed25519 signature over canonical payload
  protocol_version: int  # = 1
}
```

Same JSON-over-Phase-6-transport pattern, same anchor-verify before trust, same TTL-expire-on-leave. Operators NOT running cross-host inference simply don't publish — the DHT degrades to local-only.

### 3.5 What changes in `MarketplaceOrchestrator`

Minimal. The orchestrator already calls an `InferenceExecutor` Protocol; it stays the same. The choice of which executor (single-host `TensorParallelExecutor` vs. cross-host `ParallaxScheduledExecutor`) is a startup-config decision. Per-request dispatch is unchanged.

### Task 2 — Adapt Phase-1 model allocation
**Scope:** `prsm/compute/parallax_scheduling/layer_allocation.py` — adapt the DP + water-filling algorithm to PRSM's data types. Specifically: replace Parallax's GPU descriptor with a PRSM `ParallaxGPU` dataclass that includes `node_id`, `region`, `layer_capacity`, `stake_amount`, `tier_attestation`. Algorithm unchanged.

**Tests:** `tests/unit/test_parallax_layer_allocation.py` — port the upstream test cases that exist (if any), add PRSM-specific cases:
- Region-based heuristic: 4 GPUs in region A, 4 in region B → produces 2 single-region pipelines
- Latency-dominant heuristic: prefers fewer pipeline stages over more
- Water-filling: heterogeneous compute → balanced stage times
- Edge: empty pool → empty allocation (no panic)
- Edge: single GPU with capacity ≥ L → single 1-stage pipeline
- Edge: total capacity < L → no valid allocation (raises specific exception)

**Acceptance:** Allocation matches paper §3.2 expected behavior on synthetic inputs; tests green.
**Estimated: 2 days.**

### Task 3 — Adapt Phase-2 request routing
**Scope:** `prsm/compute/parallax_scheduling/request_routing.py` — adapt the per-request DAG DP sweep to consume PRSM's profile DHT (Task 5) instead of Parallax's. Algorithm unchanged.

**Tests:** `tests/unit/test_parallax_request_routing.py`:
- Single-replica pool: routes through the only available chain
- Multi-replica pool: minimum-latency path selected
- Stale profile: degrades gracefully (falls back to last-known-good)
- Coverage gap: returns specific failure (caller must rerun Phase-1)

**Acceptance:** Per-request routing latency stays within paper's reported scaling — single-digit ms at hundreds of GPUs.
**Estimated: 2 days.**

### Task 4 — Profile DHT (PRSM-native) on Phase 6 transport
**Scope:** `prsm/compute/parallax_scheduling/profile_dht.py` — implement the (τ, ρ) profile DHT using Phase 6 P2P transport. **Tests:** `tests/unit/test_parallax_profile_dht.py`:
- Publish + lookup happy path
- Stale entry expiry (TTL respected)
- Anchor verify rejects unsigned profile
- Anchor verify rejects profile signed by unregistered publisher
- Concurrent updates: latest wins on per-node basis
- Wire-format protocol version mismatch handled

**Acceptance:** Profile DHT round-trip works; anchor verification enforced.
**Estimated: 2 days.**

### Task 5 — Trust adapters (the load-bearing PRSM contribution)
**Scope:** `prsm/compute/parallax_scheduling/trust_adapter.py` — implement the four adapters from §3.3:
- `AnchorVerifyAdapter` filters GPU candidates before Phase-1 (anchor lookup against Phase 3.x.5 manifest DHT)
- `TierGateAdapter` filters GPU subset before Phase-2 (TEE attestation match against requested privacy tier)
- `StakeWeightedTrustAdapter` rescales advertised (τ, ρ) by stake confidence before scheduler consumes them
- `ConsensusMismatchHook` post-route — schedules redundant execution at configured sample rate, surfaces mismatches as Phase 7.1 challenges

**Tests:** `tests/unit/test_parallax_trust_adapter.py`:
- Anchor-fail GPU excluded from Phase-1
- Tier mismatch raises `TierGateRejected` (no chain attempted)
- Zero-stake GPU's profile gets 0× confidence (de facto excluded)
- Full-stake GPU sees 1× confidence (full advertised speed)
- Sample-rate=0 means no redundant execution
- Sample-rate=1 means every request is redundant
- Mismatch detection routes to Phase 7.1 `CONSENSUS_MISMATCH` reason
- Adapter composition: stake-weight + tier-gate + anchor-verify all enforced

**Acceptance:** All adversarial-volunteer scenarios from §3.3 demonstrate appropriate rejection / slash propagation.
**Estimated: 3 days** — load-bearing security work; budget room for review iteration.

### Task 6 — `ParallaxScheduledExecutor` — `InferenceExecutor` Protocol implementation
**Scope:** `prsm/compute/inference/parallax_executor.py` — implement the existing `InferenceExecutor` Protocol on top of the scheduler. Composes:
1. Build profile DHT lookup of currently-available GPUs
2. Apply `AnchorVerifyAdapter` → filtered GPU pool
3. Run Phase-1 allocation (cached per model+pool config; recomputed on coverage gap or imbalance threshold)
4. On request: apply `TierGateAdapter` → filter further by tier
5. Apply `StakeWeightedTrustAdapter` → rescale profile values
6. Run Phase-2 DAG sweep → GPU chain
7. Hand chain to `TensorParallelExecutor` for actual layer execution per stage
8. `ConsensusMismatchHook` post-route (configurable sample rate)
9. Sign result with PRSM `InferenceReceipt` (Phase 3.x.1 Task 2)

**Tests:** `tests/unit/test_parallax_executor.py`:
- Returns `InferenceResult.success` on happy path
- Returns `InferenceResult.failure` with specific reason when allocation fails
- Returns specific reason when tier gate rejects
- Receipt is signed with node identity (matches existing executor contract)
- Phase-1 recompute triggered on coverage gap
- Phase-1 recompute NOT triggered when localized GPU-leave is absorbed (matches paper §3.4)

**Acceptance:** Drop-in replacement for `TensorParallelExecutor` from the orchestrator's perspective.
**Estimated: 2 days.**

### Task 7 — End-to-end integration test (3 simulated nodes, mirrors Phase 3.x.5 Task 7)
**Scope:** `tests/integration/test_parallax_e2e.py` — three simulated nodes (alice, bob, charlie) with different mock GPU profiles (different layer capacities, different RTT), all anchor-registered. Validate:
- Happy path: alice publishes a model; bob requests inference; pipeline routes through alice + charlie; result returns signed
- Region-aware: alice (region A) + bob (region A) → preferred over alice + charlie (region A → B) when feasible
- Anchor enforcement: mallory (unregistered) advertises high-capacity profile; excluded from routing
- Tier gate: privacy-MAXIMUM request fails when no hardware-TEE GPU in pool
- Stake weighting: zero-stake GPU advertises faster than staked GPU; staked GPU still routed
- Consensus mismatch: malicious provider returns garbage; second-chain mismatch detected; Phase 7.1 challenge fires
- Membership churn: GPU leave during inference fails the request gracefully (no hung chain)

**Acceptance:** All seven scenarios pass; matches the test rigor pattern of Phase 3.x.5 E2E (which caught real bugs at this gate).
**Estimated: 3 days.**
