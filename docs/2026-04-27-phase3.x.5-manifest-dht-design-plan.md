## 1. Context & Goals

Phase 3.x.2 ships `FilesystemModelRegistry` — an operator's models live on their own node. Phase 3.x.3 closed the cross-node TRUST gap (verifiers can authenticate any manifest given the on-chain anchor). What's still missing is cross-node AVAILABILITY: if a popular model's publisher node goes offline or can't keep up with download demand, no other node can serve a verified copy.

This phase adds an opt-in distributed hash table that lets any node serve a verified copy of any registered manifest, keyed by `model_id`. Trust is unchanged — readers still verify via the 3.x.3 anchor; the DHT is just routing + bytes.

### 1.1 Why now (vs. why later)

Honest take: **DHT addresses a future scaling concern, not a current blocker.** Today's PRSM has a small operator set; one publisher node serving its own models works fine. The reason to ship this now and not in Phase 4+ is:

- **The pattern is warm**: 3.x.2 + 3.x.3 idioms (signed wire formats, anchor lookup, opt-in kwargs) port directly to a DHT layer. Building it now while the patterns are fresh is cheaper than coming back in 6 months.
- **The trust upgrade is in place**: without 3.x.3, a DHT would either re-introduce filesystem-trust limitations or require a parallel trust mechanism. With 3.x.3 shipped, DHT readers verify via anchor for free.
- **It unlocks investor-demo "PRSM is genuinely distributed"** narrative: the existing single-publisher serving model isn't a story. DHT-backed serving is.

### 1.2 Non-goals for Phase 3.x.5

These are explicit deferrals; not gaps.

- **Not privacy-budget journal DHT.** Journals are per-node-owner by design — distributing them doesn't compose naturally. Read access for audit is rare and out-of-band.
- **Not inference-receipt DHT.** Receipts go from settler to requester directly; no cross-node availability gap exists.
- **Not shard distribution.** This phase distributes the MANIFEST (small, ~1–10 KB). Shard distribution is a much bigger track that interacts with Phase 7-storage encryption + Reed-Solomon erasure coding; explicitly deferred.
- **Not active replication / repair.** Operators self-host. If a manifest's publishers all go offline, it disappears from the DHT. No churn-resilience guarantees in v1.
- **Not write-side conflict resolution.** First-publisher-on-DHT wins per `model_id`; later-publishing-with-same-model_id is rejected by readers via anchor mismatch. Documented.
- **Not gossip/PubSub.** DHT is request/reply only — `find_providers(model_id)` + `fetch_manifest(node_id, model_id)`. No invalidation broadcast.
- **Not a registry-mode replacement.** `FilesystemModelRegistry` continues to be the write authority. DHT is a read-side fallback when the local registry doesn't have the model.

### 1.3 Backwards compatibility

- All existing `FilesystemModelRegistry(root, anchor=...)` callers keep working unchanged. DHT is opt-in via a new `dht=` kwarg.
- DHT-published manifests use the same `ModelManifest` wire format as the local sidecar — no new schema.
- Phase 3.x.3 anchor verification is still the authoritative trust check. DHT bytes that fail anchor verification are rejected.

### 1.4 Trust model

| Concern | Defense |
|---|---|
| Malicious node serves wrong manifest bytes | Reader verifies signature via 3.x.3 anchor; mismatch → drop, try next provider. |
| Malicious node serves OLD manifest (key rotation eluded) | Reader sees `publisher_signature` valid under OLD key but new anchor key returns False — caught. |
| Malicious node refuses to serve (DOS by absence) | Reader tries next provider; eventually fails with `ManifestNotFoundError`. Retry-with-jitter at the application layer. |
| Sybil flood: attacker spawns many nodes claiming to provide a manifest | All sybils serve content the reader rejects; cost is a few wasted RPCs. Documented limitation. |
| Cache poisoning: stale entry in `find_providers` after publisher leaves | `find_providers` results carry a TTL; readers re-query on miss. |

## 2. Scope

### 2.1 In scope

**New module: `prsm/network/manifest_dht/`**

- `protocol.py` — wire-format messages: `FindProviders`, `ProvidersResponse`, `FetchManifest`, `ManifestResponse`. JSON over the existing TCP/SOCKS transport (`prsm.node.transport_adapter`).
- `local_index.py` — per-node index of manifests this node CAN serve (model_id → manifest bytes path). Populated by `FilesystemModelRegistry.register()` when DHT is wired in.
- `dht_client.py` — `ManifestDHTClient` with:
  - `announce(model_id)` — tell the network this node can serve the manifest
  - `find_providers(model_id) → list[node_id]` — Kademlia-style closest-peers iterative lookup
  - `fetch_manifest(node_id, model_id) → ModelManifest` — pull bytes from one provider
  - `get_manifest(model_id) → ModelManifest` — composes find + fetch + anchor verification
- `dht_server.py` — RPC handlers that respond to inbound `FindProviders` / `FetchManifest` requests. Plugs into the existing PRSM node RPC dispatcher.

**Integration into `FilesystemModelRegistry`**

- `FilesystemModelRegistry(root, *, anchor=None, dht=None)` — new optional `dht` kwarg.
- When `dht` configured:
  - `register(model, identity)` ALSO publishes to DHT (`dht.announce(model_id)` + indexes the manifest path).
  - `get(model_id)` falls back to DHT lookup if the model isn't local. **Anchor verification still mandatory** — DHT-fetched manifests that fail anchor verify raise `ManifestVerificationError`.

**Tests**

- `tests/unit/test_manifest_dht_protocol.py` — message format + serialization roundtrip + signing-payload-style canonical-bytes pin.
- `tests/unit/test_manifest_dht_local_index.py` — index populates on register; lookup; persistence across restart.
- `tests/unit/test_manifest_dht_client.py` — client behavior with mocked transport (find_providers iterative routing, fetch_manifest, get_manifest happy + anchor-verify-fail + no-providers paths).
- `tests/unit/test_manifest_dht_server.py` — server RPC handlers respond correctly to all 4 message types.
- `tests/unit/test_manifest_dht_exports.py` — public-surface smoke (mirrors prior phases).
- `tests/integration/test_manifest_dht_e2e.py` — three simulated nodes (alice/bob/charlie), alice publishes a model, bob fetches via DHT, charlie verifies via anchor; tampering scenarios; offline-publisher-but-cached-elsewhere scenario.

### 2.2 Out of scope (named for traceability)

| Out-of-scope item | Where it goes |
|---|---|
| Privacy-budget journal DHT | Not planned; per-node-owner semantics don't compose |
| Inference-receipt DHT | Not planned; no availability gap exists |
| Shard distribution via DHT | Future phase (interacts with Phase 7-storage erasure coding) |
| Active replication / repair on operator churn | Future phase if real operator-churn data shows need |
| PubSub-style invalidation broadcast | Future; current TTL-based cache is sufficient |
| Write conflict resolution beyond first-write-wins | Future on-chain registration (artifact-CID anchoring deferred from 3.x.3) |
| MCP `prsm_dht_lookup` tool | Optional; useful for debugging, not blocking |

## 3. Architecture

```
┌────────────────────────────────────────────────────────────────────┐
│ Caller (verifier on any node)                                      │
│   registry.get("llama-3-8b")                                        │
│         │                                                           │
│         ▼                                                           │
│   FilesystemModelRegistry — local sidecar present? → return        │
│         │ (no — fall back to DHT if dht= configured)                │
│         ▼                                                           │
│   ManifestDHTClient.get_manifest("llama-3-8b")                     │
│      ├─ find_providers("llama-3-8b") → [node_id_a, node_id_b, …]   │
│      ├─ fetch_manifest(node_id_a, "llama-3-8b") → bytes             │
│      ├─ verify via anchor (Phase 3.x.3 wrapper)                    │
│      └─ if verify OK → return; else next provider                  │
└────────────────────────────────────────────────────────────────────┘
                            │
                            │  TCP / SOCKS RPC over Phase 6 transport
                            ▼
┌────────────────────────────────────────────────────────────────────┐
│ Provider node (different operator)                                 │
│   ManifestDHTServer (RPC handler)                                  │
│      ├─ FindProviders(model_id) → look up local + Kademlia find    │
│      └─ FetchManifest(model_id) → read manifest.json from registry │
└────────────────────────────────────────────────────────────────────┘
```

### 3.1 Wire format (canonical JSON)

All four messages are JSON dicts over the transport. Each carries a `type` field for dispatch.

```python
# Sent by client:
{"type": "find_providers", "model_id": "...", "request_id": "..."}
{"type": "fetch_manifest", "model_id": "...", "request_id": "..."}

# Sent by server:
{"type": "providers_response", "request_id": "...",
 "providers": [{"node_id": "...", "address": "host:port"}, ...]}
{"type": "manifest_response", "request_id": "...",
 "manifest": {<ModelManifest.to_dict()>}}

# Errors:
{"type": "error", "request_id": "...", "code": "NOT_FOUND" | ..., "message": "..."}
```

Sizes: `find_providers` ~80 bytes; `manifest_response` typically 1–10 KB depending on shard count. Well within transport-layer message limits.

### 3.2 Routing (Kademlia-style iterative lookup)

Same XOR-distance routing as the existing `KademliaDHT` in `prsm.compute.collaboration.p2p.node_discovery`. We don't reimplement — we wrap.

`ManifestDHTClient.find_providers(model_id)`:
1. Compute `target = sha256(model_id)[:32]` (32 bytes — same width as node_id derivation).
2. Get the K nearest peers from the local routing table.
3. Query each: "Do you have it locally? If not, who's closest in your view?"
4. Iterate until either a provider responds with bytes OR the search converges with no providers found.

Replication factor (K) configurable; default K=3 (matches existing `KademliaDHT.K`).

### 3.3 Anchor verification is MANDATORY

The DHT client REFUSES to return a manifest unless:
- An anchor is configured at construction
- The fetched manifest's signature verifies under the anchor's key for `publisher_node_id`

Without anchor: `RuntimeError` at construction. There's no "trust the network" mode. This is the trust-model invariant from Phase 3.x.3 enforced at the DHT layer.

### 3.4 Failure modes

| Failure | Result |
|---|---|
| No providers respond | `ManifestNotFoundError` |
| Provider responds with bytes that fail anchor verify | Drop, try next provider; if all fail → `ManifestNotFoundError` |
| Anchor RPC failure | `AnchorRPCError` propagates (transport-level distinction) |
| Transport timeout on a single provider | Drop that provider, try next |
| All providers timeout | `ManifestNotFoundError` |
| Local index points at a manifest file that's been deleted | Log warning, return error to requester |

## 4. Tasks

### Task 1 — Wire-format protocol module
**Scope:** `prsm/network/manifest_dht/protocol.py` — frozen dataclasses for the four message types + JSON serialize/deserialize. Schema version pinned. Domain-separated from anchor / receipt / manifest signatures (we don't sign DHT messages — the manifest payload carries its own signature).
**Tests:** `tests/unit/test_manifest_dht_protocol.py` — roundtrip, unknown-type rejection, missing-field rejection, schema-version pinning.
**Acceptance:** Wire format pinned; tests cover each message type's encode/decode.
**Estimated: 3–4 hours.**

### Task 2 — Local provider index
**Scope:** `prsm/network/manifest_dht/local_index.py` — `LocalManifestIndex(root: Path)`. Maps `model_id → path-to-manifest.json` for manifests this node CAN serve. Populated implicitly by `FilesystemModelRegistry.register()` when DHT is wired (Task 5). Persists as a small JSON file under `<root>/dht_index.json`; rebuilt from filesystem walk if missing.
**Tests:** `tests/unit/test_manifest_dht_local_index.py` — register adds an entry, lookup returns path, persistence across instances, missing file → rebuild from filesystem walk.
**Acceptance:** Index round-trips through restart; `model_id → path` is unambiguous.
**Estimated: 4–6 hours.**

### Task 3 — DHT client (find_providers + fetch_manifest)
**Scope:** `prsm/network/manifest_dht/dht_client.py` — `ManifestDHTClient(routing_table, transport, anchor)`. Three operations:
- `announce(model_id, my_node_id)` — local-only state update (no broadcast in v1; see §1.2)
- `find_providers(model_id) → list[(node_id, address)]` — Kademlia iterative lookup over the existing routing table
- `fetch_manifest(node_id, model_id) → ModelManifest` — RPC fetch from one peer
- `get_manifest(model_id) → ModelManifest` — composes find + fetch + anchor verification

Anchor is REQUIRED at construction; absence raises `RuntimeError` (no trust-the-network mode).

**Tests:** `tests/unit/test_manifest_dht_client.py` — each operation with a mocked transport; happy path; no-providers; provider returns bytes that fail anchor verify (drop and try next); anchor RPC failure propagates.
**Acceptance:** Client behavior pinned by tests; the no-anchor-no-trust invariant is enforced + tested.
**Estimated: 6–8 hours.**

### Task 4 — DHT server (RPC handlers)
**Scope:** `prsm/network/manifest_dht/dht_server.py` — `ManifestDHTServer(local_index, routing_table)` with handlers for the four message types. Plugs into the existing PRSM node RPC dispatcher pattern (similar to Phase 6 P2P transport's RPC layering).
**Tests:** `tests/unit/test_manifest_dht_server.py` — each message type → correct response shape; unknown type → error response; concurrent requests don't corrupt state.
**Acceptance:** Server responds correctly to all message types.
**Estimated: 4–6 hours.**

### Task 5 — Integration into `FilesystemModelRegistry`
**Scope:** `prsm/compute/model_registry/registry.py` — add `dht=None` kwarg to `FilesystemModelRegistry.__init__`. When configured:
- `register(model, identity)` ALSO calls `dht.announce(model.model_id)` after writing the local sidecar/manifest. Failures in DHT announce are logged but do NOT fail the registration (local registry is the write authority).
- `get(model_id)` falls back to `dht.get_manifest(model_id)` when the local sidecar is missing. The fetched manifest is anchor-verified (Task 3); the local cache is populated for next time. `ManifestVerificationError` propagates if anchor verify fails.

**Tests:** Extend `tests/unit/test_model_registry.py` — `dht=` integration with mocked DHT client. New scenarios: register-publishes-to-DHT, get-falls-back-to-DHT, DHT-fetched-manifest-fails-anchor-verify-raises, DHT-announce-failure-doesnt-break-register.
**Acceptance:** With `dht=` configured, an empty registry can serve manifests fetched from the network; without `dht=`, behavior is unchanged from Phase 3.x.3.
**Estimated: 4–6 hours.**

### Task 6 — Public exports + smoke test
**Scope:** `prsm/network/manifest_dht/__init__.py` — public exports. `tests/unit/test_manifest_dht_exports.py` — smoke test mirroring `test_publisher_key_anchor_exports.py`.
**Acceptance:** Public surface pinned by tests.
**Estimated: 1 hour.**

### Task 7 — End-to-end integration test (3 simulated nodes)
**Scope:** `tests/integration/test_manifest_dht_e2e.py` — three simulated nodes (alice, bob, charlie). Alice registers a model; bob fetches via DHT; charlie verifies via anchor. Plus negative scenarios: malicious provider serves tampered bytes (anchor verify catches; bob retries from charlie); offline publisher (alice goes offline; bob still gets the manifest from charlie's cache).
**Acceptance:** End-to-end flow demonstrates `register on alice → fetch on bob → verify on charlie` works; tampering scenarios all caught.
**Estimated: 4–6 hours.**

## 6. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Sybil flood (attacker spawns many fake providers) | Low | Low | All bogus providers serve bytes the reader rejects via anchor verify. Cost is a few wasted RPCs. Documented. |
| First-publisher conflict on a model_id | Low | Medium | First-write-wins on each node's local index; cross-node, the first publisher to register on the anchor wins (Phase 3.x.3 write-once contract). DHT readers verify via anchor → only the legitimate publisher's bytes pass. |
| Stale `find_providers` results after publisher offline | Medium | Low | TTL on cached results; readers retry with fresh lookup on miss. |
| Network partition leaves orphan publisher | Medium | Low | Operator concern; readers in the orphaned partition see `ManifestNotFoundError`. Reconnection heals organically. |
| Manifest > transport message size (e.g., 10K shards) | Low | Medium | Manifest size scales with shard count; 10K shards × ~150 bytes = 1.5 MB per manifest. Likely under default transport limits. If not, document the cap; chunked transfer is a future feature. |
| RPC dispatch contention between manifest-DHT and other PRSM RPCs | Low | Low | Server uses the existing dispatcher pattern; per-message overhead is small. |
| DHT amplification attack (small request → large manifest response) | Medium | Low | Manifests are 1–10 KB; amplification factor is bounded. Rate-limiting at the transport layer (Phase 6) handles abuse. |
