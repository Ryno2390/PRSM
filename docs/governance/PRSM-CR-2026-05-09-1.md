## 2. Background

### 2.1 Current state (verified on-chain 2026-05-09)

The mainnet RoyaltyDistributor at `0x3E8201B2cdC09bB1095Fc63c6DF1673fA9A4D6c2` (v1, deployed 2026-05-04 per `provenance-base-1777917793612.json`) lacks an `Ownable` surface and lacks a recovery path for stranded donations. Per `A-08-recoverStranded-design.md`, this means:

- Direct FTNS donations to the v1 distributor address are permanently stranded — no on-chain mechanism exists to sweep them
- No owner-gated administrative surface for future operational needs

### 2.2 v2 source state

`contracts/contracts/RoyaltyDistributor.sol` in the current repo state implements three improvements per `A-08-recoverStranded-design.md`:

1. `Ownable2Step` — owner = Foundation Safe, 2-step transfer flow defends against typo-on-transfer
2. `totalClaimable` accumulator — kept in lockstep with `claimable[]` writes; enables `recoverStranded` math
3. `recoverStranded(address to)` — owner-gated sweep of donations not credited to any creator/operator/network slot

20 Hardhat tests pass against the v2 source as of commit reference at the head of `main` on ratification date.

### 2.3 Sepolia rehearsal status

Per ceremony plan §2.4 + §8, this ceremony is gated on a Sepolia rehearsal completing within 7 days preceding mainnet execution. The Sepolia rehearsal runbook is at `docs/governance/2026-05-09-A-08-sepolia-rehearsal-runbook.md`; the rehearsal manifest will be archived as `contracts/deployments/sepolia-a08-rehearsal-<timestamp>.json` upon successful completion.

**This resolution becomes operative only after the rehearsal manifest has been committed to the repo.** The founder confirms rehearsal completion as part of the ratification signature (§5 below).

## 3. Ratification scope

The council ratifies the following five binding decisions:

### 3.1 Adoption of ceremony plan as canonical reference

`docs/governance/2026-05-09-A-08-v2-redeploy-ceremony-plan.md` (commit hash recorded in §5 ratification block) is the canonical reference for the mainnet ceremony. Deviation from the documented step-by-step in §4 of the plan requires a superseding resolution. Specifically: the constructor argument table in plan §3 is binding; the step ordering in plan §4 is binding; the rollback plan in plan §6 is binding.

### 3.2 Authorization for deploy gas spend

Foundation authorizes operational expenditure of up to **0.01 ETH on Base mainnet** for the ceremony's gas costs (deploy v2 + transferOwnership; the multisig acceptOwnership leg is paid by the Safe-funded executor of the cosigned tx, not separately authorized here). Spend within this cap requires no further council approval. Spend exceeding this cap pauses the ceremony pending a superseding CR.

Source of funds: deployer hot key (operational wallet, not Foundation Safe). Foundation Safe is not debited for ceremony gas.

### 3.3 Pre-commitment to rollback plan §6

The rollback / abort tree in ceremony plan §6.1–§6.5 is pre-ratified. The founder may execute any rollback path enumerated in §6 without additional council approval. Specifically:

- Mid-ceremony abandonment (§6.4): networks.py left pinned to v1; no economic impact
- Post-ceremony rollback (§6.5): revert networks.py commit on `main`, push release reverting v2 pin, sweep v2 balance via `recoverStranded` from Foundation Safe

### 3.4 Post-ceremony documentation duties

The founder commits to executing ceremony plan Step 4.8 within 24 hours of mainnet acceptOwnership completion:

- Add audit-prep §7.x entry to `docs/2026-04-27-cumulative-audit-prep.md` capturing new address + transfer txs + post-deploy verification proof
- Update memory entry `project_t10_a08_2026_05_07.md` to reflect ceremony-complete state
- Tag the ceremony-completion commit `a-08-v2-redeploy-ceremony-complete-<date>`

### 3.5 Operator-side migration commitment

Per ceremony plan §5, operators currently using `PRSM_ROYALTY_DISTRIBUTOR_ADDRESS` env override pinned to v1 retain control over their own migration timing. Foundation does NOT initiate migration of v1 claimable balances; existing balances remain claimable indefinitely on v1. New on-chain royalty distributions route through v2 only after the operator's node updates to a release containing the post-ceremony networks.py pin commit.
