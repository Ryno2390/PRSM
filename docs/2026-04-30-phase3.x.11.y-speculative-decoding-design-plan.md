## 1. Problem statement

Phase 3.x.11 sharded autoregressive decode trades memory pressure (no longer concentrated on the tail) for **wire latency per token**. For a 3-stage chain over WAN with 50ms RTT:
- Per-token wire latency: 3 hops × 50ms = 150ms minimum
- Token throughput: ~6-7 tokens/second (before model decode time)

Phase 3.x.11.x (wire-level pipelining via chunked PREFILL) amortizes the *prefill* phase by overlapping chunk emission/consumption across stages — but the *decode* phase remains serialized: the executor cannot dispatch INCREMENTAL N+1 to Stage 1 until INCREMENTAL N's tail returns the next token. Each stage is idle T-1 / T of the time per decode step.

The standard solution is **speculative decoding**: a small, fast **draft model** proposes the next K tokens; the full sharded chain verifies them in a single batched forward pass. On accept, K tokens delivered in one chain pass instead of K separate dispatches. On reject, fall back to standard one-token decode.

**Phase 3.x.11.y closes this gap.** A draft model proposes 4-8 candidate tokens per step; the sharded chain verifies them; the executor accepts the longest matching prefix and rolls back the KV-cache for any rejected suffix.

After this phase, an operator with a 3-stage WAN chain at 50ms RTT can serve 30+ tokens/second under typical accept-rates (4-8x speedup), bringing sharded WAN deployments into the same throughput class as single-host LAN deployments.

## 2. Scope

**In scope.**
- **Draft model interface** — new `DraftModel` Protocol with `propose(input_ids, k) → List[int]` returning K speculative tokens. Default impl: a small co-located HF model loaded alongside the tail (n-gram is too weak; same-architecture small model gives the highest accept-rates).
- **Verification protocol** — new `VERIFY` decode mode on `RunLayerSliceRequest`. Activation-blob carries `[token_id_for_pos_N, token_id_for_pos_N+1, ..., token_id_for_pos_N+K]` (K+1 tokens including the parent); each stage runs them in a single batched forward; tail returns logits for each position so the executor can compare proposed-vs-verified.
- **Acceptance + rollback** — executor compares draft tokens to verified tokens position-by-position; accepts the longest matching prefix; sends per-stage `RollbackCacheRequest` to truncate KV-cache for rejected suffix.
- **Wire-format extension** — new `RollbackCacheRequest(request_id, n_positions_to_drop)` wire message + handler. KV-cache truncation is in-place on the `KVCacheHandle.payload` (model adapter does the actual tensor slice).
- **Receipt extension** — per-iteration attestation envelope (Phase 3.x.11.x Task 1) is already iteration-shaped; speculative iterations record `accepted_count: int` so receipts capture acceptance-rate distribution.
- **Composition tests** with multi-stage chain producing genuinely-distinct output across stages with K ≥ 4 speculation depth.
- **E2E with real distilgpt2 + small draft model** (e.g., a 6-layer distilgpt2 as full chain + a 2-layer draft variant) — proves the architecture works end-to-end.

**Out of scope (deferred).**
- **Adaptive K tuning** — auto-adjust speculation depth based on observed accept-rate. v1 ships with operator-configured K. Phase 3.x.11.y.x.
- **Multi-draft consensus** — propose K candidates per draft × multiple drafts → consensus. Research direction; Phase 3.x.11.y' (apostrophe).
- **Cross-request draft caching** — share draft KV-cache across same-prompt-prefix requests. Phase 3.x.11.y''.
- **Tier C compatibility** — speculative decoding adds another timing surface (acceptance-rate per token) to Phase 3.x.11.q's existing carry-over. Phase 3.x.11.q.y bundle.
- **Cache swap-out / paging** — Phase 3.x.11.z.
- **Mid-stream re-routing** — Phase 3.x.12.

## 3. Architecture

### 3.1 Per-iteration speculation flow

```
Phase 3.x.11 baseline (per-token chain redispatch):
  Iteration N:    Stage1(t=N) → Stage2(t=N) → ... → StageT(t=N) → emit 1 token
  Iteration N+1:  Stage1(t=N+1) → ... → emit 1 token
  ...
  Cost per token: T network round-trips + T forward passes

Phase 3.x.11.y (speculative):
  Iteration N:    DraftModel proposes [d1, d2, d3, d4]
                  Stage1(VERIFY [parent, d1, d2, d3, d4]) → batched
                  → Stage2 → ... → StageT → 5 logit vectors
                  Executor: accept matching prefix, e.g., [d1, d2] match
                            but verified[3] ≠ d3 → accept count = 2,
                            emit verified[0:3] as 3 tokens (d1, d2, true_3)
                  RollbackCacheRequest to all stages (drop 2 positions:
                    d3, d4 were speculatively cached but rejected)
  Iteration N+1:  next round of speculation starting from accepted prefix
  ...
  Cost per ACCEPT_COUNT+1 tokens: T network round-trips + T forward passes
                  Effective speedup: (ACCEPT_COUNT+1)x at typical
                  accept-rates of 4-8 per round
```

Key insight: a single chain pass now produces ACCEPT_COUNT+1 tokens instead of 1. Network latency per token drops by the same factor (under typical accept-rate of 60-80% per position with same-arch draft).

### 3.2 DraftModel Protocol

```python
class DraftModel(Protocol):
    """Proposes speculative tokens for the verification chain.

    Co-located with the executor (or with the tail stage; v1
    co-locates with executor for simplicity). Stateful — caller
    reset() at PREFILL boundaries; call propose() per
    speculation round. Cache lifecycle independent of the
    KVCacheManager (different model, different layer set)."""

    def reset(self, *, request_id: str, prompt_input_ids: List[int]) -> None:
        """Initialize per-request draft state from the prompt."""
        ...

    def propose(
        self,
        *,
        request_id: str,
        parent_token_id: int,
        k: int,
        temperature: float,
    ) -> List[int]:
        """Propose K speculative tokens following parent_token_id.
        Returns up to K token ids. If draft model produces EOS
        early, returns fewer than K — caller treats this as
        ``draft proposes K but only proposes-up-to-EOS``."""
        ...

    def commit(
        self,
        *,
        request_id: str,
        accepted_token_ids: List[int],
    ) -> None:
        """Caller indicates which speculative tokens were
        accepted. Draft model rolls forward its own KV-cache
        to match. Rejected speculative tokens are dropped from
        draft state."""
        ...

    def evict(self, *, request_id: str) -> None:
        """Per-request cleanup. Idempotent."""
        ...
```

v1 default impl: `HFDraftModel` wrapping a small HuggingFace `AutoModelForCausalLM` with a `DynamicCache` (same pattern as Phase 3.x.11 Task 7's `_DistilGPT2ShardedAdapter`). Operator-configured at `RpcChainExecutor` construction; `None` means "no speculation, fall back to Phase 3.x.11 single-token decode."

### 3.3 Wire-format extension: VERIFY decode mode + RollbackCacheRequest

```python
class DecodeMode(str, Enum):
    PREFILL = "prefill"
    INCREMENTAL = "incremental"
    VERIFY = "verify"  # NEW Phase 3.x.11.y
```

`RunLayerSliceRequest` activation interpretation:
- `PREFILL`: Stage 1 input = `input_ids` (full prompt); Stage > 1 input = full prompt's hidden state from prior stage.
- `INCREMENTAL`: Stage 1 input = `[next_token_id]` (1 token); Stage > 1 input = single-position hidden state.
- `VERIFY` (NEW): Stage 1 input = `[parent_token_id, draft_1, draft_2, ..., draft_K]` (K+1 tokens); Stage > 1 input = K+1 positions of hidden state. **Each stage forwards the FULL K+1 batch with cached KV from the parent**, returning K+1 hidden states (or for the tail, K+1 logit vectors).

`RunLayerSliceResponse` extensions:
- `verified_token_ids: Optional[List[int]]` — tail-only; populated on VERIFY. K+1 entries: argmax per position.
- `accepted_count: Optional[int]` — tail-only; populated on VERIFY. The count of speculative tokens that match the verified output (longest matching prefix).
- Existing `next_token_id` + `is_terminal` semantics adjust: on VERIFY, `next_token_id` = the FIRST disagreement token (the executor will use it as the seed for the next speculation round); `is_terminal` = True if any verified token is EOS or `tokens_generated + accepted_count + 1 >= max_tokens`.

`RollbackCacheRequest` wire message (parallel to Phase 3.x.11 Task 6's `EvictCacheRequest`):
```python
@dataclass(frozen=True)
class RollbackCacheRequest:
    request_id: str
    n_positions_to_drop: int  # truncate the LAST N positions from the cache
    protocol_version: int = CHAIN_RPC_PROTOCOL_VERSION
```

Server-side `KVCacheManager.rollback(request_id, n)` truncates the cache by N positions. Idempotent (rollback past pos 0 → no-op). Best-effort (server-side TTL still bounds zombies on broadcast failure).

### 3.4 Acceptance algorithm

Standard speculative-decoding acceptance:
1. Tail returns `verified_token_ids = [v0, v1, ..., vK]` (K+1 argmaxes).
2. Compare position-by-position: `accepted_count = max i such that verified_token_ids[0..i] == [parent, d_1, ..., d_i]`.
3. **Emit `accepted_count + 1` tokens**: `[v_0, ..., v_{accepted_count}]`. Note: `v_0` is always the verified version of `parent`'s next-position output. If `accepted_count == K`, all draft tokens accepted + we get one bonus token from the verifier; if `accepted_count == 0`, only one token emitted (the verifier's correction).
4. **Rollback the cache** by `K - accepted_count` positions (the rejected suffix was speculatively cached during VERIFY; we need to drop them so the next round starts from a consistent state).
5. Send `RollbackCacheRequest(request_id, K - accepted_count)` to ALL chain stages.
6. Update draft model state via `draft.commit(accepted_token_ids=[v_0, ..., v_{accepted_count}])`.

**Guarantees:**
- Output distribution is **identical** to non-speculative greedy decode (when temperature=0 / argmax verification). Speculation is a perf optimization, not a sampling change.
- For temperature > 0, requires careful sampling via the [Leviathan et al. 2023] speculative-sampling correction. v1 ships **greedy-only** (`temperature == 0` required for speculation; raise on positive temperature). Sampling-correct speculation deferred to Phase 3.x.11.y.x.

### 3.5 Executor per-token chain loop with speculation

```python
def _execute_chain_streaming_sharded_speculative(
    self, *, request, chain,
):
    # PREFILL — same as Phase 3.x.11
    next_token_id, _, prefill_outcomes = self._run_chain_iteration_sharded(
        decode_mode=DecodeMode.PREFILL, ...,
    )
    self._draft.reset(request_id=..., prompt_input_ids=...)
    yield StreamToken(token_id=next_token_id, ...)
    tokens_emitted = 1

    # Speculation loop
    while not is_terminal and tokens_emitted < max_tokens:
        # Step 1: draft proposes K tokens
        proposed = self._draft.propose(
            request_id=...,
            parent_token_id=next_token_id,
            k=self._speculation_depth,
            temperature=0.0,
        )

        # Step 2: chain verifies
        verified, accepted_count, is_terminal, outcomes = (
            self._run_chain_iteration_sharded_verify(
                input_ids=[next_token_id] + proposed,
                proposed=proposed,
                ...,
            )
        )

        # Step 3: emit accepted_count + 1 tokens
        emitted = verified[: accepted_count + 1]
        for tok in emitted:
            yield StreamToken(token_id=tok, ...)
            tokens_emitted += 1
            if tokens_emitted >= max_tokens:
                is_terminal = True
                break

        # Step 4: rollback cache for rejected suffix
        rollback_n = len(proposed) - accepted_count
        if rollback_n > 0:
            self._rollback_cache_on_stages(chain, request.request_id, rollback_n)
            self._draft.commit(accepted_token_ids=emitted)

        # Step 5: continue with the LAST emitted token as next seed
        next_token_id = emitted[-1]

    # Terminal — same eviction broadcast as Phase 3.x.11
    yield ChainExecutionResult(...)
    self._evict_cache_on_stages(chain, request.request_id)
    self._draft.evict(request_id=request.request_id)
```

Speculation depth K is operator-configured (`speculation_depth: int = 4` default). K too low → little speedup; K too high → more rejected work + per-VERIFY-request size grows linearly with K. Production sweet spot is 4-8 for same-architecture drafts on similar-distribution tasks.

### 3.6 Honest scope — speculation correctness depends on draft model quality

**v1 ships honest about three limitations:**

1. **Greedy-only.** Speculative sampling under nonzero temperature requires the Leviathan-2023 correction (rejection sampling on the verifier's distribution). v1 raises on `temperature > 0`; sampling-correct speculation is Phase 3.x.11.y.x.

2. **Accept-rate is workload-dependent.** Reported speedups in the literature (4-8x) assume same-architecture draft + similar-distribution prompts. PRSM operators serving heterogeneous workloads may see 2-3x. Honest scope: operators benchmark their workload before tuning K.

3. **Accept-rate timing surface.** Tier C structurally remains denied for sharded decode (Phase 3.x.11.q deferred); speculation adds a NEW signal (per-iteration `accepted_count` is observable on the wire) that further differentiates Tier A/B from Tier C. Threat-model addendum will need a new section when Phase 3.x.11.q lands.

## 4. Tasks

### Task 1 — Wire-format extension: VERIFY mode + verified_token_ids/accepted_count + RollbackCacheRequest
**Scope:** add `DecodeMode.VERIFY`. Add `verified_token_ids: Optional[List[int]]` + `accepted_count: Optional[int]` to `RunLayerSliceResponse` (omit-when-default for byte-equivalence). Add `RollbackCacheRequest` + `RollbackCacheResponse` wire envelopes. Extend `signing_payload` to commit verified_token_ids + accepted_count when set (mirrors the Phase 3.x.11 Task 5 pattern — without commitment, a malicious relay could swap verified tokens without invalidating the signature).
**Tests:** ≥6 — round-trip; omit-when-default byte-equivalence; bool-rejection on accepted_count; range validation on verified_token_ids (each ≥ 0, len ≤ 64 cap); RollbackCacheRequest round-trip + idempotency contract documented; signing-payload coverage extension.

### Task 2 — `KVCacheManager.rollback`
**Scope:** new `KVCacheManager.rollback(request_id, n_positions)` method. Truncates the last N positions from the cache by mutating the handle's payload. Manager doesn't introspect the payload — calls a new `model.truncate_cache(payload, n)` on the runner side. Idempotent (rollback when n > current_seq_len → drop everything).
**Tests:** ≥4 — happy path truncates by N; truncate-zero is no-op; truncate-past-zero clears the cache; concurrent rollback + get-incremental thread safety.

### Task 3 — `DraftModel` Protocol + `HFDraftModel`
**Scope:** new `prsm/compute/inference/draft_model.py` with `DraftModel` Protocol + reference `HFDraftModel` (small HF AutoModelForCausalLM + DynamicCache). reset/propose/commit/evict lifecycle. Greedy-only (raises on temperature > 0).
**Tests:** ≥6 — propose returns K tokens; propose stops early on EOS; commit advances state; commit-fewer-than-K rolls draft cache forward by accepted count only; evict idempotent; constructor validation.

### Task 4 — `ShardedAutoregressiveRunner.run_layer_slice_unary` VERIFY support
**Scope:** extend the runner to handle `decode_mode == VERIFY`. Stage 1 input = `[parent, d1, ..., dK]` (K+1 token ids); model performs a batched forward of K+1 positions with cached KV from parent; tail samples K+1 logit vectors via `apply_lm_head_and_sample_batch` (new model method) and computes `accepted_count` via prefix-match against `proposed`. Cache state after VERIFY: K+1 positions appended (executor will RollbackCacheRequest the rejected suffix in step 4 of acceptance).
**Tests:** ≥5 — VERIFY happy path with K=4, all accepted; VERIFY with all rejected (accepted_count=0); VERIFY with partial accept (e.g., 2 of 4); EOS in verified middle position → is_terminal=True; non-tail VERIFY returns hidden_state batch + verified_token_ids=None.

### Task 5 — Executor speculation loop
**Scope:** `RpcChainExecutor` constructor accepts `draft_model: Optional[DraftModel]` + `speculation_depth: int = 4`. New `_execute_chain_streaming_sharded_speculative` path branches when draft_model is wired. PREFILL same as Phase 3.x.11; speculation loop drives draft.propose → chain VERIFY → accept → rollback → emit. Eviction broadcast at terminal calls draft.evict + manager.evict + RollbackCacheRequest broadcast on every exit path.
**Tests:** ≥6 — speculation produces multiple-tokens-per-iteration; all-rejected falls back gracefully; max_tokens cap honored across speculation rounds; EOS mid-speculation terminates; cancellation triggers rollback + eviction broadcasts; greedy-equivalence (output bit-identical to Phase 3.x.11 non-speculative single-token decode for the same prompt + temperature=0).

### Task 6 — `RollbackCacheRequest` wire handler + executor broadcast
**Scope:** server-side `_handle_rollback_cache` handler routes to `kv_cache_manager.rollback`. Executor broadcasts on every speculation step where `accepted_count < K`. Best-effort (mirrors EvictCacheRequest pattern from Phase 3.x.11 Task 6).
**Tests:** ≥4 — round-trip; rollback past zero is idempotent; broadcast hits all stages; ack parsing.

### Task 7 — E2E with real distilgpt2 + small draft

Loads distilgpt2 as the full chain (split across 2 stages) AND a small draft model (e.g., a 2-layer truncated distilgpt2 — adequate for testing, not optimal). **Tests:** ≥4 — speculation accepts at least one multi-token step; output matches single-host greedy bit-identical; receipt records accepted_count; cancellation mid-speculation cleanly rolls back.

## 7. Estimated effort

~2-3 weeks engineering, 9 tasks. The big-lift items are:
- Task 4 (`ShardedAutoregressiveRunner.run_layer_slice_unary` VERIFY support — batched K+1-position forward with cached KV is non-trivial wiring on the model adapter side).

The slice opens significant new surface: a new wire-format mode + new wire envelope + new model Protocol + executor-side state machine. Risk areas: (a) cache-rollback race conditions when broadcast lags behind the next speculation step, (b) draft/verify model divergence producing weird accept-rate distributions, (c) the speculation-correctness invariant (output identical to non-speculative greedy) holding under all corner cases (EOS in middle of verified, max_tokens hit mid-speculation, rejected suffix wraparound).
