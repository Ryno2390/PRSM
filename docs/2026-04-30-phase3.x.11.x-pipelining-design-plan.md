## 1. Problem statement

Phase 3.x.11 shipped with two honest-scope deferrals that this slice closes:

1. **No chunked + sharded composition.** When a sharded executor produces an activation that exceeds `chunk_threshold_bytes` (default 10 MiB), it raises `ACTIVATION_TOO_LARGE`. The Phase 3.x.7.1 chunked-streaming path exists, and the sharded path exists, but they don't compose at v1. The Task 9 M1 remediation enforces the unary-only-when-sharded invariant on both server and executor sides — which means real operators serving large-prompt sharded workloads hit a hard wall.

2. **No per-token cryptographic commitment from stages.** The signed `InferenceReceipt` commits to the joined output text + the multi-stage TEE attestation envelope from the LAST iteration's per-stage outcomes only. Per-iteration attestations are NOT aggregated; an operator (or auditor) reading the receipt cannot verify "stage K served EVERY incremental dispatch, not just the last one". A malicious stage that swapped its KV-cache mid-stream would NOT be detected by the receipt — the per-iteration manipulation would manifest as wrong activation outputs (caught by R7 challenge-and-verify if the operator wires it) but not by the receipt's signature chain.

Phase 3.x.11.x ships:
- Chunked + sharded composition. Operators with prompts that produce >10 MiB activations can serve over the sharded path.
- Per-token receipt attestation. The receipt's TEE-attestation envelope grows from "one record per stage" to "one record per stage per dispatch", with the attestation chain length committed in the signature.

**Pipelining context.** The "pipelining" framing is wire-level, not compute-level: chunked activation transport from Stage K to Stage K+1 streams chunks as they're produced, so Stage K+1 begins consuming chunks before Stage K finishes its forward. Phase 3.x.7.1 already does this for unary; this slice extends that pattern to sharded PREFILL. Compute-level pipelining (overlapping consecutive-token forward passes with speculation) requires draft models and is the home of Phase 3.x.11.y.

## 2. Scope

**In scope.**
- **Chunked + sharded composition** for PREFILL dispatches. The executor's `_run_chain_iteration_sharded` lifts the M1 ACTIVATION_TOO_LARGE rejection for PREFILL when `streamed_send_message` is wired and the activation crosses the threshold. The server's `_dispatch_streamed` lifts the M1 sharded-runner deny for PREFILL — but only when the request decodes as `decode_mode=PREFILL`. INCREMENTAL stays unary-only (single-position activations are tiny by construction; chunked INCREMENTAL is structurally pointless).
- **Per-token receipt attestation envelope.** New `MultiIterationStageAttestation` dataclass: list-of-list-of-StageAttestation indexed by `[iteration_index][stage_index]`. The executor's `_execute_chain_streaming_sharded` accumulates per-iteration outcomes (Phase 3.x.11 already keeps `cumulative_outcomes` but discards iteration boundaries); the new envelope preserves them.
- **Receipt signing payload extension.** `InferenceReceipt.signing_payload` extended to commit the per-iteration attestation chain. Omit-when-default preserves byte-equivalence for non-sharded receipts.
- **Audit-prep §7.10** + threat-model addendum revision documenting the closed gap.

**Out of scope (deferred).**
- **Speculative decoding** (draft-and-verify across stages). Phase 3.x.11.y.
- **Compute-level pipelining** of consecutive-token forward passes. Requires speculation. Phase 3.x.11.y.
- **Per-token KV-cache commitment in receipt.** The receipt commits to per-iteration attestations (proves the stage was on-watch); it does NOT commit to the K/V tensors themselves. KV-cache commitment requires a Merkle root over the cached tensors per iteration — significant compute overhead. Phase 3.x.11.x' (apostrophe) follow-up.
- **Cache swap-out / paging.** Phase 3.x.11.z.
- **Mid-stream re-routing.** Phase 3.x.12.
- **Tier C compatibility.** Phase 3.x.11.q (depends on a fresh constant-time padding scope).

## 3. Architecture

### 3.1 Chunked + sharded PREFILL composition

Phase 3.x.11 Task 9 M1 added two guards:
- Server: `_dispatch_streamed` rejects when `self._sharded_runner is not None`.
- Executor: `_dispatch_stage` raises `ACTIVATION_TOO_LARGE` when `self._enable_sharded_decode` and the blob exceeds threshold.

This slice lifts both for PREFILL only. The new logic:

**Server (`_dispatch_streamed`):**
```python
if self._sharded_runner is not None:
    if request.decode_mode == DecodeMode.PREFILL:
        # Phase 3.x.11.x — chunked sharded PREFILL composition
        # is supported. Reassemble the activation from chunks,
        # then dispatch through the sharded runner exactly as
        # the inline path does.
        return self._dispatch_streamed_sharded(request, chunk_iter)
    # INCREMENTAL: still unary-only (single-position activations
    # don't benefit from chunking).
    return self._streamed_error(
        request.request_id,
        StageErrorCode.MALFORMED_REQUEST,
        "sharded INCREMENTAL is unary-only; chunked transport "
        "is supported on PREFILL only (Phase 3.x.11.x).",
    )
```

**Executor (`_dispatch_stage`):**
```python
if should_chunk(blob, threshold=self._chunk_threshold_bytes):
    if self._enable_sharded_decode and decode_mode != DecodeMode.PREFILL:
        # INCREMENTAL: still rejected.
        raise ChainExecutionError(...)
    # PREFILL or non-sharded: proceed with streamed dispatch.
    return self._dispatch_streamed(...)
```

**`_dispatch_streamed_sharded`** is a new server method that mirrors `_dispatch_streamed` but routes to the sharded runner:

```python
def _dispatch_streamed_sharded(
    self, request: RunLayerSliceRequest, chunk_iter: Iterable[bytes],
) -> Tuple[bytes, Iterable[bytes]]:
    # Step 1-4: same validation gates as _dispatch_sharded
    #   (model lookup, layer-range, content tier, deadline).
    # Step 5: same Tier C structural deny.
    # Step 6: validate streamed envelope (reuse existing code
    #   from _dispatch_streamed).
    # Step 7: reassemble chunks → activation tensor.
    # Step 8: dispatch to self._sharded_runner.run_layer_slice_unary
    #   (same as _dispatch_sharded).
    # Step 9: encode hidden_state output, sign response with
    #   next_token_id + is_terminal, return.
```

The reassembly path reuses Phase 3.x.7.1's `ShardAssembler` + `validate_streamed_envelope` machinery — no new chunked-bytes machinery.

### 3.2 Per-iteration attestation envelope

**Current shape (Phase 3.x.7 Task 5):**
```python
StageAttestation(stage_index, stage_node_id, tee_type, attestation)
encode_multi_stage_attestation(List[StageAttestation]) -> bytes
```

The receipt's `tee_attestation` field carries `encode_multi_stage_attestation(stage_attestations)` where `stage_attestations` is one record per chain stage.

**New shape (Phase 3.x.11.x):**
```python
@dataclass(frozen=True)
class IterationAttestation:
    iteration_index: int  # 0-indexed; 0 = PREFILL, 1+ = INCREMENTALs
    decode_mode: DecodeMode
    stage_records: List[StageAttestation]

encode_multi_iteration_attestation(
    List[IterationAttestation],
) -> bytes
```

The receipt's `tee_attestation` field carries `encode_multi_iteration_attestation(per_iteration)` for sharded receipts. For non-sharded receipts (Phase 3.x.7+ unary, Phase 3.x.8+ streaming-tail), the existing `encode_multi_stage_attestation` is preserved — the executor selects which encoder to use based on `enable_sharded_decode`. **Byte-equivalence preserved for non-sharded receipts.**

The decoder side (auditor / settler verifying a receipt) needs to recognize both shapes. Discriminator: the encoded JSON's top-level key — `"stages"` for the unary shape, `"iterations"` for the multi-iteration shape. Both are valid `tee_attestation` blobs; the receipt's signing payload commits whichever was used.

### 3.3 Receipt signing-payload extension

Phase 3.x.7 Task 5 already commits the `tee_attestation` blob via `InferenceReceipt.signing_payload`. The new envelope produces a different blob; the signature still commits to it byte-for-byte. **No signing-payload code change is required** — the existing `tee_attestation_hex` field absorbs the new shape transparently.

What DOES need to change: the receipt-level invariant that "for sharded receipts, `tee_attestation` decodes as `IterationAttestation[]`" needs to be checked at verify-time when callers know they're verifying a sharded receipt. The settler-side verifier gets a new `expect_sharded: bool` kwarg.

### 3.4 Honest scope — pipelining is wire-level, not compute-level

**Wire-level (this slice, Phase 3.x.11.x):**
- Chunked PREFILL transport overlaps Stage K's chunk emission with Stage K+1's chunk consumption. For a 100 MB PREFILL activation chunked at 10 MB, Stage K+1 starts processing chunk 1 while Stage K is still encoding chunks 8-10. Effective per-PREFILL wall-clock drops by roughly the chunked-transport-latency factor.

**Compute-level (deferred, Phase 3.x.11.y):**
- Overlapping consecutive-token decode forward passes requires speculation: Stage 1 must guess token N+1 to start work before Stage T's token-N response arrives. This is draft-and-verify; it's a fundamentally different architecture and ships as Phase 3.x.11.y.

v1 ships honest about this. The §1 "pipelining" framing is wire-level only.

## 4. Tasks

### Task 1 — Wire-format extension: IterationAttestation
**Scope:** new `IterationAttestation` dataclass in `prsm/compute/inference/multi_stage_attestation.py`. New `encode_multi_iteration_attestation` + `decode_multi_iteration_attestation` functions. Discriminator key (`"stages"` vs `"iterations"`) at the top-level JSON encoding. Validation: bool-rejection on `decode_mode`, non-negative `iteration_index`, non-empty `stage_records`.
**Tests:** ≥6 — round-trip; encode shape verification; decoder discriminator on `"stages"` (back-compat) vs `"iterations"`; reject malformed (missing iteration_index, empty stage_records, non-DecodeMode); golden-bytes byte-equivalence pin for the existing `encode_multi_stage_attestation` shape.

### Task 2 — Executor accumulates per-iteration outcomes
**Scope:** `_execute_chain_streaming_sharded` currently keeps `cumulative_outcomes: List[StageOutcome]` and discards iteration boundaries. Replace with `per_iteration_outcomes: List[List[StageOutcome]]` — one inner list per chain pass (PREFILL + each INCREMENTAL). The receipt-emit step uses the new envelope encoder. Operators with `enable_sharded_decode=False` keep the existing single-list path (unchanged).
**Tests:** ≥4 — 1-iteration receipt encodes with iteration_index=0 + decode_mode=PREFILL; 4-iteration receipt has 4 IterationAttestation entries; non-sharded execute_chain receipt unchanged (byte-equivalence); cancellation mid-stream still emits the iteration envelope with all completed iterations.

### Task 3 — Server `_dispatch_streamed_sharded`
**Scope:** new server method that handles the chunked + sharded PREFILL path. Reuses `_run_validation_gates` + `_validate_streamed_envelope` + chunk reassembly from `_dispatch_streamed`; routes to `self._sharded_runner.run_layer_slice_unary` like `_dispatch_sharded`. Tier C structural deny carries forward. INCREMENTAL stays rejected.
**Tests:** ≥5 — happy path PREFILL with chunked activation; INCREMENTAL streamed rejected (MALFORMED_REQUEST); Tier C streamed rejected (TIER_GATE); envelope validation rejects inflated payload_bytes; chunk reassembly errors → ACTIVATION_INVALID.

### Task 4 — Executor lifts the chunked + sharded PREFILL guard
**Scope:** `_dispatch_stage` opens the `should_chunk + enable_sharded_decode` branch for PREFILL only (raises ACTIVATION_TOO_LARGE on INCREMENTAL still). Verify the chunked transport calls `_dispatch_streamed` with `decode_mode=PREFILL` — server routes to `_dispatch_streamed_sharded`.
**Tests:** ≥4 — sharded PREFILL with >threshold activation succeeds end-to-end; INCREMENTAL still raises ACTIVATION_TOO_LARGE with the new message; non-sharded chunked path unchanged; non-chunked sharded path unchanged.

### Task 5 — E2E with chunked sharded PREFILL on a long prompt
**Scope:** integration test (`@pytest.mark.slow`) that loads distilgpt2, splits across 2 stages (alice 0-2, bob 3-5), drives a long-prompt PREFILL (~256 tokens → activation > 10 MiB at fp32) through the executor's per-token chain loop with `chunk_threshold_bytes=4_000_000` (lowered to force chunking on the test prompt). Verifies: chunked path is exercised (delivery_log shows streamed transport hits); bit-identical output vs single-host greedy (preserves the 3.x.11 Task 7 invariant); per-iteration attestation envelope present in the receipt.
**Tests:** ≥3 — chunked PREFILL streamed correctly; output bit-identical; receipt encodes IterationAttestation entries.

### Task 6 — Code review + merge-ready tag
**Scope:** independent review across Phase 3.x.11.x commits. HIGH/MEDIUM remediations pre-tag. Apply `phase3.x.11.x-merge-ready-20260XXX` tag. Update MEMORY.md + audit-prep §7.10.

~1 week engineering, 6 tasks. Smaller surface than Phase 3.x.11 because most of the wire-format machinery (chunked transport, multi-stage attestation envelope, sharded dispatch path) already exists. The big-lift items are:
- Task 3 (`_dispatch_streamed_sharded`) — server-side wiring + reassembly path. ~1 day.
